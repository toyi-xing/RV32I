## 第0章 本专题学习地图

### 0.0 为什么要学乱序微架构

经典五级流水线是 CPU 入门核心，但高性能 CPU 不会满足于“一条指令卡住，后面都等”。真实程序里有很多独立指令可以并行执行。乱序微架构的目标就是：

```text
在不改变 ISA 可见结果的前提下，允许硬件按数据就绪和资源可用情况重新安排执行顺序。
```

乱序执行解决的是性能问题，但它不能破坏：

- 程序顺序提交。
- 精确异常。
- 分支错误恢复。
- 内存顺序要求。
- CSR、trap 和 MMIO 的架构语义。

因此乱序核的核心不是“让指令随便跑”，而是把一条指令拆成两个世界：

| 世界 | 可以乱序吗 | 保存在哪里 | 何时变成软件可见 |
|---|---|---|---|
| 微架构执行状态 | 可以 | PRF、IQ/RS、ROB、LSQ、bypass 网络 | 不能直接被软件观察 |
| 架构提交状态 | 不可以 | GPR/CSR/memory 的提交结果 | commit/retire 时按程序顺序更新 |

这也是本篇的主线：rename 解决“名字相关”，IQ/RS 解决“谁 ready 谁先执行”，ROB 解决“最后仍按程序顺序提交”，LSQ 解决“内存相关不能靠寄存器重命名完全解决”，recovery 解决“推测错了怎么回到正确状态”。

### 0.1 从顺序流水线到乱序核的核心变化

顺序五级流水线通常是：

```text
IF -> ID -> EX -> MEM -> WB
```

乱序核常见逻辑流变成：

```text
fetch -> decode -> rename -> dispatch -> issue -> execute -> writeback -> commit
```

关键变化：

- decode 后先 rename，消除 WAR/WAW 假相关。
- 指令进入 ROB，保证最终按程序顺序 commit。
- 指令进入 issue queue / RS，等待操作数和功能单元。
- 数据 ready 后乱序 issue 和 execute。
- 结果写回 PRF，并唤醒依赖它的后续指令。
- 分支或异常出错时，从 checkpoint/ROB 恢复。

## 第1章 乱序执行解决什么

### 1.0 顺序执行的瓶颈

顺序流水线中，如果一条 load miss 卡住，后面即使有很多不依赖这条 load 的 ALU 指令，也可能一起被阻塞。这样硬件资源空着，IPC 降低。

例如：

```text
LW   x5, 0(x1)      // cache miss
ADD  x8, x2, x3     // 与 x5 无关
XOR  x9, x4, x6     // 与 x5 无关
SUB  x10, x7, x11   // 与 x5 无关
```

乱序核希望在 `LW` 等数据时，先执行后面的独立指令。

从性能模型看，顺序核很容易被单个长延迟事件拖住：

$$
IPC_{seq} \approx \frac{1}{CPI_{base} + penalty_{stall}}
$$

乱序核通过更大的窗口寻找独立工作，目标是把一部分 stall latency 藏起来：

```text
long-latency load miss
  while waiting:
    execute independent ALU ops
    resolve independent branches
    prepare later memory addresses
```

但窗口越大，硬件越复杂。ROB、IQ、LSQ、PRF 都要变大，wakeup/select 和 bypass 会更难收敛。所以乱序执行是用面积、功耗、验证复杂度换取平均 IPC 的典型设计。

### 1.1 乱序不是乱提交

乱序执行允许：

- 指令乱序发射。
- 指令乱序完成。
- 功能单元乱序写回。

但通常仍要求：

- 指令按程序顺序 commit。
- exception 看起来发生在某条精确指令边界。
- 分支错误路径不改变架构状态。

这就是 ROB 存在的核心原因。

### 1.2 数据相关的三类问题

| 相关 | 含义 | 是否真依赖 | 乱序中怎么处理 |
|---|---|---|---|
| RAW | 读后写 | 是 | 等生产者结果 ready |
| WAR | 写后读 | 否 | register renaming 消除 |
| WAW | 写后写 | 否 | register renaming 消除 |

RAW 是真实数据依赖，不能凭空消除。WAR/WAW 是名字依赖，只要把架构寄存器映射到不同物理寄存器，就可以消除。

### 1.3 乱序核的基本不变量

乱序核再复杂，也要维持几个不变量：

- 每条已分派指令在 ROB 中有唯一条目。
- 每个架构目的寄存器在 RAT 中映射到一个最新物理寄存器。
- 每个物理寄存器要么空闲，要么被某个架构状态或未提交指令引用。
- commit 永远按 ROB head 到 tail 顺序进行。
- 分支恢复后，错误路径分配的资源必须释放或失效。

这些不变量可以理解成乱序核的“守恒关系”。例如物理寄存器不能凭空丢失：

$$
PRF_{total} = PRF_{arch\_mapped} + PRF_{inflight} + PRF_{free}
$$

如果 free list 释放过早，一个还可能被分支恢复需要的旧物理寄存器会被重新分配；如果释放过晚，free list 会逐渐耗尽，机器看起来像随机卡死。验证时要用断言和 scoreboard 同时检查 RAT、ROB old mapping 和 free list 是否一致。

## 第2章 Register Renaming：从架构寄存器到物理寄存器

### 2.0 为什么需要重命名

假设程序如下：

```text
ADD x5, x1, x2
SUB x5, x3, x4
OR  x6, x5, x7
```

第二条 `SUB` 和第一条 `ADD` 都写 `x5`，这是 WAW 名字冲突。第三条 `OR` 应该读第二条 `SUB` 的结果。用物理寄存器可改写成：

```text
ADD p10, p1, p2
SUB p11, p3, p4
OR  p12, p11, p7
```

这样 `ADD` 和 `SUB` 可以独立完成，只有真正读 `p11` 的 `OR` 依赖 `SUB`。

### 2.1 RAT

RAT 记录：

$$
arch\_reg \rightarrow physical\_reg
$$

rename 时：

1. 源寄存器查 RAT，得到源物理寄存器编号。
2. 目的寄存器从 free list 分配新的物理寄存器。
3. RAT 更新为新的目的映射。
4. ROB 记录旧的目的映射，用于 commit 后释放或 recovery。

rename 的关键是每条指令都在“当前推测路径”的 RAT 上工作。以两条连续写同一架构寄存器为例：

```text
rename before:
  RAT[x5] = p10

I1: ADD x5, x1, x2
  allocate p20
  old mapping = p10
  RAT[x5] = p20

I2: SUB x5, x3, x4
  allocate p21
  old mapping = p20
  RAT[x5] = p21
```

I1 commit 后释放的是 `p10`，I2 commit 后释放的是 `p20`。如果 I1 还没 commit 时分支恢复到 I1 之前，就必须恢复 `RAT[x5]=p10`，并释放错误路径分配的 `p20/p21`。这个例子能把 rename、ROB old mapping、free list recovery 三者串起来。

### 2.2 PRF

PRF 存放所有物理寄存器值。它比架构寄存器多，才能容纳：

- 已提交架构状态。
- 未提交新结果。
- 推测路径临时结果。

PRF 设计难点：

- 多读多写端口昂贵。
- bypass 网络复杂。
- ready bit 与数据写回要严格对齐。

### 2.3 free list

free list 管理空闲物理寄存器。

分配时从 free list pop；释放时把旧物理寄存器 push 回去。释放时机常见有两种：

- commit 后释放旧映射。
- 分支恢复时释放错误路径新分配的物理寄存器。

多宽 rename 时，free list 不是简单弹出一个寄存器。假设 rename width 为 $N$，同周期最多需要分配 $N$ 个目的物理寄存器，还可能同周期从 commit 回收若干旧寄存器：

$$
free\_count_{next} = free\_count_{old} - alloc\_count + release\_count
$$

工程上要处理：

| 场景 | 风险 | 处理 |
|---|---|---|
| free list 剩余不足 | 半条 fetch bundle 被 rename | 要么整体 stall，要么支持部分 rename |
| 同周期 allocate/release | 重复分配刚释放但仍被恢复需要的寄存器 | 明确旁路和优先级 |
| branch recovery | wrong-path 分配未回收 | 用 checkpoint 或 allocation log 恢复 |
| x0 | 被错误分配物理寄存器 | 固定映射到 zero 或 rename 时跳过 |

### 2.4 busy table / ready bit

busy table 表示某个物理寄存器的值是否 ready：

- 新分配目的寄存器时，标记 busy。
- 执行完成写回时，标记 ready。
- issue queue 等待源操作数 ready。

### 2.5 rename checkpoint

为了快速分支恢复，rename stage 常对分支指令建立 checkpoint：

- 保存 RAT。
- 保存 free list 状态或分配历史。
- 保存部分 predictor/RAS 状态。

分支预测错误时，恢复 checkpoint，并 kill 后续 wrong-path 指令。

checkpoint 的粒度是设计取舍：

| 方案 | 保存内容 | 恢复速度 | 面积/复杂度 |
|---|---|---:|---|
| full RAT checkpoint | 每个分支保存完整 RAT | 快 | 面积大 |
| incremental log | 记录分支后的映射变化 | 中等 | 恢复需回放/撤销 |
| ROB walk | 从 ROB 逐条回滚 | 慢 | checkpoint 面积小 |

高性能前端更倾向快速恢复，因为 branch mispredict penalty 直接影响 IPC；小型乱序核可能接受 ROB walk，以降低面积和实现复杂度。

## 第3章 ROB：乱序执行和顺序提交的桥

### 3.0 ROB 的核心作用

ROB 是乱序核的“程序顺序骨架”。它让指令可以乱序执行，但最后按程序顺序 commit。

每条指令进入 ROB 时会得到一个 ROB index。ROB entry 通常包含：

- valid。
- busy / completed。
- PC。
- instruction type。
- destination architectural register。
- new physical register。
- old physical register。
- exception 信息。
- branch 信息。
- store 信息。

ROB entry 不一定保存完整执行结果。有两种常见风格：

| 风格 | 结果放哪里 | ROB 保存什么 | 特点 |
|---|---|---|---|
| ROB 保存结果 | ROB entry 存 result，commit 时写架构寄存器 | result、rd、exception | 结构直观，但 ROB 宽且写端口复杂 |
| PRF 保存结果 | 执行结果写 PRF，ROB 只保存 phys tag 和状态 | phys rd、old phys、done、exception | 更适合现代 rename/PRF 设计 |

本篇默认采用 PRF 风格：ROB 负责顺序和恢复，PRF 负责数据值，RAT/free list 负责名字映射和资源生命周期。

### 3.1 ROB 环形缓冲

ROB 常实现为 circular buffer：

```text
head -> oldest instruction
tail -> next allocation position
```

提交从 head 开始，分派从 tail 写入。

ROB full 时必须 backpressure 前端或 rename/dispatch。

### 3.2 commit 规则

ROB head 指令可以 commit 的典型条件：

- valid。
- completed。
- 没有 unresolved exception。
- 不是等待特殊序列化事件。
- 对 store 来说，满足允许对外可见的条件。

多发射核可能每周期 commit 多条，但仍必须从 ROB head 开始连续提交。

多提交宽度不是“每拍随便提交最多 N 条”。它必须从 head 连续扫描，遇到第一个不能提交的 entry 就停止：

```text
ROB head order:
  E0 done
  E1 done
  E2 not done
  E3 done

commit width = 4
this cycle can commit: E0, E1
E3 cannot bypass E2
```

这样才能保证架构状态按程序顺序更新。commit 同时还会触发资源释放，例如释放 old physical register、让 store queue 中对应 store 对外可见、释放 branch checkpoint、更新性能计数器等。

### 3.3 精确异常

精确异常要求：

- 异常指令之前的指令都已提交。
- 异常指令之后的指令都没有改变架构状态。
- trap handler 看到一致的架构状态。

ROB 的做法是：

1. 异常指令执行时，只把 exception 标记写进 ROB。
2. 等它走到 ROB head。
3. 停止提交后续指令。
4. 写 CSR/trap 状态并 redirect 到 trap vector。

### 3.4 为什么 store 通常到 commit 才对外可见

store 改变 memory，是架构可见副作用。为了精确异常，很多设计让 store 在 commit 时才真正写入 D-cache 或 store buffer。

如果 store 在执行阶段就对外可见，而更老指令后来发生异常，就无法回滚这个 store，精确异常会被破坏。

store 在乱序核里通常拆成两步：

```text
execute stage:
  compute store address
  prepare store data
  check permission/alignment
  write store queue entry

commit stage:
  when store reaches ROB head and is safe
  drain store queue entry to D-cache/store buffer/MMIO
```

对 normal cacheable memory，store 进入 store buffer 后可能还没真正写到下级 memory，但它已经按内存模型成为可管理的 older store。对 MMIO/device store，通常还要更保守：不能在 wrong-path 发出，不能被合并，不能越过 FENCE 或相关顺序点。

## 第4章 Reservation Station / Issue Queue：等待、唤醒和选择

### 4.0 RS/IQ 解决什么

rename 后的指令不一定立刻能执行，因为：

- 源操作数未 ready。
- 功能单元忙。
- load/store 需要等待地址或顺序。
- 多周期单元有限。

RS/IQ 负责保存这些等待执行的指令。

### 4.1 entry 内容

一个 IQ entry 通常包含：

```text
valid
op type
src0 physical tag + ready + value/need-read
src1 physical tag + ready + value/need-read
dest physical tag
ROB index
age / sequence
branch mask
immediate / control bits
```

### 4.2 wakeup

当执行单元写回结果时，会广播目的物理寄存器 tag：

```text
writeback: tag = p17, data = result
```

IQ 中所有等待 `p17` 的 entry 被唤醒。这个过程通常用 CAM 比较 tag。

wakeup 可以发生在“数据真正写回 PRF”同拍，也可以提前一拍预测结果即将可用。前者简单可靠，后者有利于缩短依赖链 latency，但会引入 replay 风险。比如 load hit 预测为一拍返回，结果 D-cache 发现 miss，则已经被提前唤醒的消费者必须被取消或重放。

| wakeup 类型 | 数据是否确定可用 | 优点 | 风险 |
|---|---|---|---|
| non-speculative wakeup | 是 | 控制简单 | 依赖链多等一拍 |
| speculative wakeup | 不一定 | 性能更好 | miss/仲裁失败时需要 replay |

入门设计建议先讲 non-speculative wakeup，把正确性闭环讲清；项目里如果用了 speculative wakeup，要说明如何检测失败和恢复。

### 4.3 select

ready entry 可能很多，但功能单元数量有限。select 逻辑负责选择本周期发射谁。

常见策略：

- oldest-first：优先最老指令，减少饥饿。
- round-robin：简单公平。
- per-FU queue：按功能单元分队列。
- unified IQ：统一队列，更灵活但更复杂。

select 还要匹配功能单元类型。一个 ready 的 multiply 指令不能发给 ALU，一个 load 不能绕过 LSU 的地址/顺序检查。实际发射条件通常是：

$$
issue\_eligible = valid \land srcs\_ready \land fu\_available \land \neg blocked\_by\_ordering
$$

其中 `blocked_by_ordering` 对 ALU 通常为 0，对 load/store、FENCE、AMO、CSR 等序列化指令可能很关键。

### 4.4 wakeup-select 是关键路径

高性能乱序核里，wakeup-select 很容易成为 critical path：

```text
result tag broadcast -> CAM compare -> ready update -> select -> issue
```

为了提高 Fmax，常见做法：

- 分簇执行。
- 分离整数/浮点/访存队列。
- 提前选择 speculative select。
- 限制 issue width 和 IQ size。

## 第5章 执行后端

### 5.0 执行后端包含什么

执行后端负责真正计算结果。常见功能单元包括：

- ALU：整数加减、逻辑、比较。
- AGU：地址生成。
- LSU：load/store。
- MDU：乘除法。
- FPU：浮点。
- vector unit：向量运算。
- branch unit：分支解析。

### 5.1 单周期和多周期单元

| 单元 | 延迟 | 设计要点 |
|---|---|---|
| ALU | 常见 1 cycle | 旁路和频率关键 |
| branch unit | 常见 1 cycle | redirect 和恢复关键 |
| AGU | 常见 1 cycle | 与 D-cache/TLB 耦合 |
| multiplier | 多 cycle 或流水 | 结果写回仲裁 |
| divider | 多 cycle / 迭代 | 可能阻塞或独立排队 |
| FPU | 多级流水 | rounding、exception flags |

### 5.2 result bus 和 writeback

执行结果要写回 PRF，并广播 tag 唤醒依赖者。问题在于：

- 多个 FU 同时完成。
- PRF 写端口有限。
- result bus 带宽有限。
- bypass 网络复杂。

因此需要 writeback arbitration。如果写回端口不够，某些完成指令要等待，ROB 里的 completed 标记也不能提前乱置。

writeback 在乱序核里通常是“完成事件”的真正边界：

```text
FU produces result
  -> arbitrate result bus / PRF write port
  -> write PRF
  -> mark phys reg ready
  -> mark ROB entry done
  -> wakeup dependent IQ entries
```

如果 FU 计算完成但没抢到 writeback 端口，不能提前把目的物理寄存器标 ready，否则消费者可能读到旧值或 X。对多周期单元尤其重要：完成队列、writeback arbiter、ROB done bit 和 busy table ready bit 必须保持一致。

### 5.3 bypass 网络

旁路用于减少写回后再读 PRF 的延迟：

```text
FU result -> dependent FU input
```

乱序核旁路比五级流水线复杂得多，因为生产者和消费者可能来自不同队列、不同功能单元和不同 pipeline latency。

旁路网络的规模会随发射宽度和功能单元数量快速增长。粗略地说，若有 $P$ 个生产者端口和 $C$ 个消费者输入端口，全连接 bypass 需要接近：

$$
N_{bypass\_paths} \approx P \times C
$$

这还没算数据位宽、控制选择和物理布线。实际设计常用分簇、限制跨簇旁路、增加一拍转发延迟等方式控制时序。面试中可以这样表达：乱序核不是“端口越多越好”，宽度增加会让 PRF、bypass、wakeup-select 和物理布线一起变难。

### 5.4 后端资源和 IPC

理论 IPC 受多个因素限制：

$$
IPC \le \min(fetch\_width, decode\_width, rename\_width, issue\_width, commit\_width, FU\_throughput)
$$

但真实 IPC 还会被 cache miss、branch mispredict、TLB miss、依赖链和内存顺序限制。

## 第6章 LSQ、内存乱序和存储相关性

### 6.0 为什么 load/store 更难乱序

ALU 指令只读写寄存器，rename 后依赖清楚。load/store 读写 memory，地址可能要执行后才知道。两个内存操作是否相关，很多时候必须等地址出来才能判断。

### 6.1 LSQ 的职责

LSQ 通常分 load queue 和 store queue：

- 跟踪所有未提交 load/store。
- 保证必要的内存顺序。
- 做 store-to-load forwarding。
- 检测 load violation。
- 管理 store commit。

LSQ 是乱序核里最接近“真实世界”的结构，因为 memory 没有像寄存器那样被完全重命名。一个 store queue entry 通常保存：

```text
valid
rob_idx
virtual/physical address
addr_ready
data
data_ready
byte_mask
memory attribute
exception
committed/drain_ready
```

load queue 则要保存 load 地址、数据返回状态、是否从 store forwarding 得到数据、是否需要 replay。LSQ 不只是排队，它还要回答两个问题：这个 load 能不能执行？这个 store 什么时候能对外可见？

### 6.2 store-to-load forwarding

如果 younger load 读取的地址命中 older store，而 store 数据已经 ready，可以直接从 store queue 前递：

```text
older store: addr=A, data=D
younger load: addr=A
load result = D
```

注意字节掩码和部分重叠是最容易错的地方。

对不同宽度访问，forwarding 不是简单地址相等：

| 场景 | 是否可直接 forwarding | 说明 |
|---|---|---|
| store word，load word，同地址 | 可以 | 全覆盖 |
| store word，load byte/half，同地址范围内 | 可以 | 选择对应 byte lane |
| store byte，load word，同地址 | 不完整 | 需要合并其他 bytes 或等待 memory |
| 多个 older store 覆盖一个 load | 复杂 | 按年龄选择最近 store 并做 byte merge |
| store 地址未知 | 不能确定 | load 可能等待或预测无冲突 |

因此 LSQ 常需要地址 CAM、byte mask 比较和年龄比较。store-to-load forwarding 的 bug 往往表现为随机数据错，而且只在特定字节宽度和特定 store/load 距离下出现。

### 6.3 memory disambiguation

为了性能，处理器可能让 load 先于地址未知的 older store 执行。如果后来发现 older store 地址和该 load 冲突，就发生 load violation，需要 replay 或 flush 相关指令。

保守策略：

- load 等所有 older store 地址都 ready。
- 正确简单，但性能差。

激进策略：

- load 预测无冲突，先执行。
- 性能好，但需要违例检测和恢复。

### 6.4 FENCE、AMO、LR/SC

乱序核必须特别处理：

- `FENCE`：约束前后内存操作顺序。
- `AMO`：原子读改写。
- `LR/SC`：reservation 成功/失败条件。
- MMIO/device memory：不可随意乱序、合并或推测。

这些指令常被作为序列化点处理，至少要让相关 older memory operation 达到规定完成点。

一个保守但清晰的策略是：

```text
FENCE reaches issue/commit boundary:
  wait all older loads resolved
  wait all older stores drained to required visibility point
  block younger memory operations until fence completes
```

AMO 和 LR/SC 还要求原子性或 reservation 语义。它们不能被拆成普通 load + store 后随便乱序执行。若实现 cache coherence，AMO/LR/SC 还会和 coherence 状态、reservation granule、外部 invalidation 交互；入门项目可以先把它们设计成 LSU 的序列化操作。

## 第7章 Speculation、分支恢复和异常恢复

### 7.0 推测执行的边界

乱序核离不开 speculation：

- 分支方向和目标推测。
- load/store 无冲突推测。
- cache/TLB miss 下的继续执行。

但推测结果不能提前成为架构状态。错误路径上的结果只能存在于 PRF、ROB、IQ 等可回滚结构里。

可以把乱序核中的状态分成三类：

| 状态 | wrong-path 能否修改 | 恢复方式 |
|---|---|---|
| 架构状态：GPR/CSR/committed memory | 不能 | 只能在 commit 更新 |
| 可回滚微架构状态：RAT/free list/IQ/ROB/LSQ | 可以 | checkpoint、ROB walk、valid kill |
| 性能状态：predictor/cache/TLB 部分状态 | 视设计而定 | 可接受污染或提供恢复/失效 |

这个分类能帮助回答很多追问：为什么 store 不能早发？为什么 predictor 可以被错路污染但 GPR 不行？为什么 TLB refill 在错路上是否允许要看实现？

### 7.1 分支 checkpoint

分支进入 rename 时建立 checkpoint。通常保存：

- RAT。
- free list 分配状态。
- branch predictor 历史。
- RAS 状态。
- 可能还有 store/load 推测状态。

如果分支预测正确，checkpoint 可释放。如果错误，用 checkpoint 恢复 rename 状态，并 flush younger 指令。

### 7.2 ROB-based recovery

另一种恢复方式是从 ROB 中回滚 younger 指令：

- 精确但可能慢。
- 不需要为每个分支保存完整 RAT。
- 对深流水和大窗口恢复代价较高。

实际设计可能混合使用 checkpoint 和 ROB walk。

### 7.3 异常恢复

异常不是马上进入 trap，而是写入 ROB entry，等到成为 ROB head 时再处理。这样可以保证：

- 更老指令已提交。
- 更年轻指令被清掉。
- CSR/trap 状态和 `epc/cause/tval` 精确。

异常和分支错误同拍发生时，要按指令年龄处理。常见原则：

```text
older exception beats younger branch mispredict
older branch mispredict kills younger exception
exception only traps when it reaches ROB head
```

如果 younger wrong-path load 产生 page fault，但更老 branch 后来发现预测错，这个 page fault 不应进入 trap，因为那条 load 本来就不该存在于正确路径上。ROB 中的 valid/branch mask/kill 标记必须保证这一点。

### 7.4 wrong-path 副作用

wrong-path 指令不允许：

- commit。
- 更新架构寄存器。
- 对外可见 store。
- 更新 CSR。
- 触发不可回滚 MMIO 写。

预测器这类微架构状态是否允许 wrong-path 更新，要看设计策略。若允许，必须能接受污染或提供恢复机制。

## 第8章 RTL 结构骨架

### 8.0 ROB entry 示例

```systemverilog
typedef struct packed {
  logic        valid;
  logic        done;
  logic [31:0] pc;
  logic [4:0]  arch_rd;
  logic [6:0]  phys_rd;
  logic [6:0]  old_phys_rd;
  logic        has_exception;
  logic [5:0]  exception_cause;
  logic        is_branch;
  logic        branch_mispredict;
  logic        is_store;
} rob_entry_t;
```

### 8.1 ROB commit 骨架

```systemverilog
always_comb begin
  commit_valid = 1'b0;
  commit_trap  = 1'b0;

  if (rob[head_q].valid && rob[head_q].done) begin
    if (rob[head_q].has_exception) begin
      commit_valid = 1'b0;
      commit_trap  = 1'b1;
    end else begin
      commit_valid = 1'b1;
    end
  end
end
```

多 commit width 设计中，要从 head 开始连续检查，遇到第一个未完成、异常或序列化指令就停止。

commit 还要产生统一的架构提交包，用于写 CSR、释放资源、驱动 store drain 和验证 trace：

```text
commit_packet:
  valid
  pc
  inst
  arch_rd
  phys_rd
  old_phys_rd
  writes_gpr
  is_store
  has_exception
  exception_cause/tval
```

验证中的 commit trace 通常也来自这里。乱序核内部执行顺序再复杂，commit packet 必须像一台顺序 ISA 机器一样逐条输出。

### 8.2 Rename 骨架

```systemverilog
always_comb begin
  src0_phys = rat[rs1_arch];
  src1_phys = rat[rs2_arch];
  old_dst_phys = rat[rd_arch];
  new_dst_phys = free_list_head;
end

always_ff @(posedge clk or negedge rst_n) begin
  if (!rst_n) begin
    // 初始化 RAT，使架构寄存器映射到初始物理寄存器
  end else if (rename_fire && rd_arch != 5'd0) begin
    rat[rd_arch] <= new_dst_phys;
    busy_table[new_dst_phys] <= 1'b1;
  end
end
```

这个骨架还缺少真实 rename 必须有的 backpressure 条件：

```text
rename_can_fire =
  decode_valid
  && rob_has_space
  && iq_has_space
  && lsq_has_space_if_mem_op
  && free_list_has_enough_regs
```

如果这些条件漏掉，最常见结果是覆盖 ROB tail、IQ entry 溢出、free list 下溢或 load/store 没有 LSQ entry 却继续进入后端。

### 8.3 Issue queue wakeup 示例

```systemverilog
for (genvar i = 0; i < IQ_ENTRIES; i++) begin : gen_wakeup
  always_ff @(posedge clk or negedge rst_n) begin
    if (!rst_n) begin
      iq[i].src0_ready <= 1'b0;
      iq[i].src1_ready <= 1'b0;
    end else if (wakeup_valid) begin
      if (iq[i].valid && !iq[i].src0_ready && (iq[i].src0_tag == wakeup_tag))
        iq[i].src0_ready <= 1'b1;
      if (iq[i].valid && !iq[i].src1_ready && (iq[i].src1_tag == wakeup_tag))
        iq[i].src1_ready <= 1'b1;
    end
  end
end
```

### 8.4 Store commit 骨架

```systemverilog
assign store_can_drain =
  rob[head_q].valid &&
  rob[head_q].done  &&
  rob[head_q].is_store &&
  store_queue_head_addr_ready &&
  store_queue_head_data_ready;
```

真实设计还要处理 cache miss、store buffer、FENCE、AMO、MMIO 和异常恢复。

## 第9章 验证方法

### 9.0 验证目标

乱序核验证的核心是：

```text
内部可以乱，但 commit trace 必须像一个正确的顺序 ISA 机器。
```

验证要同时覆盖：

- rename 映射正确。
- ROB 顺序提交。
- IQ wakeup/select 正确。
- 执行结果写回 PRF 正确。
- LSQ 内存顺序正确。
- 分支和异常恢复正确。
- wrong-path 不产生架构副作用。

### 9.1 directed test

| 场景 | 关注点 |
|---|---|
| RAW 链 | 消费者等待生产者 wakeup |
| WAW 重命名 | 后写不覆盖前写的物理寄存器 |
| WAR 重命名 | 后写不破坏前读 |
| ROB full | 前端反压正确 |
| IQ full | dispatch 停顿正确 |
| 多 FU 同时写回 | writeback 仲裁正确 |
| branch mispredict | checkpoint 恢复正确 |
| precise exception | 异常到 ROB head 才 trap |
| load-store same address | store-to-load forwarding |
| load violation | replay 或 flush 正确 |
| FENCE | 等待 older memory operation |
| MMIO store | 不在 wrong-path 对外可见 |

### 9.2 random test 与 ISS

乱序核必须使用 commit trace 与 ISS/reference model 比对：

- commit PC。
- commit instruction。
- commit architectural register write。
- commit memory effect。
- commit exception。
- commit privilege / CSR。

不能用内部执行顺序和 ISS 比，因为内部本来就是乱的。

乱序核 random test 还需要强约束“可复现”。建议记录：

```text
random seed
commit sequence number
commit packet
branch recovery events
exception/trap events
cache/TLB miss injection
LSQ replay events
```

定位失败时先找第一条 commit mismatch。如果没有 mismatch 但机器卡死，则看 ROB/IQ/LSQ/free list 是否存在循环反压：例如 ROB head 等 store drain，store drain 等 D-cache miss，D-cache miss 等 miss queue 空间，而 miss queue 又被错误路径请求占满。

### 9.3 SVA 示例

#### 9.3.1 commit 必须从 ROB head 开始

```systemverilog
// 不可综合：验证断言
property p_commit_from_head;
  @(posedge clk) disable iff (!rst_n)
    commit_valid |-> (commit_rob_idx == rob_head_q);
endproperty

assert property (p_commit_from_head);
```

#### 9.3.2 wrong-path 不提交

```systemverilog
// 不可综合：验证断言
property p_killed_inst_no_commit;
  @(posedge clk) disable iff (!rst_n)
    killed_rob_valid |-> !commit_killed_rob;
endproperty

assert property (p_killed_inst_no_commit);
```

#### 9.3.3 释放物理寄存器不能仍被 RAT 引用

```systemverilog
// 不可综合：验证断言，函数仅示意
property p_free_reg_not_mapped;
  @(posedge clk) disable iff (!rst_n)
    free_list_push_valid |-> !is_mapped_in_rat(free_list_push_reg);
endproperty

assert property (p_free_reg_not_mapped);
```

### 9.4 coverage

覆盖建议：

- rename：有目的寄存器/无目的寄存器、x0、free list 接近空。
- ROB：空、半满、满、多提交、异常 head。
- IQ：单源 ready、双源 ready、等待 wakeup、多路竞争。
- FU：单周期、多周期、同时完成、写回冲突。
- branch：预测对、预测错、嵌套分支、多 checkpoint。
- LSQ：load 先行、store forwarding、load violation、FENCE。
- recovery：branch recovery、exception recovery、trap recovery。

还应覆盖关键交叉项：

| 交叉场景 | 为什么重要 |
|---|---|
| branch mispredict + free list 接近空 | recovery 后资源计数容易错 |
| exception + younger store ready | store 不能对外可见 |
| load violation + dependent chain | replay 不能漏掉依赖者 |
| writeback conflict + ROB commit | done bit 不能早置 |
| FENCE + store buffer 非空 | younger memory op 必须被挡住 |
| MMIO load/store + speculation | device side effect 不能发生在 wrong-path |

## 第10章 时序、综合、后端和 PPA 影响

### 10.0 最大时序热点

乱序核最典型的 critical path：

- rename 多读多写 RAT。
- free list 多分配多释放。
- wakeup-select。
- PRF 多端口读写。
- bypass 网络。
- LSQ 地址 CAM 比较。
- ROB 多提交扫描。

### 10.1 宽度和窗口大小权衡

| 参数 | 增大带来的好处 | 代价 |
|---|---|---|
| fetch/decode width | 前端供给更强 | 分支预测和解码复杂 |
| rename width | 每周期更多指令进入窗口 | RAT/free list 多端口 |
| ROB size | 容忍更长延迟 | 面积、功耗、恢复复杂 |
| IQ size | 暴露更多 ILP | wakeup-select 更慢 |
| issue width | 并行执行更多 | FU、PRF、bypass 复杂 |
| commit width | 更快退休 | ROB head 扫描复杂 |

### 10.2 分簇设计

为了控制时序和布线，执行后端可能分簇：

- 每个 cluster 有自己的 IQ、FU、局部 bypass。
- 跨 cluster 转发需要额外 latency。
- 编译器或硬件调度要尽量把相关指令放同簇。

这是高性能设计常见的 PPA 折中。

### 10.3 物理设计影响

- PRF 和 bypass 网络要靠近执行单元。
- ROB 和 RAT 靠近 rename/commit 控制。
- LSQ 靠近 D-cache/TLB。
- 广播 tag 和结果的总线要避免跨全核长距离乱拉。

### 10.4 功耗影响

乱序核功耗很容易高，原因包括：

- CAM 比较多。
- predictor、IQ、ROB、LSQ 每周期大量读写。
- PRF 多端口。
- bypass 网络切换活动大。

常见优化：

- 分银行。
- clock gating。
- operand gating。
- 分级调度。
- 限制无效 wakeup 广播。

## 第11章 常见 bug、边界条件和 debug 方法

### 11.0 常见 bug

| bug | 现象 | 修复方向 |
|---|---|---|
| RAT 恢复不完整 | 分支错后寄存器值乱 | checkpoint 保存/恢复完整映射 |
| free list 释放过早 | 旧值被覆盖 | 旧物理寄存器 commit 后再释放 |
| ROB done 提前置位 | 未完成指令提交 | 写回仲裁后再标记 completed |
| wakeup tag 比较错 | 消费者永远不发射或读错值 | CAM 比较加 valid 和 tag 宽度检查 |
| x0 被重命名 | x0 读到非零 | x0 不分配物理寄存器或映射到固定零寄存器 |
| store wrong-path 对外可见 | 分支错后内存被写坏 | store commit 后才 drain |
| load violation 没恢复 | 读到旧内存值 | LSQ 检测冲突并 replay/flush |
| exception 提前 trap | 更老指令未提交 | exception 等到 ROB head |
| 多写回端口仲裁丢结果 | 随机寄存器错 | FU completion 与 writeback ready 握手 |
| FENCE 当普通指令 | memory order 错 | 序列化相关 memory operation |

### 11.1 边界条件

- ROB 几乎满时 branch mispredict。
- free list 只剩 0/1 个寄存器时多宽 rename。
- 同周期 commit 和 allocate。
- 同周期 wakeup 和 issue。
- load miss 期间更年轻分支 mispredict。
- exception 和 branch mispredict 同拍。
- MMIO load/store 与 speculation。

### 11.2 debug 方法

建议按 commit trace 定位第一条错误：

1. 如果 commit PC 错，先看 branch recovery 和 ROB head。
2. 如果寄存器值错，查 RAT、PRF、wakeup、writeback。
3. 如果内存值错，查 LSQ、store queue、store forwarding。
4. 如果异常错，查 ROB exception 标记和 commit 点。
5. 如果卡死，查 ROB/IQ/LSQ/free list 是否互相反压。

### 11.3 波形观察点

```text
rename_valid, dispatch_valid, issue_valid, wb_valid, commit_valid
rat[rs/rd], free_list_head, phys_src/dst, old_phys_dst
rob_head, rob_tail, rob_full, rob_entry.done/exception
iq_valid, src_ready, wakeup_tag, select_grant
lsq_head/tail, load_addr, store_addr, forwarding_hit
branch_checkpoint, mispredict, recovery_valid
```

## 第12章 面试问法

### 12.0 基础题

#### 1. 乱序执行和顺序提交分别是什么意思

简洁答法：

```text
乱序执行是指指令可以按操作数 ready 和资源可用情况乱序发射、执行和完成；
顺序提交是指最终架构状态仍按程序顺序更新，从而保证精确异常和软件可见行为。
```

#### 2. ROB 的作用是什么

答题要点：

- 保存程序顺序。
- 记录未提交指令状态。
- 支持精确异常。
- 支持分支恢复和顺序 commit。

#### 3. register renaming 解决什么

答题要点：

- 消除 WAR/WAW 名字相关。
- 让独立指令并行执行。
- RAW 真相关仍需等待。

### 12.1 进阶追问

#### 1. 为什么 store 通常到 commit 才对外可见

要点：

- store 是不可轻易回滚的架构副作用。
- 如果更老指令后来异常，提前 store 会破坏精确异常。
- 所以 store 通常执行阶段算地址/数据，commit 后才 drain。

#### 2. wakeup-select 为什么难做

要点：

- 结果 tag 广播到所有 IQ entry。
- CAM 比较后更新 ready。
- 同周期还要选择 oldest ready 指令发射。
- 宽度和队列越大，时序越难收敛。

#### 3. 分支预测错误怎么恢复

要点：

- flush younger wrong-path 指令。
- 恢复 RAT/free list/checkpoint。
- 释放或无效化错误路径资源。
- redirect PC 到正确目标。

#### 4. precise exception 在乱序核里怎么保证

要点：

- 异常先记录在 ROB entry。
- 等异常指令到 ROB head。
- 提交更老指令，清掉更年轻指令。
- 再进入 trap。

### 12.2 项目追问

#### 1. 如果 ROB 满了怎么办

回答框架：

- 停止 dispatch/rename。
- 前端反压。
- 等 commit 释放 ROB entry。
- 不能覆盖未提交指令。

#### 2. 如果 free list 空了怎么办

回答框架：

- 不能继续 rename 有目的寄存器的指令。
- 可以等待 commit 释放旧物理寄存器。
- 对无目的寄存器指令是否允许继续，要看设计是否支持部分分派。

#### 3. load 可以越过 store 吗

回答框架：

- 若 older store 地址已知且不冲突，可以。
- 若地址未知，保守设计等待，激进设计可预测无冲突。
- 若后来发现冲突，需要 replay/flush。

## 第13章 练习题与答案要点

### 13.1 练习题 1：判断相关类型

题目：

```text
ADD x5, x1, x2
SUB x5, x3, x4
OR  x6, x5, x7
```

指出相关类型。

答案要点：

- `ADD` 和 `SUB` 对 `x5` 是 WAW。
- `SUB` 和 `OR` 对 `x5` 是 RAW。
- rename 后 WAW 消失，OR 仍依赖 SUB 的物理寄存器。

### 13.2 练习题 2：为什么 ROB 能保证精确异常

答案要点：

- ROB 按程序顺序排列指令。
- 异常只在 ROB entry 标记。
- 只有异常指令到 head 时才 trap。
- 更年轻指令还未 commit，可安全清除。

### 13.3 练习题 3：设计 free list 释放时机

题目：

```text
一条写 x10 的指令 rename 时分配 p40，旧映射是 p12。什么时候释放 p12？
```

答案要点：

- 当这条指令 commit 后，x10 的架构状态已经确定为 p40。
- 此时旧映射 p12 不再是架构可见最新值，可以释放。
- 不能在 rename 时立即释放，否则遇到分支恢复或异常会丢旧值。

### 13.4 练习题 4：load violation 场景

题目：

```text
更年轻 load 先执行，后来更老 store 地址算出与它相同。怎么办？
```

答案要点：

- 该 load 可能读错旧值。
- 需要 replay 该 load 及其依赖，或 flush 到安全点重新执行。
- LSQ 要能检测 older store 与 younger load 地址冲突。

### 13.5 练习题 5：计算理论 IPC 上限

题目：

```text
fetch/decode/rename/issue/commit 宽度分别是 4/4/4/3/4，功能单元每周期最多完成 3 条，理论 IPC 上限是多少？
```

答案：

$$
IPC \le \min(4, 4, 4, 3, 4, 3) = 3
$$

实际 IPC 还会低于 3，因为有依赖、miss、mispredict 和资源冲突。

### 14.2 本篇总结

高级微架构不是一堆名词，而是一套严密的状态管理系统：

- rename 让名字相关变成物理寄存器依赖。
- ROB 让乱序执行仍能顺序提交。
- IQ/RS 让 ready 的指令及时发射。
- 执行后端提供并行计算能力。
- LSQ 管住 memory 的真实依赖和顺序。
- checkpoint/recovery 保证推测失败后能回到正确状态。

面试里能把这条链路从 fetch 讲到 commit，再补上精确异常、store commit、分支恢复和验证思路，基本就能体现出对高性能 CPU 微架构的系统理解。
