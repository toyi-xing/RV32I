## 第0章 本专题学习地图

### 0.0 这篇文档到底解决什么

CSR、异常、中断和特权级，是处理器从“会算”走向“能跑操作系统、能接外设、能被软件可靠控制”的分界线。

如果只会 RV32I/RV64I 基础指令，你只能解释寄存器和算术逻辑；一旦要支持操作系统、定时器、中断控制器、用户态程序、系统调用、调试和异常恢复，就必须把下面几件事串起来：

- 软件能读写哪些 CSR。
- 什么事件会进入 trap。
- trap 进入哪个 privilege。
- `mepc/sepc` 记录什么。
- `mcause/scause` 如何编码原因。
- `mtvec/stvec` 如何选择入口。
- `mstatus/sstatus` 如何保存和恢复中断态与上层权限。
- `mideleg/medeleg` 如何把一部分 trap 委派给更低 privilege。
- `MRET/SRET` 如何把 PC 和 privilege 栈恢复回来。

## 第1章 特权级、Trap 和软件栈

当前阶段建议：简单了解本章，重点知道特权级和 trap 是真实 CPU/MCU/SoC 的控制基础，但最小 RV32I 教学核可以先不实现。

### 1.0 为什么特权级是 CPU 的骨架

没有特权级，任何软件都能直接改时钟、中断、页表、外设和调试状态。那样的 CPU 只能跑极简裸机程序，难以承载操作系统，也很难形成稳定的软件分层。

RISC-V 的 privileged architecture 要解决的第一件事是：**同一颗 CPU 上运行的软件并不都应该拥有同样的权限**。用户程序可以做加减乘除、访问自己被允许的内存；操作系统内核可以管理页表、调度任务、配置设备；最底层固件还要负责启动、复位、平台中断和一些机器级资源。

所以 privilege 的本质不是“多几个寄存器名”，而是给硬件加上一条判断线：

```text
当前这条指令、这次访存、这次 CSR 访问，在当前权限下是否合法？
```

最常见的三层是：

| 名称 | 权限定位 | 典型软件 |
|---|---|---|
| `M-mode` | 机器级，权限最高，能管理平台最底层资源 | boot ROM、固件、SBI、裸机运行时 |
| `S-mode` | 监管级，面向操作系统内核 | Linux kernel、RTOS kernel 的特权部分 |
| `U-mode` | 用户级，权限最低，面向普通应用 | app、用户进程、受限制的任务 |

你可以先把它理解成“CPU 内部有一个当前权限状态 `priv_q`”。decode、CSR 文件、访存权限检查、中断进入条件都会看这个状态。

### 1.1 privilege level 和 privilege mode

这里容易混乱，是因为中文里常把 privilege level 和 privilege mode 都说成“特权级”。在学习阶段可以这样区分：

| 概念 | 主要回答的问题 | 硬件里怎么看 | 例子 |
|---|---|---|---|
| privilege level | 权限高低关系是什么 | 用来比较“谁比谁权限高” | M 高于 S，S 高于 U |
| privilege mode | 当前 hart 正在以什么权限执行 | 当前状态寄存器/状态机的值 | `priv_q = M/S/U` |

因此，`M-mode/S-mode/U-mode` 里的 `mode` 不是另一个和 `level` 平行的新概念，而是“某个 privilege level 对应的当前执行模式”。更直接地说：

```text
privilege level 更强调权限层级关系。
privilege mode 更强调 hart 当前真的处在哪个执行状态。
M-mode 就是当前处于 M 这个 privilege level 对应的执行模式。
```

类比一下：

- privilege level = 公司里的职位等级：管理员 / 主管 / 普通员工。
- privilege mode = 你当前刷卡进入系统后，系统实际按哪个身份给你放行。

大多数入门场景里，这两个词可以近似合在一起理解：当前是 `M-mode`，就拥有 M 级权限；当前是 `U-mode`，就只能做 U 级允许的事。后续遇到虚拟化、仿真、debug 等更复杂系统时，才会更严格地区分“软件自认为的层级”和“硬件当前实际运行的 mode”。

常见组合如下，说的是一颗 CPU 实现了哪些 mode，而不是说 level 和 mode 可以随便配对：

| 硬件实现的 mode | 典型用途 | 直觉理解 |
|---|---|---|
| `M` | 简单嵌入式系统、极小 MCU | 所有代码都在最高权限跑，软件分层少 |
| `M + U` | 需要隔离用户任务的嵌入式系统 | 固件/运行时在 M，普通任务在 U |
| `M + S + U` | 运行 Unix-like OS 的系统 | M 管平台，S 跑 OS 内核，U 跑应用 |

后面说“trap 进入 M-mode”或“trap 被委派到 S-mode”，意思就是：硬件把当前 `priv_q` 切到对应 mode，并按那一级的 CSR 规则写 `mepc/mcause` 或 `sepc/scause`。

### 1.2 trap 为什么和特权级有关

trap 可以先通俗理解成：**当前正常执行流被某个事件打断，CPU 保存必要现场，然后把 PC 跳到一段专门的处理程序**。这段处理程序叫 trap handler。

它和普通 branch/jump 都会改变 PC，但语义不一样：

| 控制流变化 | 触发原因 | 跳转目标来自哪里 | 是否需要记录原因 |
|---|---|---|---|
| branch/jump | 程序主动写出的控制流 | 指令立即数或寄存器 | 通常不需要 |
| trap | 异常、中断、系统调用、非法操作等事件 | `mtvec/stvec` 等 CSR 指定的入口 | 需要写入 `mepc/sepc`、`mcause/scause`、`mtval/stval` 等 CSR |

trap 之所以必须和 privilege 绑定，是因为很多 trap 不是“普通程序自己处理一下”就能解决的事情，而是需要更高权限的软件介入。

| 场景 | 为什么不能只让当前程序自己处理 |
|---|---|
| `U-mode` 程序执行 `ECALL` (环境调用) | 用户程序想请求操作系统服务，例如文件、进程、内存管理，需要进入内核 |
| `U-mode` 访问受保护地址 | 用户程序不能自己决定是否有权限访问，需要 OS 检查页表和权限 |
| 外设 interrupt 到来 | 普通应用不能直接操作中断控制器、设备寄存器和调度器 |
| 非法 CSR 访问 | 低特权级不能随便改 `mstatus/mtvec/mie` 这类控制状态 |

所以 trap entry 硬件通常要同时做几件事：

1. 记录被打断的位置，例如写 `mepc/sepc`。
2. 记录 trap 原因，例如写 `mcause/scause`。
3. 记录附加信息，例如 fault address 写入 `mtval/stval`。
4. 根据当前 privilege、委派配置和 trap 类型，选择进入 `M-mode` 还是 `S-mode` 的 handler。
5. redirect PC 到 `mtvec/stvec` 指定的入口，并 flush 掉不该继续提交的 younger instruction。

从 privilege 变化方向看，trap 又可以分成两类：

| 分类 | 含义 | 例子 |
|---|---|---|
| vertical trap | trap 后进入更高 privilege 的 handler | `U-mode` 执行 `ECALL`，进入 `S-mode` 或 `M-mode` |
| horizontal trap | trap 后仍由同一 privilege 的 handler 处理 | `S-mode` 发生 page fault，若 M-mode 已委派给 S-mode，则进入 S-mode 的 page fault handler |

这两个词不是最核心的硬件实现点，先记住直觉即可：trap 是一种“带原因记录、带权限切换规则的 PC redirect”。后面讲 `mstatus/sstatus`、`mepc/sepc`、`mcause/scause` 和 `MRET/SRET` 时，本质都是在解释 trap 进入和返回时状态怎么保存、权限怎么切换、PC 怎么恢复。

### 1.3 系统 OS 视角：trap 是进入内核或固件的硬件入口

这里涉及系统 OS，但本篇不讲完整 OS 怎么实现，只讲 OS/裸机运行时为什么需要这些硬件机制。

从软件栈看，trap 是低权限程序、外设事件或硬件错误进入高权限处理程序的统一入口：

| 软件场景 | trap 在系统里扮演的角色 | 硬件必须提供什么 |
|---|---|---|
| 用户程序执行 `ECALL` | 请求 OS 服务，例如系统调用 | 记录 `epc/cause`，切到更高 privilege 的 handler |
| timer interrupt 到来 | OS 获得周期性控制权，可做调度或超时处理 | 在指令边界接受 interrupt，避免破坏精确状态 |
| load/store 触发 page fault | OS 判断是非法访问还是需要建立映射 | 把 faulting PC 和 fault address 提供给 handler |
| 裸机程序访问非法外设地址 | 固件可以打印错误、复位或进入 debug | 报告 access fault，不能默默返回随机数据 |

因此，trap 和 privilege 绑定的根本原因是：很多事件的处理权不属于当前普通程序，而属于 OS 内核、机器模式固件或调试环境。硬件负责把“为什么进来、从哪里进来、进来前是什么权限”记录清楚；完整的寄存器保存、任务切换、系统调用分发等动作则是 trap handler 的软件责任。

### 1.4 面试中最重要的一句话

简洁版：

```text
特权级决定谁能访问哪些 CSR、外设和控制状态；trap 是把当前执行流切到更高权限的处理程序；返回指令再用保存的 EPC 和 privilege stack 恢复现场。
```

深入版：

```text
RISC-V 的 privileged architecture 本质上是一个带两级栈的控制流和状态保存机制。
异常和中断发生时，硬件要把 EPC、cause、tval、xIE、xPIE、xPP 等状态写进 CSR；
trap handler 做完处理后，用 xRET 指令把 PC 和 privilege 恢复回去。
```

## 第2章 CSR 的意义、寻址和访问语义

当前阶段建议：简单了解本章，重点看懂 CSR 和 GPR 的区别、CSR 指令的大致作用，完整 Zicsr 语义可等加入最小 trap/中断时再细读。

### 2.0 CSR 是什么

CSR(控制状态寄存器) 可以先理解成：**CPU 暴露给软件的一组控制面和状态面寄存器**。GPR 保存普通程序正在计算的数据，CSR 保存“这颗 hart 现在怎么运行、出错后去哪里、哪些中断打开、当前是什么权限、上一次 trap 的原因是什么”。

RISC-V 规范给每个 hart 定义了一块独立的 CSR 地址空间，CSR 地址是 12 bit，所以理论上有 $2^{12}=4096$ 个 CSR 编号。它不是 memory 地址，也不是 GPR 编号，而是 CSR 指令里专门携带的一段 CSR 地址。

常见用途可以这样看：

| CSR 类型 | 例子 | 谁会使用 | 解决什么问题 |
|---|---|---|---|
| trap 入口 | `mtvec/stvec` | 固件或 OS 初始化时写 | 告诉硬件发生 trap 后跳到哪里 |
| trap 现场 | `mepc/sepc`、`mcause/scause`、`mtval/stval` | 硬件 trap entry 写，handler 读 | 告诉软件从哪条指令进来、为什么进来、附加错误信息是什么 |
| 权限和中断状态 | `mstatus/sstatus`、`mie/sie`、`mip/sip` | 硬件和软件都会改 | 控制当前 privilege、中断开关、pending 状态 |
| 委派控制 | `medeleg/mideleg` | M-mode 固件或内核配置 | 决定哪些 trap 交给 S-mode 处理 |
| 计数和性能状态 | `cycle/instret` 等 | 软件读取 | 统计周期数、退休指令数或性能事件 |
| 内存管理相关 | `satp` 等 | OS 内核写 | 控制页表、地址翻译和虚拟内存 |

因此，CSR 的访问入口有两类：

1. **软件显式访问**：程序执行 `CSRRW/CSRRS/CSRRC` 这类 CSR 指令。
2. **硬件自动更新**：trap entry、xRET、中断 pending、计数器递增等硬件事件直接改 CSR。

这也是为什么 CSR 比普通寄存器麻烦：它不仅存数，还直接参与 CPU 控制流、权限检查、异常恢复和 OS 运行。

### 2.1 CSR 和 GPR 的区别

`0801` 里讲过 `x0` 到 `x31` 这组 GPR。GPR 和 CSR 都是软件能看见的架构状态，但用途完全不同。

| 对比项 | GPR | CSR |
|---|---|---|
| 主要用途 | 保存普通计算数据、地址、函数参数、返回值 | 保存 CPU 控制状态、异常状态、中断状态、权限状态 |
| 编号方式 | 指令中的 5 bit `rs1/rs2/rd` 选择 `x0` 到 `x31` | 指令中的 12 bit CSR 地址选择一个 CSR |
| 常见访问指令 | `ADD`、`LW`、`SW`、`ADDI` 等普通指令 | `CSRRW`、`CSRRS`、`CSRRC` 及立即数形式 |
| 读写权限 | 普通程序基本都能读写，`x0` 例外 | 受 privilege、只读位、实现选项限制 |
| 副作用 | 读 GPR 通常没有副作用；写 `x0` 被丢弃 | 读/写某些 CSR 可能有副作用或特殊规则 |
| 硬件写入来源 | 主要来自指令写回 | 指令、trap entry、xRET、中断控制、计数器都可能写 |
| 设计关注点 | 数据相关、forwarding、寄存器堆端口 | 权限检查、字段合法化、trap 优先级、精确异常 |

CSR 指令会同时用到 GPR 和 CSR，所以才容易让人混淆。比如：

```assembly
csrrw x5, mstatus, x6
```

这条指令不是“只操作 GPR”，也不是“只操作 CSR”，而是同时做两件事：

```text
x5      <- old mstatus
mstatus <- x6
```

这里 `x5` 是 GPR 目的寄存器，`x6` 是 GPR 源寄存器，`mstatus` 是 CSR。也就是说，`rd/rs1` 这些字段仍然是 GPR 字段，但这条指令的核心副作用发生在 CSR 上。

### 2.2 CSR 指令是什么

CSR 指令就是 RISC-V ISA 里专门访问 CSR 的指令，属于 `Zicsr` 扩展。它和 `ADD/LW/BEQ` 一样，仍然是一条正常的 CPU 指令：会被 IF 取指、ID 译码、进入流水线、在提交点更新架构状态。

区别在于它的操作对象不是 ALU 普通数据通路，而是 CSR 文件：

```text
普通 ALU 指令:
  GPR[rs1], GPR[rs2] -> ALU -> GPR[rd]

load 指令:
  GPR[rs1] + imm -> memory address -> GPR[rd]

CSR 指令:
  CSR[csr_addr] 和 GPR[rs1]/uimm -> CSR file -> CSR[csr_addr]，并可把旧 CSR 值写到 GPR[rd]
```

所以“CSR 指令”和“CPU 流水线里 EX 执行的指令”不是两个层次的东西。CSR 指令仍然在 CPU 流水线里执行，只是它通常会走一条控制状态访问路径，而不是普通 ALU 计算路径。简单五级流水线里，CSR 可以在 ID/EX 读、在 WB/commit 写；更严谨的实现会把 CSR 更新放到 commit 边界，避免被 flush 的 CSR 指令产生副作用。

### 2.3 CSR 的读、写和原子读改写语义

CSR 指令的核心语义常被称为 read-modify-write，也就是“读-改-写”：

1. 先读出旧 CSR 值。
2. 根据指令类型和操作数计算新 CSR 值。
3. 必要时把新 CSR 值写回 CSR。
4. 必要时把旧 CSR 值写入 GPR `rd`。

可以抽象成：

$$
\begin{aligned}
csr_{old} &= CSR[addr] \\
rd\_wdata &= csr_{old} \quad \text{if CSR read is enabled} \\
csr_{new} &= f(csr_{old}, operand) \\
CSR[addr] &\leftarrow legalize(csr_{new}) \quad \text{if CSR write is enabled}
\end{aligned}
$$

这里说的“原子读改写”，重点是“单条 CSR 指令的软件可见效果不可拆开”。软件不应该看到“旧值已经读出、但新值还没写回”的半截状态；如果这条 CSR 指令后来被 flush，也不能留下 CSR 写副作用。它不是这里重点讲的多核原子内存操作，只是“这条 CSR 指令自身读、改、写这几个动作在架构语义上是一个整体”。

`rd=x0`、`rs1=x0` 之所以会影响 CSR，是因为 CSR 指令同时连接 GPR 字段和 CSR 文件：

| 情况 | 直觉解释 | CSR 行为 |
|---|---|---|
| `rd=x0` | GPR 目的寄存器是恒零寄存器，旧 CSR 值没人接收 | 对 `CSRRW/CSRRWI`，规范允许不读 CSR，从而避免读副作用 |
| `rs1=x0` | GPR 源操作数数值为 0 | 对 `CSRRW`，仍然是把 0 写入 CSR；对 `CSRRS/CSRRC`，表示置位/清位掩码为 0，所以不写 CSR |
| `uimm=0` | 立即数字段为 0 | 对 `CSRRSI/CSRRCI`，表示掩码为 0，所以不写 CSR |

这和普通 GPR 的 `x0` 规则相关，但不是同一件事。普通指令里 `rd=x0` 通常只是“写回结果被丢弃”；CSR 指令里，`rd=x0` 还可能表示“不需要读旧 CSR 值”。普通指令里 `rs1=x0` 只是源操作数为 0；CSR 置位/清位类指令里，`rs1=x0` 同时意味着“掩码为 0，不需要写 CSR”。

为什么要这么细？因为有些 CSR 读写可能有 side effect。例如某些状态位读后清除、某些 pending 位写 1 清除、某些计数器或设备相关状态读写有特殊语义。即使入门小核没有实现复杂副作用，也应该按规范把“读使能”和“写使能”分清楚。

RTL 中读路径和写路径可能在不同 stage。若 CSR 在 ID 读、WB 写，而中间又来了 trap 更新同一个 CSR，就必须明确优先级和重放/kill 策略。简单核通常把 CSR 指令当成一种会写架构状态的指令，在 commit 点统一生效；若该指令被 flush，就不能写 CSR，也不能触发 CSR 读副作用。

### 2.4 常见 CSR 指令

下面表格按最常见的 RV32/RV64 汇编形式来理解即可：

| 指令 | 汇编格式 | 旧 CSR 是否写 `rd` | 新 CSR 如何产生 | 关键点 |
|---|---|---|---|---|
| `CSRRW` | `csrrw rd, csr, rs1` | `rd != x0` 时写 | `CSR_new = GPR[rs1]` | `rd=x0` 时不读 CSR；`rs1=x0` 时写 0 |
| `CSRRS` | `csrrs rd, csr, rs1` | 总是读旧 CSR，通常写 `rd` | 旧 CSR 与 `GPR[rs1]` 按位或 | `rs1=x0` 时不写 CSR，常用来只读 CSR |
| `CSRRC` | `csrrc rd, csr, rs1` | 总是读旧 CSR，通常写 `rd` | `CSR_new = CSR_old & ~GPR[rs1]` | `rs1=x0` 时不写 CSR，常用来只读 CSR |
| `CSRRWI` | `csrrwi rd, csr, uimm` | `rd != x0` 时写 | `CSR_new = zext(uimm)` | `rd=x0` 时不读 CSR |
| `CSRRSI` | `csrrsi rd, csr, uimm` | 总是读旧 CSR，通常写 `rd` | 旧 CSR 与零扩展后的 `uimm` 按位或 | `uimm=0` 时不写 CSR |
| `CSRRCI` | `csrrci rd, csr, uimm` | 总是读旧 CSR，通常写 `rd` | `CSR_new = CSR_old & ~zext(uimm)` | `uimm=0` 时不写 CSR |

这里的“通常写 `rd`”还要叠加 GPR 的基本规则：如果 `rd=x0`，GPR 写回结果被丢弃。但要注意，GPR 写回被丢弃不等于 CSR 访问一定没有发生；到底读不读 CSR、写不写 CSR，要按上表的 CSR 指令语义判断。

几个常见伪指令也能帮助建立直觉：

| 伪指令 | 等价形式 | 含义 |
|---|---|---|
| `csrr rd, csr` | `csrrs rd, csr, x0` | 只读 CSR，把旧值放入 `rd` |
| `csrw csr, rs1` | `csrrw x0, csr, rs1` | 只写 CSR，不需要旧值 |
| `csrs csr, rs1` | `csrrs x0, csr, rs1` | 按 `rs1` 掩码置位 CSR |
| `csrc csr, rs1` | `csrrc x0, csr, rs1` | 按 `rs1` 掩码清位 CSR |

### 2.5 CSR 的硬件实现代价

从 RTL 角度，CSR 访问不是“加一个寄存器数组”这么简单。要考虑：

- 地址译码。
- 权限检查。
- 只读位屏蔽。
- WARL 规范化。
- 读副作用。
- 写副作用。
- 与 trap/interrupt 状态更新的冲突。

对于小核，CSR 文件通常在 ID 或 EX 阶段完成访问；对于更复杂的核，可能需要把 CSR 的读路径和 trap 入口状态更新拆开，以免拉长关键路径。

CSR 文件更像一个带权限、字段语义和异步事件入口的控制面模块：

| 子功能 | 解决什么问题 | RTL 关键点 | 验证重点 |
|---|---|---|---|
| 地址译码 | 4096 个 CSR 编号空间中选择目标 | decode 表、只实现子集 | 未实现 CSR 产生 illegal instruction |
| privilege check | 当前模式是否允许访问 | CSR 地址高位、当前 privilege、counter enable | U/S/M 访问权限覆盖 |
| 字段更新 | RO/WARL/WLRL/W1C 等语义 | 字段级写函数 | 写非法值后的读回合法化 |
| trap update | trap 入口写 EPC/cause/tval/status | trap 写端口优先级 | 普通 CSR 写和 trap 同拍冲突 |
| interrupt state | `mie/mip/sie/sip` pending/enable | 外部事件置位、软件清位 | pending 不丢、清除优先级明确 |
| counter/timer | 周期计数、性能计数 | 宽计数器、读一致性 | 高低位读跨拍问题 |

这也是为什么 CSR 模块经常成为小 CPU 的控制关键路径。它同时连着 decode、writeback、trap controller、interrupt controller、debug 和性能计数器，扇出很大。工程上应尽量把“权限检查/非法判断”“读数据 MUX”“字段写入”和“trap 状态更新”分层，而不是堆成一个超大的 `always_comb`。

### 2.6 适用条件

不是所有 CSR 都必须实现全部特性：

- 只有 `M-mode` 的简单核，可以把很多 S 级 CSR 做成只读零或不可访问。
- 支持 OS 的核，至少要有 `mstatus/mie/mip/mtvec/mepc/mcause/mtval/medeleg/mideleg` 的完整基础功能。
- 如果支持 `S-mode`，则还要支持 `sstatus/sie/sip/stvec/sepc/scause/stval` 以及相关返回逻辑。

## 第3章 mstatus/sstatus：中断栈和权限栈

当前阶段建议：可以先跳过字段细节，只需知道 `mstatus/sstatus` 用来在 trap 进入和返回时保存/恢复中断使能与来源权限。

### 3.0 为什么状态要“栈式保存”

`mstatus/sstatus` 也是 CSR，但它们不是普通配置寄存器，而是 trap 进入和返回时最核心的“状态备份区”。

trap 不是普通跳转。普通 `JAL/JALR` 只需要改 PC；trap 还会改变 privilege，并且通常会临时关闭当前级中断。handler 进入后，硬件和软件必须能回答两个问题：

1. trap 前是从哪个 privilege 来的，返回时要回到哪里。
2. trap 前当前级中断是不是开着，返回时要不要恢复。

因此 RISC-V 采用了很小的一层硬件状态栈：

- `xIE`：当前 privilege 的全局中断使能。
- `xPIE`：陷入前的中断使能备份。
- `xPP`：陷入前的 privilege 备份。

这里的 `x` 可以是 `M` 或 `S`，表示“当前 trap 进入哪一级，就使用哪一级对应的状态字段”。进入 M-mode handler 时使用 `MIE/MPIE/MPP`；进入 S-mode handler 时使用 `SIE/SPIE/SPP`。

这套机制解决的是 nested trap 的最小现场保存问题。硬件不会替软件保存所有 GPR，也不会自动保存内存上下文；它只保存“能不能被再次打断”和“从哪一级返回”这两个控制状态。更完整的寄存器现场保存仍然是 trap handler 的软件责任。

```text
hardware saves:
  xEPC / xCAUSE / xTVAL
  xIE -> xPIE
  current privilege -> xPP

software handler saves:
  GPRs it needs
  stack frame
  OS-specific context
```

面试中如果被问“trap 时硬件保存现场吗”，要回答得准确：硬件只保存有限的控制现场，不保存完整软件上下文。

### 3.1 M 级状态

`mstatus` 可以理解成 M-mode 视角下的总状态寄存器。它里面字段很多，但第一次学习时先抓住下面三个：

- `MIE`：机器全局中断使能。
- `MPIE`：进入 trap 前的 `MIE` 备份。
- `MPP`：进入 trap 前的 privilege 级别。

这三个字段配合起来，完成“进入 trap 时关中断并记住旧状态，返回时恢复旧状态”：

| 字段 | trap entry 时做什么 | `MRET` 时做什么 | 直觉 |
|---|---|---|---|
| `MIE` | 清 0 | 从 `MPIE` 恢复 | handler 刚进来时先不被同级中断打断 |
| `MPIE` | 保存旧 `MIE` | 置 1 | 记住 trap 前 M 级中断是否打开 |
| `MPP` | 保存旧 privilege | 用它恢复 `priv_q`，再清到最低实现 privilege | 记住从 M/S/U 哪一级进来的 |

trap 进入 `M-mode` 时，硬件会：

1. 把当前 `MIE` 复制到 `MPIE`。
2. 把 `MIE` 清 0。
3. 把当前 privilege 写入 `MPP`。

返回时，`MRET` 会：

1. 让 `MIE = MPIE`。
2. 把 privilege 切回 `MPP`。
3. 把 `MPIE` 置 1。
4. 把 `MPP` 置为最低实现 privilege。

可用一个简化关系理解：

$$
\text{trap entry: } (MIE, MPIE, MPP) \leftarrow (0, MIE_{old}, priv_{old})
$$

$$
\text{MRET: } MIE \leftarrow MPIE,\quad priv \leftarrow MPP
$$

### 3.2 S 级状态

`sstatus` 是 `mstatus` 的受限视图：S-mode 软件不能随便看到和修改所有 M-mode 状态，只能看到 OS 需要管理的一部分字段。它里面最关键的是：

- `SIE`。
- `SPIE`。
- `SPP`。

当 trap 被委派到 `S-mode` 时：

1. `SPIE = SIE`。
2. `SIE = 0`。
3. `SPP` 记录来源 privilege。

`SRET` 则恢复这些状态。

可以把 S 级状态理解成 OS 内核自己的 trap 小栈。比如用户程序在 U-mode 发生 page fault，trap 被委派到 S-mode 后，硬件把来源 privilege 记录到 `SPP`，把旧的 `SIE` 备份到 `SPIE`，然后进入内核的 page fault handler。handler 处理完以后执行 `SRET`，硬件再根据 `SPP/SPIE/sepc` 回到原来的用户执行流。

### 3.3 为什么要把 `xPP` 设成最低 privilege

这是一个很实用的调试设计：

- `xRET` 之后把 `xPP` 清到最低 privilege。
- 如果 trap handler 忘了正确保存/恢复状态，很容易在后续读 CSR 时暴露出来。

这类设计不是为了“省逻辑”，而是为了让软件 bug 更早暴露。

### 3.4 进阶位：MPRV、SUM、MXR

这些位不是理解 trap 的第一层，但它们说明 privilege 不只影响“能不能执行某条指令”，还会影响访存权限。对跑 OS 的系统来说，内核经常要检查、复制、模拟用户态内存访问，所以需要更细的控制位：

- `MPRV`：让 M-mode 的数据访存按 `MPP` 的权限来解释。
- `SUM`：允许 S-mode 访问 U-mode 可访问页面。
- `MXR`：允许把可执行页当成可读页。

它们常见于 trap handler、异常模拟和内核复制用户态内存的场景。

这几个位体现了特权系统和 MMU/访存权限的连接：

| 位 | 主要影响 | 典型场景 | 常见坑 |
|---|---|---|---|
| `MPRV` | 数据访存按 `MPP` 指示的权限执行 | M-mode 模拟低权限 load/store | 只影响数据访问，不影响取指 |
| `SUM` | S-mode 是否允许访问 U page | 内核 copy_from_user/copy_to_user | 长期开启会扩大内核误访问风险 |
| `MXR` | executable page 是否可读 | 读取代码页、加载器/调试 | 权限检查实现漏掉 X->R |

因此特权级不是只影响“能不能写某个 CSR”。它会通过 PMP/PMA/MMU 权限检查影响每一次取指和数据访问。`0805` 中的 TLB/MMU 会继续展开这条路径。

### 3.5 小结

`mstatus/sstatus` 的本质是“trap 上下文的两级栈”。它们把“是否允许中断”和“从哪一级返回”绑定在一起，避免 trap handler 自己把现场弄丢。

## 第4章 Trap 入口：mtvec/stvec、mepc/sepc、mcause/scause、mtval/stval

当前阶段建议：简单了解本章，先记住 `xvec` 决定去哪里、`xepc` 决定从哪里回来、`xcause/xtval` 说明为什么进来。

### 4.0 trap 入口要解决什么

一旦发生异常或中断，CPU 需要快速回答四个问题：

1. 去哪里处理。
2. 是什么原因。
3. 该从哪里返回。
4. 有没有补充的 fault 信息。

RISC-V 用一组 CSR 解决这四个问题：

| 通用写法 | M-mode 对应 CSR | S-mode 对应 CSR | 解决的问题 |
|---|---|---|---|
| `xvec` | `mtvec` | `stvec` | handler 入口地址在哪里 |
| `xepc` | `mepc` | `sepc` | 之后从哪里恢复执行 |
| `xcause` | `mcause` | `scause` | 为什么进入 trap |
| `xtval` | `mtval` | `stval` | 有没有 fault 地址、非法指令编码等附加信息 |

这里的 `x` 只是文档里的占位符，不是一个真实 CSR 名字。trap 进入 M-mode 时写 M 组 CSR；trap 进入 S-mode 时写 S 组 CSR。

可以先记住一条主线：`xvec` 决定“去哪里”，`xepc` 决定“回来哪里”，`xcause/xtval` 决定“为什么来、带什么细节”。

### 4.1 mtvec/stvec 的 direct 和 vectored

`mtvec/stvec` 是 handler 入口配置寄存器，由 BASE 和 MODE 组成。软件在初始化阶段把 handler 地址写进去，之后硬件发生 trap 时就从这里取入口地址。

- `direct`：所有 trap 都跳到 BASE。
- `vectored`：同步异常跳到 BASE；异步中断跳到 `BASE + 4 * cause`。

这意味着：

- direct 模式更简单。
- vectored 模式对常见中断延迟更友好，但需要更多地址计算和更严格的对齐约束。

direct 模式下，硬件只负责跳到统一入口，具体是 timer interrupt、external interrupt 还是 illegal instruction，由 handler 读取 `xcause` 再分发。vectored 模式下，常见中断可以直接跳到不同小入口，减少软件分发开销；但同步异常仍然走 BASE，因为异常处理通常需要更完整的上下文判断。

### 4.2 mepc/sepc 记录什么

`mepc/sepc` 记录的是 handler 将来可能要返回的位置。对同步 exception，通常就是“触发 trap 的那条指令地址”；对 interrupt，则是“被中断时下一条应该继续执行的程序点”。

对于同步 exception：

- 记录 faulting instruction 的地址。

对于 interrupt：

- 记录被打断的程序点。

对于 `ECALL/EBREAK`：

- 记录该指令本身地址，而不是下一条指令地址。

这是面试高频点，很多人会把它和普通跳转地址混掉。尤其是 `ECALL`：硬件把 `xepc` 设为 `ECALL` 自己，handler 如果希望系统调用返回到下一条指令，通常由软件把 `xepc` 加上当前指令长度后再执行 `xRET`。硬件不默认帮你跳过，因为不同 exception 的处理策略不一样，有些要重试，有些要终止程序，有些才跳过。

### 4.3 mcause/scause 如何编码

`mcause/scause` 的高位是 Interrupt 标志位，低位是原因编码。

因此原因有两大类：

- Interrupt=0：同步异常。
- Interrupt=1：异步中断。

这使得 trap handler 可以先看高位，再分发到相应异常/中断处理分支。

用软件视角看，handler 大概会做这样的判断：

```text
if xcause.interrupt == 1:
    按 timer / software / external interrupt 分发
else:
    按 illegal instruction / page fault / ecall / access fault 分发
```

硬件视角则要保证 cause 编码和 trap 来源一致，不能把同步异常编码成 interrupt，也不能把 S-mode 的 cause 写进 M-mode 的 CSR 组。

### 4.4 mtval/stval 的定位价值

`mtval/stval` 用来补充 fault 细节，比如：

- 地址非对齐。
- 访问错误。
- page fault。
- 某些 illegal instruction 的原始指令位。

工程上它的作用很直接：

- 简化异常处理代码。
- 提高 debug 可读性。
- 降低软件重解码成本。

不是所有 trap 都必须提供有意义的 `xtval`，有些场景可以写 0。判断重点是：如果软件要定位“哪个地址错了”或“哪条指令非法”，`xtval` 就很有价值；如果只是普通 timer interrupt，它通常没有额外 fault 地址。

### 4.5 trap 入口的硬件动作

以委派到 M-mode 为例，硬件通常要做：

1. 选定 trap 目标 privilege。
2. 写 `xepc`。
3. 写 `xcause`。
4. 写 `xtval`。
5. 更新 `xstatus` 的 `xIE/xPIE/xPP`。
6. 重定向 `pc` 到 `mtvec/stvec`。
7. flush 年轻指令。

可以把 trap entry 看成一条特殊的、由硬件发起的控制流指令。它既要写 CSR，又要 redirect PC，还要 kill younger instruction。顺序五级流水线中，一个常见实现是在 commit 或接近 commit 的边界接受 trap：

```text
older instruction   -> already committed or allowed to commit
trap instruction    -> writes xepc/xcause/xtval/xstatus and redirects PC
younger instruction -> flushed, no GPR/memory/CSR side effect
```

这就是 precise exception 的硬件基础。若 trap 在 EX 阶段刚被检测到就立刻写 CSR，而更老的 MEM/WB 指令还可能异常或写状态，就可能出现软件看到的 EPC/cause 与实际提交顺序不一致。

不同 trap 来源需要保存的信息也不同：

| 来源 | `xepc` 应记录 | `xcause` | `xtval` 常见内容 | 注意点 |
|---|---|---|---|---|
| illegal instruction | 非法指令 PC | illegal instruction | 非法指令编码或 0 | 指令可能来自压缩/非压缩路径 |
| load misaligned | load 指令 PC | load address misaligned | faulting address | 不能同时写回 rd |
| store access fault | store 指令 PC | store access fault | faulting address | 不能产生部分写副作用 |
| ECALL | ECALL 指令 PC | 按当前 privilege 区分 | 0 | handler 通常软件决定是否跳过 ECALL |
| interrupt | 被打断的下一条可恢复 PC | interrupt bit=1 + cause | 通常 0 | 在指令边界接受 |

### 4.6 direct 与 vectored 的地址公式

对 `vectored` 模式，异步中断入口地址可写成：

$$
pc_{trap} = BASE + 4 \times cause
$$

对同步异常，仍然是：

$$
pc_{trap} = BASE
$$

### 4.7 小结

`mtvec/stvec` 决定去哪里，`mepc/sepc` 决定从哪里回来，`mcause/scause` 决定为什么去，`mtval/stval` 决定软件看见什么细节。四者组合起来，才能让 trap 处理既快又可调试。

## 第5章 异常、中断和委派

当前阶段建议：简单了解 exception 和 interrupt 的区别即可，delegation 和复杂中断路由可等支持 S-mode/OS 时再详细看。

### 5.0 异常和中断的区别

这是最常见的面试题之一。

简洁答案：

```text
exception 是当前指令执行引起的同步事件；
interrupt 是外部或异步事件，不由当前指令直接造成。
```

更准确地说：

- exception：与当前指令强相关，能定位到具体 faulting instruction。
- interrupt：与指令流异步，通常在指令边界被接受。

从流水线角度看，exception 像是某条指令在执行途中“自己出了问题”：译码发现非法指令、访存发现地址不对齐、MMU 发现 page fault。interrupt 则像是 CPU 外部或平台控制逻辑发来请求：定时器到了、外设需要服务、其他 hart 发来 software interrupt。它不属于当前这条指令的计算结果，所以通常等到一个安全的指令边界再接收。

### 5.1 常见 exception

常见 exception 可以按“在哪个阶段被发现”建立直觉：

| exception | 典型发现位置 | 含义 |
|---|---|---|
| illegal instruction | ID/decode | 指令编码不合法，或当前 privilege 不允许执行 |
| ecall | ID/EX | 软件主动请求执行环境服务，例如系统调用 |
| ebreak | ID/EX | 软件断点或调试入口 |
| instruction address misaligned | IF/PC 选择 | 取指地址不满足对齐要求 |
| instruction access fault | IF/取指总线 | 指令取值访问失败，例如访问非法物理区域 |
| load/store address misaligned | EX/MEM | 数据访存地址不满足访问宽度对齐 |
| load/store access fault | MEM/总线或 PMP/PMA | 数据访问目标区域不允许或总线返回错误 |
| page fault | IF/MEM 的地址翻译 | 虚拟地址翻译或页权限检查失败 |

这些都是“某条具体指令导致的事件”，所以要能精确记录 faulting PC，并保证年轻指令不产生副作用。

### 5.2 常见 interrupt

在机器级常见的根中断源包括：

- software interrupt。
- timer interrupt。
- external interrupt。

它们在 `mip/mie` 中分别对应挂起位和使能位。更高层系统里，外部中断通常还会由平台级中断控制器统一汇聚。

| interrupt | 常见来源 | 软件通常用它做什么 |
|---|---|---|
| software interrupt | 其他 hart 或平台软件置位 | 核间通知、调度请求 |
| timer interrupt | 计时器比较命中 | OS 时间片调度、超时处理 |
| external interrupt | PLIC 或平台中断控制器 | 外设服务，例如 UART、网卡、存储控制器 |

interrupt 不是由当前指令“算出来”的，但接受 interrupt 时仍然要保证精确状态：老指令已经提交，年轻指令被 flush，`xepc` 指向可恢复的程序点。

### 5.3 委派机制

`medeleg/mideleg` 的作用是把某些 trap 从 M-mode 分流给 S-mode。为什么需要它？因为支持 OS 的系统里，很多 trap 本来就应该由 OS 内核处理，如果全部先进入 M-mode 固件，再由固件转给 S-mode，路径会更长，职责也不清晰。

典型例子：

| trap | 更适合谁处理 | 原因 |
|---|---|---|
| U-mode `ECALL` | S-mode OS | 这是用户程序请求系统调用 |
| page fault | S-mode OS | OS 管页表、虚拟内存和进程地址空间 |
| timer interrupt | S-mode OS 或 M-mode 固件，取决于平台设计 | OS 常用它做调度，但底层 timer 可能由 M-mode 管 |
| M-mode 自身非法指令 | M-mode | 机器级错误不能下放给更低权限处理 |

基本规则是：

- 默认所有 trap 都先到 M-mode。
- 通过 delegation 位，可以把一部分同步异常或中断交给 S-mode。
- 但 trap 不会从更高 privilege 再“向下转”到更低 privilege；这类行为要么在更低级 trap handler 里发生，要么在对应 privilege 下横向处理。

这条规则是工程上非常容易写错的点。

委派判断不能只看 `medeleg/mideleg` 位本身，还要看 trap 发生时的当前 privilege。一个实用判断表如下：

| 当前 privilege | trap 是否被 delegate 到 S | 目标 privilege |
|---|---|---|
| U-mode | 对应 delegation bit=1 且实现 S-mode | S-mode |
| U-mode | delegation bit=0 | M-mode |
| S-mode | 对应 delegation bit=1 | S-mode |
| S-mode | delegation bit=0 | M-mode |
| M-mode | 即使 delegation bit=1 | M-mode |

最后一行很关键：M-mode 自己发生的 trap 不会被“向下委派”到 S-mode。否则机器级 handler 自身的非法指令或访问错误会跳进监管级 handler，破坏特权层次。

注意，委派不是软件函数调用，也不是 handler 里再跳一次。它是在 trap 入口硬件选择阶段就决定目标 privilege 和目标 CSR 组：被委派到 S-mode 就写 `sepc/scause/stval/sstatus` 并跳 `stvec`；不委派就写 `mepc/mcause/mtval/mstatus` 并跳 `mtvec`。

### 5.4 中断使能和 pending 的关系

一个中断真正能进入 trap，通常要满足：

$$
\text{pending} \land \text{enable} \land \text{not delegated away}
$$

M 级中断还要同时考虑当前 privilege 和 `MIE`；S 级中断则要看 `SIE` 和对应的 `sip/sie` 位。

中断路径通常可以拆成三类信号：

| 信号 | 来源 | 含义 |
|---|---|---|
| pending | 外设、timer、software interrupt 或 PLIC/CLINT | 事件已经发生 |
| local enable | `mie/sie` 中对应位 | 这类中断是否允许 |
| global enable | `mstatus.MIE/SIE` | 当前 privilege 是否允许被中断打断 |

一个中断 pending 但未被接受，不代表硬件错。常见原因是全局中断关闭、单项 enable 关闭、当前 privilege 屏蔽、被委派到另一个层级，或 debug/halt/WFI 等特殊状态改变了接受条件。

### 5.5 优先级和 bounded time

中断不是 pending 之后就立刻无条件打断流水线。CPU 要综合 pending、enable、当前 privilege、委派、debug、WFI、已有 trap 等条件，再决定是否接受。

规范要求，pending 变化后，CPU 必须在有界时间内重新评估是否应当陷入。对 RTL 来说，这意味着：

- 中断 pending 不应“永久卡住”。
- xRET、相关 CSR 写入后，要立刻重新判断 trap 条件。

所谓 bounded time，就是不能因为实现里少连了一条重新评估路径，导致软件已经打开中断、pending 也已经置位，但 CPU 永远不进 handler。它不是要求“下一拍一定进入”，而是要求实现不能无限拖延。

### 5.6 一个实用的 trap 选择公式

在实现上可以把 trap 选择写成：

$$
trap\_req = exc\_valid \lor irq\_accept
$$

其中：

$$
irq\_accept = pending \land enable \land \neg delegate\_mask \land \neg masked\_by\_priv
$$

这只是工程抽象，真实设计还要考虑优先级、kill、debug、WFI 和 double trap。

实现 trap 选择时，建议把同步 exception 和异步 interrupt 分开处理：

```text
1. 先从 pipeline 中选择最老的有效 exception。
2. 若没有可接受 exception，再判断 pending interrupt。
3. 选出目标 privilege 和目标向量。
4. 写对应 xEPC/xCAUSE/xTVAL/xSTATUS。
5. redirect PC 并 flush younger instruction。
```

这样可以避免“年轻 interrupt 抢在老 exception 前面进入 trap”的非精确行为。对简单顺序核，通常在 commit 边界接受 interrupt，就是为了自然满足指令年龄顺序。

### 5.7 小结

exception 更像“指令本身出事了”，interrupt 更像“外部事件找上门了”。委派机制让一部分 trap 直接交给更低 privilege 的 software handler，从而减少 M-mode 的负担。

## 第6章 常见特权指令：ECALL、EBREAK、MRET、SRET、WFI

当前阶段建议：简单了解 `ECALL/EBREAK/MRET` 的用途即可，最小教学核若不支持 CSR/trap 可以暂时不实现这些指令。

### 6.0 ECALL/EBREAK

`ECALL` 是软件主动进入执行环境的标准入口。最常见场景是用户程序需要 OS 服务，比如读写文件、申请内存、创建线程。用户程序不能直接操作内核数据结构和设备寄存器，于是执行 `ECALL`，让硬件产生一个同步 exception，进入更高 privilege 的 handler。

它根据当前 privilege 产生不同的 exception code，例如 U-mode 的 `ECALL` 和 S-mode 的 `ECALL` 原因编码不同，因此能够被选择性委派。这样 M-mode 固件可以把 U-mode 系统调用委派给 S-mode OS，也可以保留某些机器级调用自己处理。

`EBREAK` 则主要用于调试和断点场景，通常进入 breakpoint trap。

工程上要记住两点：

1. 它们都是同步 exception。
2. 它们的 EPC 指向自己，不是下一条指令。

这意味着 handler 处理完后必须明确决定下一步：

| 指令 | handler 常见处理 | 为什么 EPC 指向自己很重要 |
|---|---|---|
| `ECALL` | 读取系统调用号和参数，执行服务，通常把 `xepc` 加到下一条再返回 | 硬件不预设系统调用一定成功或一定跳过 |
| `EBREAK` | 进入调试器、通知 debugger、停住或单步 | 调试器需要知道断点指令本身在哪里 |

### 6.1 MRET/SRET

`MRET` 和 `SRET` 是 trap return 指令，也就是 handler 的出口。它们不是普通 `JALR`，因为普通跳转只能改 PC；trap return 还必须恢复 privilege 和中断使能状态。

它们的统一动作可以概括为：

1. `xIE <- xPIE`
2. `priv <- xPP`
3. `xPIE <- 1`
4. `xPP <- lowest supported privilege`
5. `pc <- xEPC`

此外，如果新 privilege 低于 M，`MPRV` 也会被清掉。

这和第 3 章的 `xIE/xPIE/xPP` 正好对应：

```text
trap entry:
  保存旧 interrupt enable 到 xPIE
  保存旧 privilege 到 xPP
  pc 跳到 xvec

xRET:
  从 xPIE 恢复 interrupt enable
  从 xPP 恢复 privilege
  pc 跳回 xepc
```

如果 handler 修改过 `xepc`，返回点就会跟着改变。系统调用通常会这么做；非法指令或 page fault 则可能选择重试、修正映射后重试，或者终止当前任务。

### 6.2 WFI

`WFI` 用于等待中断，常见于空闲循环或低功耗路径。软件表达的是“现在没事做，可以等到有中断再继续”；硬件可以据此做时钟门控、低功耗等待，也可以简单把它当成 NOP。

对硬件设计者而言，关键是：

- 不能把 WFI 的存在理解成“必须停机”。
- 不能把中断唤醒逻辑只绑在某一种 trap 路径上。
- 软件要假设 WFI 可能被任何合法实现方式处理。

### 6.3 指令与 privilege 的合法性

特权指令不是所有 mode 都能随便执行。decode 阶段除了识别 opcode，还要结合当前 `priv_q` 和实现支持的 mode 做合法性检查。

典型规则如下：

| 场景 | 正确行为 | 直觉 |
|---|---|---|
| U-mode 执行 `MRET` | illegal instruction | 用户程序不能伪装成机器级 handler 返回 |
| U-mode 直接写 `mstatus/mtvec` | illegal instruction 或 CSR access fault 类非法访问 | 用户程序不能改机器级控制状态 |
| 不支持 S-mode 的核执行 `SRET` | illegal instruction | 硬件没有对应状态可恢复 |
| S-mode 执行 M-mode 专用指令 | illegal instruction | 低权限不能越权控制机器级资源 |
| WFI 在某些配置下受限 | 按规范和配置处理，可能当 NOP 或非法 | 低权限等待中断可能影响调度和功耗策略 |

这类检查通常在 ID 阶段尽早做，作为同步 exception 随指令进入流水线；真正接受 trap 时仍要遵守精确异常规则，不能因为早发现就提前破坏架构状态。

### 6.4 小结

这几条指令是 trap 系统的“入口”和“出口”。如果它们的行为不正确，CSR 设计再完整，操作系统也跑不稳。

## 第7章 RTL 结构：CSR 文件、Trap 控制器和 Privilege 状态机

### 7.0 硬件分块

一个支持 M/S/U 的顺序核，通常可以把这部分分成三块：

```text
          +----------------------+
decode -> | CSR file / privilege | <-> exception/interrupt inputs
          +----------+-----------+
                     |
                     v
          +----------------------+
          | trap controller      | ---> redirect_pc / flush / kill
          +----------+-----------+
                     |
                     v
          +----------------------+
          | privilege state      | ---> current priv / xIE / xPIE / xPP
          +----------------------+
```

实际流水线里还需要一条 commit/trap 接口，把 `0802` 的流水线控制和本篇的 CSR/trap 状态连接起来：

```text
pipeline commit side
  commit_valid
  commit_pc
  commit_inst
  commit_exception_valid/cause/tval
  commit_is_xret
  commit_is_csr
        |
        v
trap/CSR control side
  decide trap/xRET/CSR write
  update xEPC/xCAUSE/xTVAL/xSTATUS
  redirect PC
  kill younger instructions
```

这条接口的核心原则是：同一条指令的 CSR 写、trap 写和 PC redirect 必须在同一个架构边界上讲得通。否则容易出现 CSR 已经写了，但指令后来被 flush；或者 trap redirect 发生了，但更年轻 store 已经写到 MMIO。

### 7.1 CSR 文件的职责

CSR 文件至少要承担：

- 按地址读出 CSR。
- 按指令类型执行写入规则。
- 处理读副作用和写副作用。
- 对只读位、WARL 位做规范化。
- 对 trap path 的 CSR 更新提供独立入口。

### 7.2 trap controller 的职责

trap controller 负责：

- 统一收敛 exception、interrupt、xRET、debug、reset 优先级。
- 决定 trap target privilege。
- 选择 `mtvec/stvec` 的入口地址。
- 填写 `xepc/xcause/xtval`。
- 发出 flush 和 pc redirect。

### 7.3 privilege 状态机

最低限度需要跟踪：

- 当前 privilege。
- 当前是否在 trap handler 中。
- M/S 全局中断使能。
- 陷入前 privilege。

如果设计支持 S-mode，建议把 S 级状态也完整建模，不要只用一个“当前模式枚举”硬撑所有逻辑。

privilege 状态机的更新来源主要有三类：

| 来源 | 状态变化 | 常见错误 |
|---|---|---|
| trap entry | `priv_q` 切到目标 privilege，保存 `xPP/xPIE`，关闭 `xIE` | 忘记区分 M/S 两套字段 |
| xRET | `priv_q` 恢复到 `xPP`，恢复 `xIE`，清理 `xPP` | 返回后 pending interrupt 没重新评估 |
| CSR write | 软件直接改 `mstatus/sstatus/mie/mip` 等 | WARL/只读位处理错误 |

这三类更新可能同拍竞争。一个简单、可验证的策略是：reset/debug/trap/xRET 这类控制转移优先，普通 CSR 指令只在有效提交且未被 trap/flush 时写入。若项目支持精确异常，不能允许一条已经产生 exception 的 CSR 指令继续执行普通 CSR 写副作用。

这和 `0802` 第7章的 structural hazard 有一点相似：多个来源可能在同一拍想更新同一组控制状态。但 CSR/trap 更强调架构语义和优先级，不只是“端口不够”。因此文档里通常不会把它简单归类为普通结构冒险，而是单独讲 CSR 写、trap 写、xRET 和 debug/reset 的优先级。

### 7.4 CSR 指令 RTL 骨架

```systemverilog
module csr_file (
  input  logic        clk,
  input  logic        rst_n,

  input  logic        csr_req_valid,
  input  logic [11:0] csr_addr,
  input  logic [2:0]  csr_op,       // CSRRW/CSRRS/CSRRC/...
  input  logic [31:0] csr_wdata,
  input  logic [4:0]  csr_uimm,
  input  logic        csr_rd_is_x0,
  input  logic        csr_rs1_is_x0,
  input  logic [31:0] csr_rdata_in,  // 读出当前 CSR 值

  input  logic        trap_update_valid,
  input  logic [11:0] trap_csr_addr,
  input  logic [31:0] trap_csr_wdata,

  output logic [31:0] csr_rdata_out,
  output logic        illegal_csr_access
);

  logic [31:0] csr_old;
  logic [31:0] csr_new;
  logic         csr_do_read;
  logic         csr_do_write;

  always_comb begin
    csr_old = csr_rdata_in;
    csr_new = csr_old;
    csr_do_read = 1'b0;
    csr_do_write = 1'b0;
    illegal_csr_access = 1'b0;

    if (csr_req_valid) begin
      unique case (csr_op)
        3'd0: begin // CSRRW
          csr_do_read  = !csr_rd_is_x0;
          csr_do_write = 1'b1;
          csr_new      = csr_wdata;
        end
        3'd1: begin // CSRRS
          csr_do_read  = 1'b1;
          csr_do_write = !csr_rs1_is_x0;
          csr_new      = csr_old | csr_wdata;
        end
        3'd2: begin // CSRRC
          csr_do_read  = 1'b1;
          csr_do_write = !csr_rs1_is_x0;
          csr_new      = csr_old & ~csr_wdata;
        end
        3'd3: begin // CSRRWI
          csr_do_read  = !csr_rd_is_x0;
          csr_do_write = 1'b1;
          csr_new      = {27'b0, csr_uimm};
        end
        3'd4: begin // CSRRSI
          csr_do_read  = 1'b1;
          csr_do_write = (csr_uimm != 5'b0);
          csr_new      = csr_old | {27'b0, csr_uimm};
        end
        3'd5: begin // CSRRCI
          csr_do_read  = 1'b1;
          csr_do_write = (csr_uimm != 5'b0);
          csr_new      = csr_old & ~{27'b0, csr_uimm};
        end
        default: begin
          illegal_csr_access = 1'b1;
        end
      endcase
    end
  end

  always_ff @(posedge clk or negedge rst_n) begin
    if (!rst_n) begin
      csr_rdata_out <= '0;
    end else begin
      if (trap_update_valid) begin
        // trap-related CSR write has priority over ordinary CSR instruction write
        // 具体写入由 trap controller 提供
      end
      if (csr_req_valid && csr_do_read)
        csr_rdata_out <= csr_old;
      if (csr_req_valid && csr_do_write)
        /* 对应 CSR 寄存器写回逻辑 */;
    end
  end
endmodule
```

上面代码是结构骨架，不是完整可综合实现。真实项目里应把每个 CSR 拆成字段级更新函数，尤其是 `mstatus/sstatus/mie/mip` 这类有复杂字段语义的寄存器。

对字段级 CSR，推荐写成“读值组装 + 写值规范化 + 字段更新”的三段结构：

```text
csr_read_mux(addr) -> csr_old
csr_legalize(addr, csr_old, csr_wdata) -> csr_new
csr_commit_write(addr, csr_new, write_enable)
```

这样做的好处是 WARL 位、只读位和 side effect 不会散落在主译码里。比如 `mstatus.MPP` 只能写实现支持的 privilege 编码；`mip` 的某些 pending 位可能由硬件中断源置位，软件不能随意写清；计数器类 CSR 可能有高低位读一致性问题。

### 7.5 trap 控制 RTL 骨架

```systemverilog
always_comb begin
  trap_taken   = 1'b0;
  trap_mode    = priv_q;
  trap_vector  = '0;
  trap_epc     = '0;
  trap_cause   = '0;
  trap_tval    = '0;
  flush_pipe   = 1'b0;
  pc_redirect  = 1'b0;

  if (reset) begin
    pc_redirect = 1'b1;
    trap_vector = reset_vector;
  end else if (trap_req_valid) begin
    trap_taken  = 1'b1;
    trap_mode   = trap_target_priv;
    trap_vector = trap_target_base;
    trap_epc    = trap_fault_pc;
    trap_cause  = trap_fault_cause;
    trap_tval   = trap_fault_tval;
    flush_pipe  = 1'b1;
    pc_redirect = 1'b1;
  end
end
```

这个骨架只表达了 redirect，真实 trap controller 还要输出 CSR 更新包：

```text
trap_csr_update:
  target_priv
  epc_wdata
  cause_wdata
  tval_wdata
  status_update
  vector_pc
```

对 M-mode trap，写 `mepc/mcause/mtval/mstatus`；对 S-mode trap，写 `sepc/scause/stval/sstatus`。委派到 S-mode 时不应误写 M-mode 的 EPC/cause；进入 M-mode 时也不应误写 S-mode 的那一组。

### 7.6 优先级建议

实际优先级通常应是：

```text
reset > debug halt / fatal error > trap > branch redirect > stall
```

如果同一拍有多种事件，必须清楚定义谁覆盖谁。不然验证波形会非常难看。

### 7.7 关键工程点

- `valid bit` 必须参与所有副作用写使能。
- wrong-path 指令不允许写 CSR、寄存器和 memory。
- trap 入口要 flush 年轻指令。
- `xRET` 之后要重新评估 pending interrupt。

## 第8章 验证方法

### 8.0 验证目标

这一类逻辑不能只看单条波形。必须同时证明：

1. CSR 读写语义正确。
2. trap 入口写对了 CSR。
3. privilege 切换正确。
4. interrupt/exception 的优先级正确。
5. 委派和返回不会丢现场。

### 8.1 directed test

建议至少覆盖以下场景：

| 场景 | 关注点 |
|---|---|
| `CSRRW` 读写 | `rd=x0` 是否跳过读 |
| `CSRRS/CSRRC` | `rs1=x0` 是否只读不写 |
| `CSRRSI/CSRRCI` | `uimm=0` 是否只读不写 |
| `mepc/sepc` | trap 发生后 EPC 是否是 faulting PC |
| `mcause/scause` | 高位 interrupt 标志是否正确 |
| `mtval/stval` | fault 地址或指令 bits 是否正确 |
| `MRET/SRET` | privilege 和 `xIE/xPIE/xPP` 是否恢复 |
| `ECALL/EBREAK` | EPC 是否指向本指令 |
| `mideleg/medeleg` | 委派后是否进入 S-mode |
| nested trap | trap handler 内再来一次 trap 如何处理 |
| WFI | enabled interrupt 唤醒行为是否合理 |

### 8.2 random test 与 ISS 比对

随机测试建议用 committed trace 与 ISS 比较：

- `commit_pc`
- `commit_inst`
- `commit_priv`
- `commit_rd/wdata`
- `commit_csr_addr/wdata`
- `commit_exception`
- `commit_interrupt`

重点是不要只比寄存器文件。trap 的正确性往往体现在 CSR 和 privilege 变化上。

### 8.3 scoreboard

scoreboard 可以维护两类状态：

1. 架构状态：GPR、PC、privilege、CSR。
2. 事件状态：pending interrupt、delegate mask、trap depth。

这样可以在第一条 mismatch 出现时快速定位是“数据错了”还是“控制流错了”。

### 8.4 SVA

#### 8.4.1 trap 发生后必须更新 EPC

```systemverilog
// 不可综合：验证断言
property p_trap_updates_epc;
  @(posedge clk) disable iff (!rst_n)
    trap_taken |-> ##1 (epc_written && (epc_wdata == trap_pc));
endproperty

assert property (p_trap_updates_epc);
```

#### 8.4.2 xRET 恢复 privilege

```systemverilog
// 不可综合：验证断言
property p_xret_restores_priv;
  @(posedge clk) disable iff (!rst_n)
    xret_fire |=> (priv_q == xpp_old);
endproperty

assert property (p_xret_restores_priv);
```

#### 8.4.3 wrong-path 不能写副作用

```systemverilog
// 不可综合：验证断言
property p_wrong_path_killed;
  @(posedge clk) disable iff (!rst_n)
    flush_pipe |=> (!wb_valid_q && !dmem_we_o && !csr_we_o);
endproperty

assert property (p_wrong_path_killed);
```

### 8.5 coverage

覆盖建议：

- privilege 迁移：M->M、U->M、U->S、S->M、S->S。
- trap 类型：exception、interrupt、ECALL、EBREAK、illegal instruction。
- trap 目标：M-mode、S-mode。
- 委派开关：delegated / non-delegated。
- `xRET` 类型：MRET、SRET。
- CSR 指令：读写、只读、掩码为 0、写只读位。
- `mtvec/stvec`：direct/vectored。
- `xPP`：U、S、M。
- `xIE`：0/1。

### 8.6 验证时最常见的误判

很多人只看波形里 trap 进入了，没有继续看：

- `mepc/sepc` 是不是错了 4。
- `mcause/scause` 是否把 interrupt bit 搞反。
- `xRET` 后 privilege 是否恢复。
- 年轻指令是否真的没提交。

这些才是最终的判定点。

## 第9章 时序、CDC、综合和后端影响

### 9.0 CSR / Trap 为什么会影响 Fmax

这部分逻辑虽然看起来“控制多、数据少”，但很容易成为关键路径来源：

- CSR 地址译码扇出大。
- trap 优先级判定复杂。
- interrupt pending 需要跨多个 source 汇总。
- `mtvec/stvec` 的 redirect 需要快速完成。

如果把所有判断都塞在 ID 阶段，`Fmax` 很容易掉。

### 9.1 时序热点

常见热点有：

- CSR decode -> privilege check -> write enable。
- exception detect -> cause encode -> trap redirect。
- branch/trap/load stall 合并控制。
- `mip/sip` pending 位汇总。

可行的优化方向：

- 把 trap 原因提前一拍预译码。
- 把 CSR 写入分字段寄存。
- 把 interrupt pending 分级汇总。
- 在不影响 ISA 的前提下，把某些 redirect 推到更靠后的 stage。

### 9.2 CDC

真正需要关注 CDC 的地方通常不是 CSR 本身，而是这些来源：

- 外部中断输入。
- timer compare 事件。
- debug halt 请求。
- 平台级 interrupt controller 输出。

这些信号往往不是核心时钟域同步产生，必须经过同步器或握手桥。否则 `mip` / `sip` 更新会出现亚稳态风险。

### 9.3 综合影响

综合工具可能把一些看似“状态保存”的逻辑优化掉，但 trap 语义不能丢：

- `xRET` 相关字段写回不能被错误消除。
- `valid bit` 屏蔽必须保留。
- 只读/写一次的 WARL/WLRL 规则不能被普通寄存器化简破坏。

### 9.4 后端影响

trap/interrupt 路径经常走在比较密集的控制网络上，place & route 时要注意：

- trap 控制跨多个 stage。
- `pc redirect` 和 flush 网络容易有长线。
- 外部中断同步链要靠近输入边界。

如果你的 CPU 是面向低功耗 MCU，`WFI` 和中断唤醒路径还可能影响 clock gating 的实现。

## 第10章 常见 bug、边界条件和 debug 方法

### 10.0 常见 bug

| bug | 现象 | 修复方向 |
|---|---|---|
| `rd=x0` 仍读 CSR | 不该有的 CSR 读副作用发生 | 按 Zicsr 规则跳过读 |
| `rs1=x0` 仍写 CSR | 只读指令触发写副作用 | 置位/清位类指令对 `x0` 特判 |
| `mepc/sepc` 写成下一条 | trap return 跳错位置 | EPC 必须指向 faulting instruction |
| interrupt bit 搞反 | trap handler 分发错 | `xcause` 高位单独编码 |
| `xRET` 不恢复 privilege | 返回后仍在 trap 模式 | 正确实现 `xPIE/xPP` 栈 |
| delegated trap 仍进 M | 委派无效 | 检查 `medeleg/mideleg` 和 privilege 条件 |
| M-mode 误向下委派 | M 自己执行非法指令却跳 S | trap 不能从高 privilege 直接向下转 |
| wrong-path 写回 | flush 不彻底 | 所有副作用必须受 valid bit 控制 |
| `mtvec/stvec` 向量地址算错 | vectored 中断进错入口 | `BASE + 4 * cause` 只用于 interrupt |
| WFI 当成强制睡眠 | 某些实现跑不起来 | WFI 允许实现为 NOP |
| pending interrupt 只看 enable 不看 privilege | 中断接受条件错误 | 同时看 current privilege / delegation / global enable |
| SRET 在不支持 S-mode 的核上直接跑 | 指令行为不对 | 应触发 illegal instruction |

### 10.1 边界条件

#### 10.1.1 `mepc/sepc` 对齐

`mepc/sepc` 的低位是否强制为 0，取决于 `IALIGN`。实现时要按 ISA 和是否支持压缩指令处理，不能把地址直接当普通 32 位寄存器写入。

#### 10.1.2 `mtvec/stvec` 对齐

BASE 需要至少 4 字节对齐，vectored 模式通常要求更强对齐。实现里最好把低位硬件清零，而不是完全交给软件。

#### 10.1.3 `ECALL/EBREAK`

它们是同步 exception，不是 interrupt；它们的 EPC 必须回到自己本身，不能当成“下一条指令异常”处理。

#### 10.1.4 trap 进入 M 和进入 S 的字段差异

委派到 S-mode 时，不要去写 M-mode 那组 EPC/Cause/Tval；委派到 M-mode 时则不要误写 S-mode 那组 CSR。

### 10.2 debug 方法

建议按下面顺序查：

1. 看第一条错的 commit。
2. 看 trap 是否发生在正确 privilege。
3. 看 EPC 是否对。
4. 看 cause 是否对。
5. 看 `xIE/xPIE/xPP` 是否对。
6. 看 flush 是否杀掉 younger 指令。
7. 看 `mideleg/medeleg` 是否把 trap 路由错了。

### 10.3 波形上优先盯的信号

```text
priv_q, next_priv, trap_taken
mstatus/sstatus, mie/sie, mip/sip
mtvec/stvec, mepc/sepc, mcause/scause, mtval/stval
medeleg/mideleg
flush_pipe, pc_redirect, pc_next
commit_valid, commit_pc, commit_exception
```

## 第11章 面试问法

### 11.0 基础题

#### 1. CSR 是什么，为什么需要它

简洁答法：

```text
CSR 是每个 hart 的控制状态寄存器集合，用来保存中断、trap、权限和各种平台控制状态。
它比普通寄存器更偏控制面，直接决定 trap、返回和特权切换行为。
```

#### 2. 异常和中断有什么区别

简洁答法：

```text
异常是当前指令执行引起的同步事件；
中断是外部或异步事件，通常在指令边界被接受。
```

#### 3. M-mode、S-mode、U-mode 分别干什么

答题要点：

- M-mode：平台最底层控制。
- S-mode：操作系统内核。
- U-mode：应用程序。

### 11.1 进阶追问

#### 1. `mepc` 和 `mcause` 进入 trap 时怎么写

要点：

- `mepc` 记录 faulting PC 或被中断的 PC。
- `mcause` 记录 trap 原因，高位区分 interrupt 和 exception。
- `mtval` 记录附加信息。

#### 2. `MRET` 做了什么

要点：

- 恢复 `MIE`。
- 恢复 privilege。
- 置回 `MPIE` 和 `MPP`。
- `pc` 跳回 `mepc`。

#### 3. `medeleg/mideleg` 的作用

要点：

- 把部分异常/中断分流给 S-mode。
- 减轻 M-mode 处理压力。
- 但不能把 M-mode 自己的 trap 直接委派到更低级别。

#### 4. `CSRRW/CSRRS/CSRRC` 的差别

要点：

- `CSRRW` 是交换。
- `CSRRS` 是读后按位置位。
- `CSRRC` 是读后按位清位。
- `rd=x0`、`rs1=x0` 会改变是否读写和副作用。

### 11.2 项目追问

#### 1. 你的 CPU 怎么保证精确异常

答题框架：

- 所有年轻指令在 trap 接收点前必须被 kill。
- 提交状态只在 commit 边界更新。
- trap 相关 CSR 在提交点统一写。
- reference model 以 commit trace 校验。

#### 2. 外部中断怎么进来

答题框架：

- 外部中断通常来自平台中断控制器。
- 先同步到核心时钟域。
- 再更新 `mip/sip` pending 位。
- 最后通过 enable/delegation 逻辑决定是否 trap。

#### 3. 为什么你的 trap handler 不会死循环

答题框架：

- 保存关键状态后再打开中断。
- 避免在脆弱阶段触发新的 exception。
- `xRET` 恢复现场时确保 `xPP/xPIE` 合法。

### 11.3 面试回答模板

简洁模板：

```text
CSR 管控制状态，异常和中断都通过 trap 进入更高 privilege 的 handler。
trap 时硬件写 EPC、cause、tval，并更新 xIE/xPIE/xPP；
xRET 再把 PC 和 privilege 恢复回来。
```

深入模板：

```text
如果是支持 OS 的 RISC-V 核，必须把 exception、interrupt、delegation、priority、vector base、EPC 和 privilege stack 一起设计。
其中 CSR 指令遵循原子读改写语义，rd=x0 或 rs1=x0 会影响是否读写和副作用；
trap 入口必须保证精确异常和年轻指令清除；
验证时既要比 commit trace，也要比 trap CSR 状态。
```

## 第12章 练习题与答案要点

### 12.1 练习题 1：解释 `rd=x0` 的 CSR 行为

题目：

```text
为什么 CSRRW 当 rd=x0 时不能读 CSR？
```

答案要点：

- 这是 Zicsr 的明确语义。
- 可以避免读副作用。
- 也允许硬件把某些 CSR 写操作做得更简单。

### 12.2 练习题 2：为什么 `ECALL` 的 EPC 指向自己

答案要点：

- 系统调用需要重新分析或重新执行该陷入点。
- handler 通常决定下一步控制流，不应该默认跳过指令。
- 这是同步异常的标准行为。

### 12.3 练习题 3：`MRET` 和 `SRET` 的共同点是什么

答案要点：

- 都从 trap handler 返回。
- 都恢复 `xIE/xPIE/xPP`。
- 都把 `pc` 设置为 `xepc`。

### 12.4 练习题 4：为什么外部中断要做同步

答案要点：

- 外部中断源通常来自异步域。
- 直接进核心逻辑会有亚稳态风险。
- 必须先同步或通过握手桥。

### 12.5 练习题 5：如何验证 delegated interrupt

答案要点：

- 在 `mideleg` 打开对应位。
- 让相应 interrupt pending 且 enable。
- 观察 trap 是否进入 S-mode 而非 M-mode。
- 检查 `scause/sepc/stval` 是否正确。
