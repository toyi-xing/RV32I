# 0824 数据通路、流水线寄存器与控制信号参考

> 文档编号：0824  
> 所属系列：082x RISC-V 最小教学核项目实践  
> 文档定位：定义第一版五级流水线教学核的数据通路、pipeline register 字段、控制信号生命周期和 side effect gating 参考  
> 前置文档：`0821 RV32I最小教学核指令集、编码与译码参考.md`、`0822 最小教学核工程目录、顶层接口与命名约定.md`、`0823 从单周期语义模型到五级流水线.md`

本文是写五级流水线 RTL 时的字段和控制参考。它不重复讲每条指令编码，也不展开 forwarding/stall/flush 的完整判断条件；本文重点是回答：

1. 每个 stage 做什么。
2. 每个 pipeline register 需要保存哪些字段。
3. 控制信号在哪里生成、在哪里使用、是否需要随指令流动。
4. 哪些动作会产生架构副作用，必须被 valid gating。

本文默认项目假设：

| 项目 | 假设 |
|---|---|
| pipeline | IF、ID、EX、MEM、WB |
| 取指 | imem 固定响应，32 bit 指令；PC 状态单独放在 `pc_reg` |
| 访存 | dmem 固定响应，第一版无 wait state |
| branch 决策 | 第一版放在 EX 阶段 |
| 写回 | WB 阶段写 GPR |
| valid | 每个 pipeline register 都保存 valid |
| 异常 | 第一版不做完整 trap，可保留 `illegal_instr` 调试字段 |

## 第1章 总数据通路

### 1.1 五级流水线主路径

```text
             +--------+      +---------+      +---------+      +---------+      +---------+      +---------+
redirect --->| pc_reg |----->|   IF    |----->|   ID    |----->|   EX    |----->|   MEM   |----->|   WB    |
stall ------>| PC 状态|      | fetch   |IF/ID | decode  |ID/EX | execute |EX/MEM| memory  |MEM/WB| writebk |
             +--------+      +---------+      +---------+      +---------+      +---------+      +---------+
                                  |                |                |                |                |
                                imem             GPR              ALU              dmem             GPR
```

默认下一 PC 是 `PC + 4`。当 EX 阶段发现 branch taken、JAL 或 JALR 时，产生 `redirect_valid` 和 `redirect_pc`，`pc_reg` 选择新的 PC，并 flush/kill 旧路径上的年轻指令。具体优先级见 `0825`。

### 1.2 每级主要职责

| stage | 主要输入 | 主要输出 | 是否产生架构副作用 |
|---|---|---|---|
| pc_reg | redirect/stall 控制、`PC+4` | 当前取指 PC、PC valid | 不产生 |
| IF | `pc_reg` 输出的 PC、imem 数据 | `if_id_pc`、`if_id_pc_plus4`、`if_id_instr`、valid | 不产生 |
| ID | IF/ID 字段、GPR 读口、decoder、imm_gen | `id_ex_*` 数据和控制 | 不产生 |
| EX | ID/EX 字段、forwarding 选择 | ALU 结果、branch 判断、target、store data | 不直接产生 |
| MEM | EX/MEM 字段、dmem | dmem 写请求、load 扩展结果 | store 会写 memory |
| WB | MEM/WB 字段 | GPR 写回、commit trace | 写 GPR，产生 commit |

第一版把 branch 决策放在 EX，因此 EX 会产生控制流 redirect，但 redirect 本身不是架构状态写入；真正要防止的是错路径指令后续写 GPR 或写 dmem。

## 第2章 pc_reg 与 IF stage

### 2.1 pc_reg 输入输出

`pc_reg` 是前端 PC 状态寄存器。为了让 stage 模块尽量保持纯组合，PC 更新从 `if_stage` 中拆出。

| 信号 | 方向 | 说明 |
|---|---|---|
| `clk_i/rst_n_i` | input | 时钟和复位 |
| `stall_pc_i` | input | 保持 PC 不变 |
| `redirect_valid_i` | input | 本拍需要改 PC |
| `redirect_pc_i` | input | 新 PC |
| `pc_plus4_i` | input | 默认顺序下一 PC |
| `pc_o` | output | 当前取指 PC |
| `pc_valid_o` | output | 当前 PC 是否有效 |

PC 更新优先级通常是：

```text
reset > redirect > stall > pc + 4
```

如果同一拍既有 redirect 又有 stall，第一版建议 redirect 优先。否则 branch 已经决定跳转时，PC 仍被 stall 卡住，容易继续取旧路径指令。更完整的优先级在 `0825` 中定义。

### 2.2 IF stage 输入输出

`if_stage` 是纯组合取指数据通路，不保存 PC，也不处理 redirect/stall。

| 信号 | 方向 | 说明 |
|---|---|---|
| `pc_i` | input | `pc_reg` 输出的当前取指 PC |
| `pc_valid_i` | input | 当前 PC 是否有效 |
| `imem_rdata_i` | input | 取回的 instruction |
| `imem_addr_o` | output | 当前取指地址，通常等于 `pc_i` |
| `if_pc_o` | output | 当前指令 PC |
| `if_pc_plus4_o` | output | `PC + 4`，回送给 `pc_reg` 作为默认下一 PC，也随指令流向后级 |
| `if_instr_o` | output | 当前取到的 instruction |
| `if_valid_o` | output | IF 当前输出是否有效 |

### 2.3 IF/ID register 字段

| 字段 | 位宽 | 来源 | 去向 | 说明 |
|---|---:|---|---|---|
| `if_id_valid` | 1 | IF valid | ID/hazard | 该槽是否是真实指令 |
| `if_id_pc` | 32 | IF PC | ID/EX/trace | 当前指令 PC |
| `if_id_pc_plus4` | 32 | IF | ID/EX/WB | JAL/JALR 写回需要 |
| `if_id_instr` | 32 | imem | decoder/trace | 原始 instruction |

IF/ID 更新动作：

| 控制 | 动作 |
|---|---|
| reset | `valid=0`，其他字段清零 |
| flush_if_id | `valid=0`，其他字段可清零 |
| stall_id | 保持 IF/ID 不变 |
| 正常 | 写入 IF 输出 |

## 第3章 ID stage

### 3.1 ID 做的事

ID 是“把 instruction 变成后续 stage 能执行的信息”的阶段。

| 工作 | 输入 | 输出 | 作用 |
|---|---|---|---|
| 提取字段 | `if_id_instr` | `rs1_addr/rs2_addr/rd_addr` | 确定寄存器编号 |
| decoder | opcode/funct | 控制信号 | 决定 ALU/memory/branch/WB 行为 |
| imm_gen | instruction、`imm_sel` | `imm` | 生成 32 bit 立即数 |
| 读 GPR | `rs1_addr/rs2_addr` | `rs1_rdata/rs2_rdata` | 提供操作数原值 |
| hazard 检查输入 | `uses_rs1/uses_rs2/rs*_addr` | 给 hazard unit | 判断是否需要 stall |

ID 本身不写 GPR、不写 memory。它主要负责“准备执行所需的信息”。

### 3.2 ID/EX register 字段

| 字段 | 位宽 | 来源 | 去向 | 说明 |
|---|---:|---|---|---|
| `id_ex_valid` | 1 | IF/ID valid | EX/hazard | 该槽是否有效 |
| `id_ex_pc` | 32 | IF/ID | EX/MEM/trace | branch target、debug |
| `id_ex_pc_plus4` | 32 | IF/ID | EX/MEM/WB | JAL/JALR 写回 |
| `id_ex_instr` | 32 | IF/ID | trace/debug | 原始 instruction |
| `id_ex_rs1_addr` | 5 | instruction | forwarding/hazard | `rs1` 编号 |
| `id_ex_rs2_addr` | 5 | instruction | forwarding/hazard | `rs2` 编号 |
| `id_ex_rd_addr` | 5 | instruction | 后级写回匹配 | `rd` 编号 |
| `id_ex_rs1_rdata` | 32 | GPR | EX | `rs1` 原始读值 |
| `id_ex_rs2_rdata` | 32 | GPR | EX/MEM | `rs2` 原始读值，store data 也用 |
| `id_ex_imm` | 32 | imm_gen | EX | ALU immediate、branch/jump offset |
| `id_ex_uses_rs1` | 1 | decoder | hazard/forwarding | 该指令是否真的读 rs1 |
| `id_ex_uses_rs2` | 1 | decoder | hazard/forwarding | 该指令是否真的读 rs2 |
| `id_ex_reg_we` | 1 | decoder | EX/MEM/MEM/WB | 是否有写 GPR 意图 |
| `id_ex_mem_re` | 1 | decoder | EX/MEM/MEM | 是否 load |
| `id_ex_mem_we` | 1 | decoder | EX/MEM/MEM | 是否 store |
| `id_ex_mem_size` | enum | decoder | MEM | byte/half/word |
| `id_ex_mem_unsigned` | 1 | decoder | MEM | load 零扩展 |
| `id_ex_alu_op` | enum | decoder | EX | ALU 运算 |
| `id_ex_op_a_sel` | enum | decoder | EX | ALU A 来源 |
| `id_ex_op_b_sel` | enum | decoder | EX | ALU B 来源 |
| `id_ex_branch_op` | enum | decoder | EX | branch 判断类型 |
| `id_ex_jump` | 1 | decoder | EX | 是否 JAL/JALR |
| `id_ex_jalr` | 1 | decoder | EX | 是否 JALR |
| `id_ex_wb_sel` | enum | decoder | WB | 写回来源 |
| `id_ex_illegal_instr` | 1 | decoder | debug/后续异常 | 第一版可用于仿真报错 |

ID/EX 是控制信号最密集的 pipeline register，因为大部分控制都在 ID 生成，后续 stage 分阶段消费。

### 3.3 bubble 写入 ID/EX

load-use stall 常见动作是向 ID/EX 插入 bubble。bubble 的本质是：

```text
id_ex_valid = 0
id_ex_reg_we = 0
id_ex_mem_we = 0
id_ex_mem_re = 0
其他控制信号清零或无害值
```

只清 `valid` 通常已经足够防止副作用，但把关键控制也清零能让波形更清楚，降低误用风险。

## 第4章 EX stage

### 4.1 EX 做的事

EX 主要完成：

| 工作 | 说明 |
|---|---|
| forwarding 后选择操作数 | `rs1/rs2` 可能来自 GPR 原值、EX/MEM、MEM/WB |
| ALU 运算 | ALU 指令、load/store 地址、AUIPC 等 |
| branch compare | 根据 `branch_op` 判断 taken |
| target 计算 | branch/JAL/JALR 目标地址 |
| store data 准备 | store 写入数据也可能需要 forwarding |
| redirect 输出 | taken branch、JAL、JALR 产生新 PC |

EX 是数据通路最关键的一站。`0825` 会具体展开 forwarding 条件。

### 4.2 ALU 输入选择

| `op_a_sel` | ALU A |
|---|---|
| `OP_A_RS1` | forwarding 后的 `rs1` |
| `OP_A_PC` | `id_ex_pc` |
| `OP_A_ZERO` | `32'b0` |

| `op_b_sel` | ALU B |
|---|---|
| `OP_B_RS2` | forwarding 后的 `rs2` |
| `OP_B_IMM` | `id_ex_imm` |
| `OP_B_FOUR` | `32'd4` |

很多实现中 `JAL/JALR` 的 `PC+4` 不一定用 ALU 算，也可以直接用 `id_ex_pc_plus4`。关键是 `wb_sel=WB_PC4` 时，WB 能拿到正确的 `PC+4`。

### 4.3 EX/MEM register 字段

| 字段 | 位宽 | 来源 | 去向 | 说明 |
|---|---:|---|---|---|
| `ex_mem_valid` | 1 | ID/EX valid | MEM/WB | 是否有效 |
| `ex_mem_pc` | 32 | ID/EX | trace | debug/commit |
| `ex_mem_pc_plus4` | 32 | ID/EX | WB | JAL/JALR 写回 |
| `ex_mem_instr` | 32 | ID/EX | trace | 原始 instruction |
| `ex_mem_rd_addr` | 5 | ID/EX | WB/forwarding | 目的寄存器 |
| `ex_mem_alu_result` | 32 | EX ALU | MEM/WB/forwarding | ALU 结果或访存地址 |
| `ex_mem_store_data` | 32 | EX forwarding 后 rs2 | MEM | store 写数据 |
| `ex_mem_reg_we` | 1 | ID/EX | WB/forwarding | 写 GPR 意图 |
| `ex_mem_mem_re` | 1 | ID/EX | MEM/hazard | load |
| `ex_mem_mem_we` | 1 | ID/EX | MEM | store |
| `ex_mem_mem_size` | enum | ID/EX | MEM | byte/half/word |
| `ex_mem_mem_unsigned` | 1 | ID/EX | MEM | load 零扩展 |
| `ex_mem_wb_sel` | enum | ID/EX | WB | 写回来源 |
| `ex_mem_illegal_instr` | 1 | ID/EX | debug/后续异常 | 第一版调试 |

branch redirect 通常不需要保存到 EX/MEM，因为它在 EX 当拍就反馈给 IF/hazard 控制。但 EX/MEM 仍需保存 branch 指令本身的 commit/debug 信息。

## 第5章 MEM stage

### 5.1 MEM 做的事

MEM 阶段处理 data memory：

| 指令类别 | MEM 动作 |
|---|---|
| ALU 指令 | 不访问 dmem，直接把 ALU 结果传到 MEM/WB |
| load | 用 `ex_mem_alu_result` 作为地址，读取 dmem，并按 size/sign 扩展 |
| store | 用 `ex_mem_alu_result` 作为地址，把 `ex_mem_store_data` 写入 dmem |
| branch/jump | 通常不访问 dmem，只向后流动用于 trace |

第一版 dmem 固定响应，因此 load 数据可以在 MEM 阶段得到，并保存到 MEM/WB。若后续 dmem 有 wait state，则 MEM 阶段会成为会 stall 的 stage。

### 5.2 store byte enable 与写数据

建议 MEM 输出给 dmem 的数据已经按地址低位对齐：

| store | `addr[1:0]` | `dmem_be` | `dmem_wdata` |
|---|---|---|---|
| `SB` | `00` | `0001` | `{24'b0, data[7:0]}` 或有效 byte 放 lane0 |
| `SB` | `01` | `0010` | `{16'b0, data[7:0], 8'b0}` |
| `SB` | `10` | `0100` | `{8'b0, data[7:0], 16'b0}` |
| `SB` | `11` | `1000` | `{data[7:0], 24'b0}` |
| `SH` | `00` | `0011` | `{16'b0, data[15:0]}` |
| `SH` | `10` | `1100` | `{data[15:0], 16'b0}` |
| `SW` | `00` | `1111` | `data[31:0]` |

第一版可以暂不处理不对齐异常，但要明确测试只访问合法对齐地址，或在 testbench 中对非法地址报错。

### 5.3 load 扩展

load 从 dmem 读出 32 bit word 后，根据地址低位选择 byte/halfword：

| load | `mem_unsigned` | 行为 |
|---|---:|---|
| `LB` | 0 | 选择 byte，符号扩展 |
| `LBU` | 1 | 选择 byte，零扩展 |
| `LH` | 0 | 选择 halfword，符号扩展 |
| `LHU` | 1 | 选择 halfword，零扩展 |
| `LW` | 不关心 | 直接 32 bit |

### 5.4 MEM/WB register 字段

| 字段 | 位宽 | 来源 | 去向 | 说明 |
|---|---:|---|---|---|
| `mem_wb_valid` | 1 | EX/MEM valid | WB | 是否有效 |
| `mem_wb_pc` | 32 | EX/MEM | commit trace | PC |
| `mem_wb_pc_plus4` | 32 | EX/MEM | WB | JAL/JALR 写回 |
| `mem_wb_instr` | 32 | EX/MEM | commit trace | 原始 instruction |
| `mem_wb_rd_addr` | 5 | EX/MEM | WB | 目的寄存器 |
| `mem_wb_alu_result` | 32 | EX/MEM | WB/forwarding | ALU 结果 |
| `mem_wb_load_data` | 32 | MEM load 扩展 | WB/forwarding | load 写回数据 |
| `mem_wb_reg_we` | 1 | EX/MEM | WB/forwarding | 写 GPR 意图 |
| `mem_wb_wb_sel` | enum | EX/MEM | WB | 写回来源 |
| `mem_wb_illegal_instr` | 1 | EX/MEM | debug/后续异常 | 第一版调试 |

MEM/WB 是 forwarding 的重要来源。对 load 指令，真正可前递的通常是 `mem_wb_load_data`，不是 EX/MEM 中的地址。

## 第6章 WB stage

### 6.1 写回数据选择

WB 根据 `wb_sel` 选择写回数据：

| `wb_sel` | 写回数据 |
|---|---|
| `WB_ALU` | `mem_wb_alu_result` |
| `WB_MEM` | `mem_wb_load_data` |
| `WB_PC4` | `mem_wb_pc_plus4` |
| `WB_IMM` | 可选，`LUI` 若直接传 U-imm；也可让 LUI 用 ALU 输出 |

如果实现中 `LUI` 统一走 ALU，即 `0 + immU`，则可以不需要 `WB_IMM`。但控制表中要保持一致，不要一部分文档说 `LUI` 走 IMM，一部分 RTL 走 ALU 却没有说明。

### 6.2 GPR 写使能

真正写 GPR 必须满足：

```systemverilog
assign wb_gpr_we =
    mem_wb_valid &&
    mem_wb_reg_we &&
    (mem_wb_rd_addr != 5'd0);
```

这条规则同时解决三件事：

| 条件 | 防止什么 |
|---|---|
| `mem_wb_valid` | bubble/flush 指令写回 |
| `mem_wb_reg_we` | store/branch 等无写回指令写回 |
| `rd != x0` | 破坏 x0 恒为 0 |

### 6.3 commit trace

第一版可以这样定义 commit：

```systemverilog
assign commit_valid_o    = mem_wb_valid;
assign commit_pc_o       = mem_wb_pc;
assign commit_instr_o    = mem_wb_instr;
assign commit_reg_we_o   = wb_gpr_we;
assign commit_rd_o       = mem_wb_rd_addr;
assign commit_rd_wdata_o = wb_rd_wdata;
```

如果希望 trace 更接近参考模型，也可以只在指令没有 illegal/stop 时 commit。后续加入异常后，commit 定义需要重新严格化。

## 第7章 控制信号生命周期

### 7.1 从 ID 到后级

| 控制信号 | ID 生成 | EX 使用 | MEM 使用 | WB 使用 | 备注 |
|---|---:|---:|---:|---:|---|
| `uses_rs1` | 是 | forwarding/hazard | 否 | 否 | ID/EX 后可不再向后传 |
| `uses_rs2` | 是 | forwarding/hazard/store data | 否 | 否 | store data forwarding 需要 |
| `reg_we` | 是 | forwarding 判断 | forwarding 判断 | GPR 写回 | 一路传到 WB |
| `imm_sel` | 是 | 否 | 否 | 否 | 如果 ID 生成 imm，不需后传 |
| `alu_op` | 是 | ALU | 否 | 否 | 保存到 ID/EX |
| `op_a_sel/op_b_sel` | 是 | ALU 输入选择 | 否 | 否 | 保存到 ID/EX |
| `branch_op` | 是 | branch compare | 否 | 否 | 保存到 ID/EX |
| `jump/jalr` | 是 | target/redirect | 否 | 否 | 保存到 ID/EX |
| `mem_re` | 是 | 地址计算旁路判断 | load 访问 | forwarding 判断 | 至少传到 EX/MEM |
| `mem_we` | 是 | 否 | store 写 | 否 | store 副作用必须 valid gating |
| `mem_size` | 是 | 否 | byte enable/load 扩展 | 否 | 传到 EX/MEM |
| `mem_unsigned` | 是 | 否 | load 扩展 | 否 | 传到 EX/MEM |
| `wb_sel` | 是 | 否 | 否 | 写回选择 | 一路传到 MEM/WB |

控制信号的原则：谁在哪一级使用，就至少保存到那一级。不要让后级直接看当前 ID 正在 decode 的新指令。

### 7.2 副作用 gating 表

| 副作用 | 所在 stage | 必须满足 |
|---|---|---|
| dmem write | MEM | `ex_mem_valid && ex_mem_mem_we` |
| GPR write | WB | `mem_wb_valid && mem_wb_reg_we && rd != 0` |
| commit trace | WB | `mem_wb_valid` |
| redirect | EX | `id_ex_valid && branch_or_jump_taken` |
| illegal instruction report | 可在 ID/EX/WB | 必须和对应 valid 指令绑定 |

wrong-path 指令可能已经流到某一级。只要它的 valid 被清掉，后续 gating 正确，它就不会改变架构状态。

## 第8章 pipeline register 更新规则

### 8.1 通用规则

每个 pipeline register 都可以按类似规则更新：

```systemverilog
always_ff @(posedge clk_i or negedge rst_n_i) begin
    if (!rst_n_i) begin
        pipe_q <= '0;
    end else if (flush_pipe) begin
        pipe_q <= '0;
    end else if (stall_pipe) begin
        pipe_q <= pipe_q;
    end else if (bubble_pipe) begin
        pipe_q <= '0;
    end else begin
        pipe_q <= pipe_d;
    end
end
```

实际优先级要结合 `0825` 的 hazard 控制。通常：

- reset 最高。
- redirect/flush 要能杀掉 wrong-path 指令。
- load-use stall 要冻结 PC 和 IF/ID，同时向 ID/EX 插 bubble。
- 后续如果有 memory wait，MEM stall 会向前级施加 backpressure。

### 8.2 为什么 bubble 要清控制

理论上只要 `valid=0` 并且所有副作用都 valid gating，控制信号不清也不会出错。但教学核建议 bubble 清零：

| 好处 | 说明 |
|---|---|
| 波形清楚 | invalid 槽不会显示奇怪的 `mem_we=1` |
| 降低误用风险 | 即使某处漏了 valid gating，也更容易暴露 |
| debug 直观 | 看到全 0 控制就知道是 bubble |

但不能只依赖清控制。真正的安全边界仍然是 valid gating。

## 第9章 forwarding 接入点

本文不展开完整条件，只说明接入位置。

### 9.1 EX 操作数 forwarding

EX 输入操作数应先经过 forwarding 选择：

```text
id_ex_rs1_rdata ----+
ex_mem_result  -----+--> forward_a -> alu_operand_a / branch_operand_a
mem_wb_result  -----+

id_ex_rs2_rdata ----+
ex_mem_result  -----+--> forward_b -> alu_operand_b / branch_operand_b / store_data
mem_wb_result  -----+
```

注意：

- branch compare 也用 forwarding 后的操作数。
- store data 也用 forwarding 后的 `rs2`。
- `rd=x0` 不能作为 forwarding 来源。
- load 在 EX/MEM 时，EX/MEM 里的 `alu_result` 是地址，不是 load data。

### 9.2 WB 写回数据也是 forwarding 来源

MEM/WB forwarding 应该使用“最终写回数据”，而不是只看某个字段：

```systemverilog
assign mem_wb_forward_data = wb_rd_wdata;
```

这样 `WB_ALU/WB_MEM/WB_PC4/WB_IMM` 都能统一处理。

## 第10章 自查清单

| 检查项 | 目标 |
|---|---|
| 每个 pipeline register 都有 valid | bubble/flush 可控 |
| IF/ID 保存 PC 和 instruction | debug 和 decode 正常 |
| ID/EX 保存 rs1/rs2 地址和值 | forwarding/hazard 有依据 |
| ID/EX 保存所有 EX/MEM/WB 会用到的控制 | 后级不直接看当前 decoder |
| EX/MEM 保存 store_data | store 不回头读 ID |
| MEM/WB 保存 load_data 和 alu_result | WB 能正确选择 |
| dmem write 被 valid gating | wrong-path store 不会发生 |
| GPR write 被 valid 和 rd!=0 gating | bubble/x0 安全 |
| commit trace 使用 WB 阶段字段 | trace 与提交顺序一致 |
| branch redirect 使用 valid gating | invalid branch 不会改 PC |
