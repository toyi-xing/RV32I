# 0822 最小教学核工程目录、顶层接口与命名约定

> 文档编号：0822  
> 所属系列：082x RISC-V 最小教学核项目实践  
> 文档定位：定义第一版 RV32I 教学核项目的目录结构、模块边界、顶层接口、地址约定和命名风格  
> 前置文档：`0820 RISC-V最小教学核设计流程与方案.md`、`0821 RV32I最小教学核指令集、编码与译码参考.md`

本文解决的是“项目怎么摆放、模块怎么命名、接口怎么定”的问题。它不讲每条指令怎么译码，也不讲 forwarding/stall/flush 的具体判断；这些分别放在 `0821` 和 `0825`。本文的目标是让后续写 RTL 时有一套稳定约定，避免同一个概念在不同文件里叫不同名字。

本文默认项目假设：

| 项目 | 假设 |
|---|---|
| ISA | RV32I 主线可执行子集 |
| XLEN | 32 bit |
| 微架构 | 单发射、顺序五级流水线 |
| memory | 第一版分离 imem/dmem，固定响应，不处理 wait state |
| reset PC | `32'h0000_0000` |
| instruction 区域 | 从 `32'h0000_0000` 开始 |
| data 区域 | 从 `32'h0001_0000` 开始，dmem wrapper 可映射到内部 `mem[0]` |
| CSR/trap/cache/MMU | 第一版不实现 |

## 第1章 推荐工程目录

### 1.1 顶层目录

建议项目目录先保持简单：

```text
rv32i_teaching_core/
    rtl/
        core/
        common/
        mem/
    tb/
        sv/
        tests/
        model/
    sw/
        asm/
        c/
        linker/
    sim/
        verilator/
        iverilog/
    scripts/
    docs/
    build/
```

各目录职责如下：

| 目录 | 放什么 | 不建议放什么 |
|---|---|---|
| `rtl/core/` | CPU core 主体 RTL：stage、decoder、ALU、regfile、hazard 等 | testbench、脚本、临时波形 |
| `rtl/common/` | package、公共 typedef、常量、参数 | 某个模块私有逻辑 |
| `rtl/mem/` | 教学用 simple_rom/simple_ram 或 memory wrapper | 复杂 SoC 总线和真实 SRAM macro 细节 |
| `tb/sv/` | SystemVerilog testbench、monitor、简单 checker | 可综合 core RTL |
| `tb/tests/` | 测试清单、期望结果、回归配置 | 编译产物 |
| `tb/model/` | 简单 reference model、trace 比对脚本需要的辅助文件 | 正式 RTL |
| `sw/asm/` | 裸机汇编测试程序 | RTL |
| `sw/c/` | 极小 C 测试程序 | 大型软件工程 |
| `sw/linker/` | 链接脚本 | 工具链安装文件 |
| `sim/` | 仿真入口、filelist、Makefile 或脚本 | 源代码唯一副本 |
| `scripts/` | `bin2mem32.py`、回归脚本、trace 处理脚本 | 手写 RTL 逻辑 |
| `docs/` | 项目内说明文档 | 大体积波形 |
| `build/` | ELF、dump、mem、vvp、obj_dir、波形等生成物 | 需要手工维护的源文件 |

`build/` 应该可以删除后重新生成。不要把唯一的测试程序、脚本或 RTL 放到 `build/` 里。

### 1.2 推荐 RTL 文件

第一版 RTL 可以按下面文件拆分：

| 文件 | 主要模块/内容 | 说明 |
|---|---|---|
| `rtl/common/core_pkg.sv` | typedef、enum、参数、地址常量 | 尽量放公共定义，不放具体组合逻辑 |
| `rtl/core/core_top.sv` | `core_top` | CPU core 顶层 |
| `rtl/core/pc_reg.sv` | `pc_reg` | 取指 PC 寄存器和 PC 更新控制 |
| `rtl/core/if_stage.sv` | `if_stage` | 纯组合取指数据通路 |
| `rtl/core/id_stage.sv` | `id_stage` | 译码、读寄存器、立即数 |
| `rtl/core/ex_stage.sv` | `ex_stage` | ALU、branch compare、target |
| `rtl/core/mem_stage.sv` | `mem_stage` | load/store 访问和 load 扩展 |
| `rtl/core/wb_stage.sv` | `wb_stage` | 写回数据选择 |
| `rtl/core/regfile.sv` | `regfile` | 32 个 GPR，x0 恒为 0 |
| `rtl/core/decoder.sv` | `decoder` | 指令译码控制 |
| `rtl/core/imm_gen.sv` | `imm_gen` | I/S/B/U/J 立即数生成 |
| `rtl/core/alu.sv` | `alu` | 算术逻辑单元 |
| `rtl/core/branch_unit.sv` | `branch_unit` | branch 条件判断，可先并入 EX |
| `rtl/core/forwarding_unit.sv` | `forwarding_unit` | 前递选择，详细规则见 `0825` |
| `rtl/core/hazard_unit.sv` | `hazard_unit` | stall/bubble/flush 控制，详细规则见 `0825` |
| `rtl/core/pipe_reg.sv` | pipeline register 通用或分立模块 | 可选，也可直接写在 `core_top` |
| `rtl/mem/simple_rom.sv` | `simple_rom` | 仿真教学用 imem |
| `rtl/mem/simple_ram.sv` | `simple_ram` | 仿真教学用 dmem |

初期不必为了“模块化”拆得过碎。例如 `branch_unit` 可以先写在 `ex_stage` 内，等逻辑稳定后再拆。但 `decoder`、`imm_gen`、`regfile`、`alu` 建议尽早独立，因为它们容易单独验证。

### 1.3 推荐仿真文件

| 文件 | 作用 |
|---|---|
| `tb/sv/tb_core.sv` | 顶层 testbench，实例化 core、imem、dmem |
| `tb/sv/clock_reset.sv` | 可选，统一 clock/reset 产生 |
| `tb/sv/commit_monitor.sv` | 可选，采集提交 trace |
| `tb/sv/mem_monitor.sv` | 可选，观察 dmem store/tohost |
| `sim/verilator/files.f` | Verilator filelist |
| `sim/iverilog/files.f` | Icarus filelist |
| `scripts/bin2mem32.py` | binary 转 32 bit word `.mem` |
| `scripts/run_one.sh` | 可选，运行单个测试 |
| `scripts/run_regress.sh` | 可选，批量运行测试 |

第一版可以先只有 `tb_core.sv` 和 `files.f`，等测试数量上来后再拆 monitor 和脚本。

## 第2章 顶层接口

### 2.1 `core_top` 接口建议

第一版 `core_top` 不直接实例化具体 memory，这样后续可以在 testbench、FPGA wrapper、SoC wrapper 中替换 imem/dmem。

```systemverilog
module core_top (
    input  logic        clk_i,
    input  logic        rst_n_i,

    output logic [31:0] imem_addr_o,
    input  logic [31:0] imem_rdata_i,

    output logic        dmem_we_o,
    output logic [3:0]  dmem_be_o,
    output logic [31:0] dmem_addr_o,
    output logic [31:0] dmem_wdata_o,
    input  logic [31:0] dmem_rdata_i,

    output logic        commit_valid_o,
    output logic [31:0] commit_pc_o,
    output logic [31:0] commit_instr_o,
    output logic        commit_reg_we_o,
    output logic [4:0]  commit_rd_o,
    output logic [31:0] commit_rd_wdata_o
);
```

这个接口有几个刻意简化：

| 简化 | 含义 |
|---|---|
| imem 无 valid/ready | 第一版假设取指固定响应 |
| dmem 无 valid/ready | 第一版假设 load/store 固定响应 |
| imem 只读 | CPU 不能通过 store 改 instruction memory |
| dmem 读数据直接输入 | 第一版先按简单 memory 模型设计 |
| commit 信号只用于仿真/调试 | 不属于真实 ISA 状态，但非常利于验证 |

后续如果加入 wait state、总线或 cache，接口会变成 request/response 形式。那时要修改 hazard 控制和流水线 stall 逻辑，属于后续扩展。

### 2.2 imem 接口语义

第一版 imem 接口：

| 信号 | 方向 | 含义 |
|---|---|---|
| `imem_addr_o` | core -> imem | 当前 IF 阶段取指地址 |
| `imem_rdata_i` | imem -> core | 对应地址返回的 32 bit instruction |

若 imem 是 32 bit word array，常见索引是：

$$
\text{word\_index} = \text{addr}[AW+1:2]
$$

因为每条指令 4 字节，`addr[1:0]` 在不支持 C 扩展时应为 `2'b00`。

### 2.3 dmem 接口语义

第一版 dmem 接口：

| 信号 | 方向 | 含义 |
|---|---|---|
| `dmem_we_o` | core -> dmem | store 写使能 |
| `dmem_be_o[3:0]` | core -> dmem | byte enable |
| `dmem_addr_o` | core -> dmem | load/store 地址 |
| `dmem_wdata_o` | core -> dmem | store 写数据，已经按 byte lane 对齐 |
| `dmem_rdata_i` | dmem -> core | load 读出的 32 bit word |

本系列推荐让 `mem_stage` 输出已经对齐后的 `dmem_wdata_o` 和 `dmem_be_o`。例如对 `SB`：

| `addr[1:0]` | `dmem_be_o` | `dmem_wdata_o` 有效 byte |
|---|---|---|
| `00` | `0001` | `[7:0]` |
| `01` | `0010` | `[15:8]` |
| `10` | `0100` | `[23:16]` |
| `11` | `1000` | `[31:24]` |

这样 simple_ram 只需要按 byte enable 写对应 lane，不必知道这是 `SB/SH/SW` 哪一种指令。

### 2.4 commit/debug 接口语义

commit 信号用于验证，不是必须综合到最终芯片里。第一版可以把 WB 阶段 valid 指令视为 commit。

| 信号 | 含义 |
|---|---|
| `commit_valid_o` | 本 cycle 有一条 valid 指令提交 |
| `commit_pc_o` | 提交指令的 PC |
| `commit_instr_o` | 提交指令的原始 instruction |
| `commit_reg_we_o` | 该提交指令是否写 GPR |
| `commit_rd_o` | 写回目的寄存器 |
| `commit_rd_wdata_o` | 写回数据 |

store 也会改变架构可见 memory，但它通常在 MEM 阶段发生。第一版可以另外在 testbench 里观察 dmem 写事件；后续 `0827` 会更系统地定义 trace 格式。

## 第3章 地址与 memory map 约定

### 3.1 第一版推荐 memory map

为了和 `0826` 的链接脚本一致，建议第一版采用：

| 区域 | 起始地址 | 用途 |
|---|---:|---|
| IMEM | `0x0000_0000` | 指令、只读常量 |
| DMEM | `0x0001_0000` | 数据、栈、测试结果 |

即使 core 内部是 Harvard 结构，软件看到的地址仍然可以有区域概念。这样后续接 SoC/MMIO 时更自然。

### 3.2 dmem wrapper 地址映射

第一版 dmem 内部可以从 `mem[0]` 开始存数据，但 CPU 发出的地址是 `0x0001_0000`。wrapper 可以做基址转换：

```systemverilog
localparam logic [31:0] DMEM_BASE = 32'h0001_0000;

logic [31:0] dmem_offset;
assign dmem_offset = dmem_addr_i - DMEM_BASE;

assign word_addr = dmem_offset[AW+1:2];
```

这意味着：

| CPU 地址 | dmem 内部位置 |
|---:|---:|
| `0x0001_0000` | `mem[0]` |
| `0x0001_0004` | `mem[1]` |
| `0x0001_0008` | `mem[2]` |

第一版可以不做越界异常，但 testbench 最好能检查访问是否落在合法范围内。非法地址至少应打印错误，避免 debug 时默默读写错误位置。

### 3.3 reset PC 与链接地址

必须保持一致：

| 项目 | 典型值 |
|---|---:|
| `RESET_PC` | `32'h0000_0000` |
| linker script 中 `.text` 起始地址 | `0x00000000` |
| imem `$readmemh` 加载起点 | `mem[0]` |

如果三者不一致，CPU 可能从空 memory 或错误地址开始执行。

## 第4章 命名约定

### 4.1 端口后缀

| 后缀 | 含义 | 示例 |
|---|---|---|
| `_i` | module input | `clk_i`、`instr_i` |
| `_o` | module output | `imem_addr_o`、`reg_we_o` |
| `_q` | 寄存器当前值 | `pc_q`、`if_id_valid_q` |
| `_d` | 寄存器下一拍值 | `pc_d`、`if_id_valid_d` |
| `_en` | 使能 | `pc_en`、`if_id_en` |
| `_we` | 写使能 | `reg_we`、`dmem_we` |
| `_be` | byte enable | `dmem_be` |
| `_sel` | 多路选择 | `wb_sel`、`op_a_sel` |

`_q/_d` 对写时序逻辑很有帮助。建议对核心状态寄存器使用这种风格，组合中间信号可以不用强制。

### 4.2 pipeline 前缀

推荐使用 stage 或 pipeline register 前缀：

| 前缀 | 含义 |
|---|---|
| `if_` | IF stage 内部信号 |
| `id_` | ID stage 内部信号 |
| `ex_` | EX stage 内部信号 |
| `mem_` | MEM stage 内部信号 |
| `wb_` | WB stage 内部信号 |
| `if_id_` | IF/ID pipeline register 字段 |
| `id_ex_` | ID/EX pipeline register 字段 |
| `ex_mem_` | EX/MEM pipeline register 字段 |
| `mem_wb_` | MEM/WB pipeline register 字段 |

例如：

```text
id_ex_rs1_addr
id_ex_rs1_rdata
ex_mem_alu_result
mem_wb_rd_wdata
```

不要把 `id_rs1_rdata` 和 `id_ex_rs1_rdata` 混用。前者表示 ID 当前组合输出，后者表示已经锁存在 ID/EX 寄存器中的值。

### 4.3 valid/stall/flush/bubble 命名

建议统一：

| 信号 | 含义 |
|---|---|
| `if_valid` | IF 当前槽是否有效 |
| `if_id_valid_q` | IF/ID 寄存器中的指令是否有效 |
| `stall_if` | IF/PC 保持 |
| `stall_id` | IF/ID 保持 |
| `bubble_ex` | ID/EX 写入 invalid bubble |
| `flush_if_id` | 清掉 IF/ID 中的错路径指令 |
| `flush_id_ex` | 清掉 ID/EX 中的错路径指令 |
| `redirect_valid` | 本 cycle 要改变 PC |
| `redirect_pc` | 新 PC |

`stall` 是“保持”，`bubble` 是“插入空槽”，`flush/kill` 是“清掉错路径”。不要用一个信号名同时表示多种动作。

### 4.4 控制信号命名

建议控制信号语义直接：

| 信号 | 含义 |
|---|---|
| `uses_rs1` | 该指令真的读取 `rs1` |
| `uses_rs2` | 该指令真的读取 `rs2` |
| `reg_we` | 该指令有写 GPR 意图 |
| `mem_re` | load |
| `mem_we` | store |
| `mem_size` | byte/halfword/word |
| `mem_unsigned` | load 零扩展 |
| `imm_sel` | 立即数类型 |
| `alu_op` | ALU 运算 |
| `branch_op` | branch 比较类型 |
| `wb_sel` | 写回来源 |

特别注意：`reg_we` 只是“指令意图写 GPR”，真正写寄存器还要满足：

```systemverilog
assign gpr_we = wb_valid && wb_reg_we && (wb_rd_addr != 5'd0);
```

## 第5章 公共 package 建议

### 5.1 `core_pkg.sv`

建议把公共类型集中到 package：

```systemverilog
package core_pkg;
    parameter int XLEN = 32;
    parameter int ILEN = 32;

    parameter logic [31:0] RESET_PC  = 32'h0000_0000;
    parameter logic [31:0] IMEM_BASE = 32'h0000_0000;
    parameter logic [31:0] DMEM_BASE = 32'h0001_0000;

    typedef enum logic [2:0] {
        IMM_NONE,
        IMM_I,
        IMM_S,
        IMM_B,
        IMM_U,
        IMM_J
    } imm_sel_e;

    typedef enum logic [3:0] {
        ALU_ADD,
        ALU_SUB,
        ALU_SLL,
        ALU_SLT,
        ALU_SLTU,
        ALU_XOR,
        ALU_SRL,
        ALU_SRA,
        ALU_OR,
        ALU_AND
    } alu_op_e;

    typedef enum logic [2:0] {
        BR_NONE,
        BR_EQ,
        BR_NE,
        BR_LT,
        BR_GE,
        BR_LTU,
        BR_GEU
    } branch_op_e;

    typedef enum logic [1:0] {
        WB_ALU,
        WB_MEM,
        WB_PC4,
        WB_IMM
    } wb_sel_e;

    typedef enum logic [1:0] {
        MEM_BYTE,
        MEM_HALF,
        MEM_WORD
    } mem_size_e;
endpackage
```

后续如果 enum 增长，不要随意改已有枚举含义。新增值可以加，但已有值的语义要稳定。

### 5.2 pipeline struct 是否使用

可以使用 packed struct 表达 pipeline register 字段，例如：

```systemverilog
typedef struct packed {
    logic        valid;
    logic [31:0] pc;
    logic [31:0] pc_plus4;
    logic [31:0] instr;
} if_id_pipe_t;
```

struct 的优点：

| 优点 | 说明 |
|---|---|
| 字段集中 | IF/ID、ID/EX 等寄存器包含什么一眼能看出 |
| 传递方便 | stage 接口可以传一个 struct |
| 默认清零方便 | bubble 时可以整体赋 `'0` |

struct 的注意点：

| 注意点 | 说明 |
|---|---|
| 初学 debug | 波形层级可能比单独信号稍微不直观 |
| 工具兼容 | 老工具对复杂 struct 支持可能一般 |
| 不要滥用 | 不相关信号不要硬塞进一个 struct |

第一版可以二选一：喜欢清晰波形就用分立信号，喜欢接口整洁就用 struct。无论选哪种，字段命名必须稳定。

## 第6章 filelist 与编译顺序

如果使用 package，filelist 中 package 要排在使用者前面：

```text
rtl/common/core_pkg.sv
rtl/core/alu.sv
rtl/core/imm_gen.sv
rtl/core/decoder.sv
rtl/core/regfile.sv
rtl/core/pc_reg.sv
rtl/core/if_stage.sv
rtl/core/id_stage.sv
rtl/core/ex_stage.sv
rtl/core/mem_stage.sv
rtl/core/wb_stage.sv
rtl/core/forwarding_unit.sv
rtl/core/hazard_unit.sv
rtl/core/core_top.sv
rtl/mem/simple_rom.sv
rtl/mem/simple_ram.sv
tb/sv/tb_core.sv
```

有些工具支持自动分析依赖，有些工具要求顺序严格。第一版建议手动维护 filelist，少依赖工具猜测。

## 第7章 不建议的做法

| 不建议 | 原因 |
|---|---|
| stage 名字频繁变 | 文档、波形、测试脚本会全部跟着乱 |
| 把 testbench 逻辑写进 core RTL | 后续综合和上板会混乱 |
| decoder 里直接改 PC | PC 更新应由控制路径统一处理 |
| hazard unit 里直接改 GPR 数据 | hazard unit 应输出控制/选择，不直接拥有数据状态 |
| imem/dmem 地址假设散落各处 | 应集中在 package 和 wrapper |
| 用 `x1/x2` 这种名字当硬件信号 | 容易和 GPR 编号混淆，建议用 `rs1_addr`、`rs1_rdata` |
| 不保留原始 `instr` | debug 和 commit trace 会困难 |
| pipeline register 不保存 `pc` | branch/debug/trace 都会不方便 |

## 第8章 自查清单

开始写 RTL 前，先检查：

| 检查项 | 是否明确 |
|---|---|
| reset PC 是多少 | 是/否 |
| `.text` 链接地址是否等于 reset PC | 是/否 |
| dmem 基址是否明确 | 是/否 |
| core_top 是否不依赖具体 memory 实现 | 是/否 |
| imem/dmem 第一版是否固定响应 | 是/否 |
| stage 和 pipeline register 前缀是否统一 | 是/否 |
| `valid/stall/flush/bubble` 名字是否有明确动作 | 是/否 |
| x0 写屏蔽位置是否明确 | 是/否 |
| commit/debug 信号是否只用于验证 | 是/否 |
| filelist 是否包含 package 且顺序正确 | 是/否 |
