# 0830 RV32I 教学核后续完善路线：从 v2.0 到最小完整裸机核心

> 文档编号：0830  
> 所属系列：083x RV32I 教学核后续完善阶段  
> 文档定位：在当前五级流水线 v2.0 基础上，规划下一阶段如何把核从“能跑 RV32I 主路径程序”推进到“更接近真实裸机 MCU core”  
> 当前基础：`v2.0-pipeline-complete`，五级流水线、forwarding、load-use stall、branch/JAL/JALR redirect flush/kill 已完成  
> 前置文档：`0803 CSR、异常中断与特权级.md`、`0804 RISC-V SoC、MMIO与外设互联.md`、`0820 RISC-V最小教学核设计流程与方案.md`、`0825 Hazard控制：forwarding、stall、flush与kill.md`、`0827 Testbench、commit trace与测试集组织.md`、`0829 综合、FPGA上板与SoC扩展方向.md`

本篇是 `083x` 系列的总规划，不替代后续实现用的 `plan.md`。

`0820`～`0829` 的目标是从零搭出一个可仿真的 RV32I 教学核；`0830` 之后的目标则不同：当前核的取指、译码、执行、访存、写回、data hazard 和 control hazard 主干已经跑通，下一阶段要补的是**架构控制面、异常恢复能力、最小外设面和工程收口能力**。

本阶段暂时不以跑 OS 为目标。因此后续规划会优先支持 M-mode 裸机程序，而不是 S-mode、U-mode、MMU、页表、进程调度或 SBI。

## 第0章 083x 文档的定位

### 0.1 0830 和 0831～083x 的分工

`0830` 只回答三件事：

1. 当前 RTL 离“更完整的裸机 core”还差哪些能力。
2. 这些能力按什么顺序加最合理。
3. 每一步应该由哪篇 `083x` 文档承接。

`0831`～`083x` 则负责写某一步的规划，但仍然不是最终执行计划。也就是说：

| 文档 | 作用 | 不做什么 |
|---|---|---|
| `0830` | 总路线，定义大目标、阶段边界和推荐顺序 | 不列代码修改顺序 |
| `0831` | 第一步稍微详细的功能规划 | 不写逐文件逐信号实施 plan |
| `plan.md` | 真正开始某一步时的执行计划 | 不承担长期总纲角色 |

本次建议先生成 `0830` 和 `0831`。不要一次性把 `0832`～`083n` 全写满，因为第一步加入 CSR/trap 后，顶层接口、pipeline register 字段、commit trace 和 testbench 观察点都会变化。等第一步完成后，再根据真实 RTL 形态写下一篇 `083x` 会更准确。

### 0.2 本阶段的大目标

下一阶段的大目标可以定义为：

```text
单 hart、M-mode-only、可运行裸机 trap handler 和最小 MMIO 外设程序的 RV32I 教学 core。
```

它应该具备：

| 能力 | 目标状态 |
|---|---|
| RV32I 完整性 | 补上当前缺少的 `FENCE/ECALL/EBREAK`，并按需要支持 Zicsr 指令 |
| CSR | 有最小 M-mode CSR 文件，软件能配置 trap 入口、读取 trap 原因、执行 `MRET` 返回 |
| exception | 非法指令、`ECALL`、`EBREAK`、访存不对齐、取指/跳转目标不对齐能进入 precise trap |
| interrupt | 后续支持 machine software/timer/external interrupt 的 pending、enable、entry 和 return |
| MMIO | 后续有最小地址译码，能接 GPIO/UART TX/timer/accelerator 控制寄存器等外设 |
| memory 接口 | 后续能从固定 1-cycle memory 过渡到带等待的 memory/MMIO 访问，并为内部总线或 APB-Lite 过渡留边界 |
| 验证 | 有 CSR/trap/interrupt/MMIO directed test，后续补 assertion、scoreboard、coverage 或轻量 UVM |
| 工程化 | 保持可综合主路径，逐步补 Yosys 检查、最小 SoC wrapper、FPGA wrapper 或 accelerator demo |

它暂时不包括：

| 暂不做 | 原因 |
|---|---|
| S-mode/U-mode | 不跑 OS 时价值不如 M-mode trap/interrupt/MMIO 高 |
| delegation | 只有 M-mode 时不需要 `medeleg/mideleg` |
| MMU/TLB/page fault | 主要服务 OS 和虚拟内存 |
| PMP | 可以作为安全/权限扩展，暂不作为当前阶段主线 |
| cache | 会引入 miss、refill、backpressure 和一致性话题，晚于最小 MMIO/interrupt |
| 多 hart | 会引入核间中断、共享存储和一致性问题 |
| Debug Module/JTAG | 有价值，但不应抢在 trap/interrupt 之前 |
| 复杂 NPU/TPU | 需要先有 MMIO、interrupt、wait state 或总线边界；第一版 accelerator 可先做 MMIO slave |

## 第1章 当前 RTL 状态

### 1.1 已经具备的基础

当前 RTL 已经完成了一个很好的下一阶段起点：

| 模块/能力 | 当前状态 |
|---|---|
| `core_pipeline5.sv` | IF/ID/EX/MEM/WB 五级流水线顶层 |
| `pipeline_pkg.sv` | 有 pipeline register struct 和 valid bit 语义 |
| `decoder.sv` | 支持当前 37 条 RV32I 主路径指令，已有 SYSTEM 类注释占位 |
| `ex_stage.sv` | branch/JAL/JALR 在 EX 产生 redirect |
| `mem_stage.sv` | load/store byte lane、符号扩展、store byte enable 已完成 |
| `mem_stage.sv` | 已能检测 load/store misaligned，但还只是上报，不进入 trap |
| `hazard_unit.sv` | 已处理 load-use stall 和 EX redirect flush/kill |
| `forwarding_unit.sv` | 已处理 EX/MEM、MEM/WB 到 EX 的 RAW forwarding |
| `tb_core_pipeline5.sv` | 有 PASS/FAIL、commit trace、illegal/misaligned 观察、DMEM/stack 统计 |
| `sw/asm` 和 `sw/c` | 已有基础 directed test、pipeline hazard test 和 C smoke/control test |

这些基础很关键。尤其是 valid bit、flush/kill 和 commit trace，后续 precise trap 会直接复用这些设计思想。

### 1.2 当前明确缺少的能力

当前核离“更完整的裸机 core”主要差下面几类内容：

| 缺口 | 当前表现 | 后续要补什么 |
|---|---|---|
| SYSTEM 类指令 | `OPCODE_SYSTEM`、`ECALL/EBREAK` 仍是注释占位 | 译码 `ECALL/EBREAK/MRET` 和 CSR 指令 |
| `FENCE` | `OPCODE_MISC_MEM` 仍是注释占位 | 无 cache/MMIO 顺序模型时可先按 NOP 完成 |
| CSR 文件 | 没有 `mstatus/mtvec/mepc/mcause/mtval` 等状态 | 新增 M-mode CSR storage、读写和 WARL 规则 |
| precise trap | illegal/misaligned 只在 testbench 中停止 | trap entry 写 CSR、redirect 到 `mtvec`、kill younger 指令 |
| `MRET` | 没有 trap return | 从 `mepc` 返回并恢复 `mstatus` 位 |
| interrupt | 没有 `mie/mip`、外部 interrupt 输入或 timer pending | 后续加入 machine interrupt 屏蔽、挂起和优先级 |
| MMIO | dmem 只接 simple RAM | 后续加入地址译码、GPIO/UART/timer |
| 可变延迟访问 | imem/dmem 默认固定响应 | 后续补 ready/valid 或 stall/backpressure |
| 更系统验证 | 当前覆盖 RV32I 主路径和 hazard | 后续补 trap、CSR、interrupt、MMIO、异常副作用测试 |

换句话说，当前不是缺 ALU/GPR/流水线主干，而是缺“真实 core 能被软件控制、出错能恢复、能和外设交互”的系统面。

### 2.2 每一步的边界

#### 阶段1：最小 M-mode CSR/trap

目标是让当前核从“遇到非法/不对齐就由 testbench 停止”，升级为“硬件进入 trap handler，软件能处理并返回”。

本阶段应包含：

- `FENCE` 按 NOP 支持。
- `ECALL/EBREAK/MRET` 译码和执行。
- Zicsr 的 6 条 CSR 指令：`CSRRW/CSRRS/CSRRC/CSRRWI/CSRRSI/CSRRCI`。
- 最小 M-mode CSR：`mstatus/mtvec/mepc/mcause/mtval/mscratch`。
- 建议补只读 ID CSR：`misa/mvendorid/marchid/mimpid/mhartid`。
- illegal instruction、instruction address misaligned、load/store address misaligned、`ECALL`、`EBREAK` 的 precise trap。
- trap redirect 优先级和 younger instruction kill。
- trap/CSR directed test。

不包含：

- interrupt。
- MMIO timer。
- S/U mode。
- delegation。
- nested trap 的完整处理策略。

#### 阶段2：最小 memory map 与 MMIO

目标是把 dmem 从“只有 simple RAM”扩展成“SRAM + 外设寄存器”的地址空间。

建议内容：

- 定义 memory map，例如 IMEM、SRAM、MMIO base。
- 新增最小平台 wrapper，把 core、ROM、RAM、MMIO 译码和外设连接起来。
- 新增简单地址译码模块。
- 接入 GPIO 寄存器块。
- 接入 UART TX，至少能输出字符。
- 加入未映射 load/store access fault，复用阶段1的 trap 路径。
- 软件测试从写 DMEM pass flag 扩展到写 MMIO 寄存器。

不建议一开始就接 AXI/APB。当前阶段可以先用固定响应的简单地址译码，等外设数量、wait state 或 accelerator 访问需求明确后，再在阶段4抽象成 request/response 或 ready/valid 内部总线。

#### 阶段3：machine interrupt 与 timer

目标是让 core 能在指令边界接受 machine interrupt，并通过 handler 返回。

建议内容：

- `mstatus.MIE/MPIE/MPP` 与 interrupt entry/return 规则。
- `mie/mip` 的 machine software/timer/external interrupt 位。
- core 顶层加入 `msip/mtip/meip` 或统一 interrupt 输入。
- 若阶段2已有 timer MMIO，则加入 `mtime/mtimecmp` 产生 `MTIP`。
- interrupt 与 synchronous exception 的优先级。
- interrupt entry 写 `mepc/mcause`，`mcause[31]` 表示 interrupt。
- `MRET` 返回 interrupt 前 PC。

#### 阶段4：可变延迟 memory/MMIO、简化内部总线与流水线 backpressure

目标是摆脱“所有 memory/MMIO 都固定响应”的假设。

建议内容：

- imem/dmem 或内部 data bus 加入 request/response 或 ready/valid。
- IF 在取指未返回时 stall。
- MEM 在 load/store/MMIO 未完成时 stall，并向前级施加 backpressure。
- response 中携带 `rdata/error`，把阶段2的 access fault 语义自然迁移到 response error。
- stall、flush、trap redirect、load-use、未完成 memory request 之间重新定义优先级。
- 增加 wait state 仿真模型，测试外设慢响应时流水线不丢指令。

这一阶段会明显改变 hazard/control 网络，所以应晚于 CSR/trap 和固定响应 MMIO 的第一版。

这一阶段可以理解为“先做项目内的简化总线”，不必一开始就上 APB/AHB/AXI：

```text
request:
  valid, ready, write, addr, wdata, be

response:
  valid, rdata, error
```

简化总线稳定后，再包装 APB-Lite/AXI-Lite 会更自然。若直接在阶段2接标准总线，容易同时引入地址译码、外设寄存器、握手状态机、流水线 stall 和错误响应，调试维度过多。

#### 阶段5：验证、可综合和上板收口

目标是让项目从“功能能跑”更接近“工程能交付”。

建议内容：

- 扩展 regression：RV32I 主路径、CSR、trap、interrupt、MMIO、wait state。
- commit trace 增加 trap/interrupt/CSR 事件。
- 增加关键 assertion：x0 不变、wrong-path 无副作用、trap 后 younger 不提交、misaligned store 不写 memory。
- 引入简单 reference/scoreboard 或 trace 对比。
- 在接口稳定后补轻量 UVM：monitor 观察 commit/trap/MMIO，总线 agent 观察 request/response，scoreboard 检查 PASS/FAIL、trap cause 和 MMIO 副作用。
- Yosys 可综合性检查脚本。
- FPGA top wrapper：clock/reset、BRAM 初始化、UART/GPIO 可观察输出。

UVM 不建议抢在阶段2之前大规模做。阶段2～4 会不断改变平台接口、MMIO 观察点和 memory response 形态，过早搭 UVM 容易频繁重构。更合适的切入点是：固定响应 MMIO、interrupt 和 wait-state/backpressure 基本成型后，把已有 directed test 的观察点沉淀成 UVM monitor/scoreboard/coverage。

#### 阶段6：accelerator/NPU 控制 demo

目标是用当前 CPU 作为控制核，调度一个小型计算外设或 NPU/TPU 雏形。

第一版 accelerator 不需要直接上复杂总线，可以先作为 MMIO slave：

```text
ACCEL_CTRL    bit0 start
ACCEL_STATUS  bit0 busy, bit1 done
ACCEL_IN0     输入寄存器
ACCEL_IN1     输入寄存器
ACCEL_RESULT  输出寄存器
```

CPU 侧软件流程：

```text
写输入寄存器
写 CTRL.start
轮询 STATUS.done，或后续等待 interrupt
读 RESULT
```

这个阶段可以分三步逐渐增强：

| 子阶段 | accelerator 形态 | 依赖 |
|---|---|---|
| demo 1 | 固定响应 MMIO slave，CPU 轮询 done | 阶段2 |
| demo 2 | done 后触发 interrupt，CPU handler 处理完成事件 | 阶段3 |
| demo 3 | 多拍计算或慢响应 status，CPU 能等待/轮询 | 阶段4 |

等 accelerator 需要主动读写 SRAM，例如矩阵乘从 SRAM 读 A/B、写 C，就需要 DMA/master 接口、仲裁或更正式的 interconnect：

```text
CPU 配置 SRC_A/SRC_B/DST/LEN/START
accelerator 作为 master 访问 SRAM
完成后置 done 或发 interrupt
CPU 读状态或处理 interrupt
```

这时再考虑 APB-Lite/AXI-Lite 控制面和 SRAM/AHB/AXI 数据面会更合理。

## 第3章 当前距离目标还差多少

从工作量上看，当前已经完成了 CPU 主干里最难讲清楚的一部分：五级流水线数据通路、data hazard 和 control hazard。下一阶段不是重写 core，而是在现有主干上补控制状态和系统接口。

大致差距可以这样估计：

| 类别 | 差距 | 影响范围 |
|---|---|---|
| ISA 补全 | 小到中 | `core_pkg.sv`、`decoder.sv`、`id_stage.sv` |
| CSR 文件 | 中 | 新增 `csr_file.sv`，影响 CSR 指令写回和 trap entry |
| precise trap | 中到大 | pipeline register、hazard/flush 优先级、MEM/WB side effect gating |
| interrupt | 中 | CSR、顶层接口、trap controller、testbench |
| MMIO | 中 | memory map、dmem wrapper、外设模块、软件测试 |
| wait state/简化总线 | 中到大 | IF/MEM stall/backpressure、request/response、hazard 优先级 |
| 验证收口 | 中到大 | directed test、trace、assertion、scoreboard、UVM、regression 脚本 |
| accelerator/NPU 控制 | 中 | MMIO 寄存器、interrupt、后续 bus master/仲裁 |

因此下一阶段最合理的判断是：

```text
数据通路主干已经完成；
要成为更完整的裸机 core，还需要补 4～5 个系统性能力包：
CSR/trap、MMIO、interrupt、可变延迟访问/简化总线、验证/综合收口。
在这些能力稳定后，再接 accelerator/NPU 会更顺，CPU 可以自然作为控制核。
```

## 第4章 为什么第一步选最小 M-mode CSR/trap

第一步选 CSR/trap，而不是先选 MMIO、interrupt 或 M 扩展，原因如下：

| 原因 | 说明 |
|---|---|
| 当前 RTL 已经有基础信号 | `illegal_instr_o`、`mem_misaligned_o`、valid bit、flush/kill 都能复用 |
| trap 是 interrupt 的前置条件 | interrupt entry/return 本质上复用 trap entry/`MRET` |
| trap 是错误处理的前置条件 | MMIO access fault、bus error、misaligned 都需要统一入口 |
| trap 能验证 precise kill 能力 | 这比普通 branch flush 更接近真实 CPU 提交语义 |
| CSR 是软件控制面的起点 | 没有 `mtvec/mepc/mcause`，软件无法可靠知道发生了什么 |

第一步完成后，当前核会从“只适合跑保证合法且对齐的测试程序”，升级为“可以跑带 trap handler 的 M-mode 裸机程序”。这会明显提升项目完整度。
