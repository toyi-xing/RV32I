# 0830 RV32I 教学核后续完善路线：从 v2.0 到最小完整裸机核心

> 文档编号：0830  
> 所属系列：083x RV32I 教学核后续完善阶段  
> 文档定位：在当前五级流水线 v2.0 基础上，规划下一阶段如何把核从“能跑 RV32I 主路径程序”推进到“更接近真实裸机 MCU core”  
> 当前基础：`v2.0-pipeline-complete`，五级流水线、forwarding、load-use stall、branch/JAL/JALR redirect flush/kill 已完成  
> 前置文档：`0803 CSR、异常中断与特权级.md`、`0804 RISC-V SoC、MMIO与外设互联.md`、`0820 RISC-V最小教学核设计流程与方案.md`、`0825 Hazard控制：forwarding、stall、flush与kill.md`、`0827 Testbench、commit trace与测试集组织.md`、`0829 综合、FPGA上板与SoC扩展方向.md`

本篇是 `083x` 系列的总规划，不替代后续实现用的 `plan.md`。

`0820`～`0829` 的目标是从零搭出一个可仿真的 RV32I 教学核；`0830` 之后的目标则不同：当前核的取指、译码、执行、访存、写回、data hazard 和 control hazard 主干已经跑通，下一阶段要补的是**架构控制面、异常恢复能力、最小外设面和工程收口能力**。

本阶段暂时不以跑 OS 为目标。因此后续规划会优先支持 M-mode 裸机程序，而不是 S-mode、U-mode、MMU、页表、进程调度或 SBI。

## 第 0 章 083x 文档的定位

### 0.1 0830 和 0831～083x 的分工

`0830` 只回答三件事：

1. 当前 RTL 离“更完整的裸机 core”还差哪些能力。
2. 这些能力按什么顺序加最合理。
3. 每一步应该由哪篇 `083x` 文档承接。

`0831`～`083x` 则负责写某一步的规划，但仍然不是最终执行计划。也就是说：

| 文档 | 作用 | 不做什么 |
|---|---|---|
| `0830` | 总路线，定义大目标、阶段边界和推荐顺序 | 不列代码修改顺序 |
| `0831` | 第一步稍微详细的功能规划 | 不写逐文件逐信号实施 plan |
| `plan.md` | 真正开始某一步时的执行计划 | 不承担长期总纲角色 |

本次建议先生成 `0830` 和 `0831`。不要一次性把 `0832`～`083n` 全写满，因为第一步加入 CSR/trap 后，顶层接口、pipeline register 字段、commit trace 和 testbench 观察点都会变化。等第一步完成后，再根据真实 RTL 形态写下一篇 `083x` 会更准确。

### 0.2 本阶段的大目标

下一阶段的大目标可以定义为：

```text
单 hart、M-mode-only、可运行裸机 trap handler 和最小 MMIO 外设程序的 RV32I 教学 core。
```

它应该具备：

| 能力 | 目标状态 |
|---|---|
| RV32I 完整性 | 补上当前缺少的 `FENCE/ECALL/EBREAK`，并按需要支持 Zicsr 指令 |
| CSR | 有最小 M-mode CSR 文件，软件能配置 trap 入口、读取 trap 原因、执行 `MRET` 返回 |
| exception | 非法指令、`ECALL`、`EBREAK`、访存不对齐、取指/跳转目标不对齐能进入 precise trap |
| interrupt | 后续支持 machine software/timer/external interrupt 的 pending、enable、entry 和 return |
| MMIO | 后续有最小地址译码，能接 GPIO/UART/timer 等外设，并预留 accelerator 控制寄存器窗口 |
| memory 接口 | 后续能从固定 1-cycle memory 过渡到带等待的 memory/MMIO 访问，并为内部简化总线和 AXI-Lite 过渡留边界 |
| 验证 | 有 CSR/trap/interrupt/MMIO directed test，后续按 wait-state/SVA、UVM demo、AXI-Lite UVM、SoC/CPU 级 UVM 逐步补齐 |
| 工程化 | 保持可综合主路径，逐步补 Yosys 检查、最小 SoC wrapper、AXI-Lite wrapper/interconnect 和可扩展外设接口 |

它暂时不包括：

| 暂不做 | 原因 |
|---|---|
| S-mode/U-mode | 不跑 OS 时价值不如 M-mode trap/interrupt/MMIO 高 |
| delegation | 只有 M-mode 时不需要 `medeleg/mideleg` |
| MMU/TLB/page fault | 主要服务 OS 和虚拟内存 |
| PMP | 可以作为安全/权限扩展，暂不作为当前阶段主线 |
| cache | 会引入 miss、refill、backpressure 和一致性话题，晚于最小 MMIO/interrupt |
| 多 hart | 会引入核间中断、共享存储和一致性问题 |
| Debug Module/JTAG | 有价值，但不应抢在 trap/interrupt 之前 |
| 复杂 NPU/TPU | 不属于当前 RV32I 教学核主线；可在 AXI-Lite 控制面稳定后另起专题展开 |

## 第 1 章 当前 RTL 状态

### 1.1 已经具备的基础

当前 RTL 已经完成了一个很好的下一阶段起点：

| 模块/能力 | 当前状态 |
|---|---|
| `core_pipeline5.sv` | IF/ID/EX/MEM/WB 五级流水线顶层 |
| `pipeline_pkg.sv` | 有 pipeline register struct 和 valid bit 语义 |
| `decoder.sv` | 支持当前 37 条 RV32I 主路径指令，已有 SYSTEM 类注释占位 |
| `ex_stage.sv` | branch/JAL/JALR 在 EX 产生 redirect |
| `mem_stage.sv` | load/store byte lane、符号扩展、store byte enable 已完成 |
| `mem_stage.sv` | 已能检测 load/store misaligned，但还只是上报，不进入 trap |
| `hazard_unit.sv` | 已处理 load-use stall 和 EX redirect flush/kill |
| `forwarding_unit.sv` | 已处理 EX/MEM、MEM/WB 到 EX 的 RAW forwarding |
| `tb_core_pipeline5.sv` | 有 PASS/FAIL、commit trace、illegal/misaligned 观察、DMEM/stack 统计 |
| `sw/asm` 和 `sw/c` | 已有基础 directed test、pipeline hazard test 和 C smoke/control test |

这些基础很关键。尤其是 valid bit、flush/kill 和 commit trace，后续 precise trap 会直接复用这些设计思想。

### 1.2 当前明确缺少的能力

当前核离“更完整的裸机 core”主要差下面几类内容：

| 缺口 | 当前表现 | 后续要补什么 |
|---|---|---|
| SYSTEM 类指令 | `OPCODE_SYSTEM`、`ECALL/EBREAK` 仍是注释占位 | 译码 `ECALL/EBREAK/MRET` 和 CSR 指令 |
| `FENCE` | `OPCODE_MISC_MEM` 仍是注释占位 | 无 cache/MMIO 顺序模型时可先按 NOP 完成 |
| CSR 文件 | 没有 `mstatus/mtvec/mepc/mcause/mtval` 等状态 | 新增 M-mode CSR storage、读写和 WARL 规则 |
| precise trap | illegal/misaligned 只在 testbench 中停止 | trap entry 写 CSR、redirect 到 `mtvec`、kill younger 指令 |
| `MRET` | 没有 trap return | 从 `mepc` 返回并恢复 `mstatus` 位 |
| interrupt | 没有 `mie/mip`、外部 interrupt 输入或 timer pending | 后续加入 machine interrupt 屏蔽、挂起和优先级 |
| MMIO | dmem 只接 simple RAM | 后续加入地址译码、GPIO/UART/timer |
| 可变延迟访问 | imem/dmem 默认固定响应 | 后续补 ready/valid 或 stall/backpressure |
| 更系统验证 | 当前覆盖 RV32I 主路径和 hazard | 后续补 trap、CSR、interrupt、MMIO、异常副作用测试 |

换句话说，当前不是缺 ALU/GPR/流水线主干，而是缺“真实 core 能被软件控制、出错能恢复、能和外设交互”的系统面。

### 2.2 每一步的边界

#### 阶段 1：最小 M-mode CSR/trap

目标是让当前核从“遇到非法/不对齐就由 testbench 停止”，升级为“硬件进入 trap handler，软件能处理并返回”。

本阶段应包含：

- `FENCE` 按 NOP 支持。
- `ECALL/EBREAK/MRET` 译码和执行。
- Zicsr 的 6 条 CSR 指令：`CSRRW/CSRRS/CSRRC/CSRRWI/CSRRSI/CSRRCI`。
- 最小 M-mode CSR：`mstatus/mtvec/mepc/mcause/mtval/mscratch`。
- 建议补只读 ID CSR：`misa/mvendorid/marchid/mimpid/mhartid`。
- illegal instruction、instruction address misaligned、load/store address misaligned、`ECALL`、`EBREAK` 的 precise trap。
- trap redirect 优先级和 younger instruction kill。
- trap/CSR directed test。

不包含：

- interrupt。
- MMIO timer。
- S/U mode。
- delegation。
- nested trap 的完整处理策略。

#### 阶段 2：最小 memory map 与 MMIO

目标是把 dmem 从“只有 simple RAM”扩展成“SRAM + 外设寄存器”的地址空间。

建议内容：

- 定义 memory map，例如 IMEM、SRAM、MMIO base。
- 新增最小平台 wrapper，把 core、ROM、RAM、MMIO 译码和外设连接起来。
- 新增简单地址译码模块。
- 接入 GPIO 寄存器块。
- 接入 UART 寄存器块，当前先实现 TX，至少能输出字符。
- 加入未映射 load/store access fault，复用阶段1的 trap 路径。
- 软件测试从写 DMEM pass flag 扩展到写 MMIO 寄存器。

不建议一开始就接 AXI/APB。当前阶段可以先用固定响应的简单地址译码，等外设数量、wait state 或 accelerator 访问需求明确后，再在阶段4抽象成 request/response 或 ready/valid 内部总线。

#### 阶段 3：machine interrupt 与 timer

目标是让 core 能在指令边界接受 machine interrupt，并通过 handler 返回。

建议内容：

- `mstatus.MIE/MPIE/MPP` 与 interrupt entry/return 规则。
- `mie/mip` 的 machine software/timer/external interrupt 位。
- core 顶层加入 `msip/mtip/meip` 或统一 interrupt 输入。
- 若阶段2已有 timer MMIO，则加入 `mtime/mtimecmp` 产生 `MTIP`。
- interrupt 与 synchronous exception 的优先级。
- interrupt entry 写 `mepc/mcause`，`mcause[31]` 表示 interrupt。
- `MRET` 返回 interrupt 前 PC。

这一阶段完成后，已经可以把当前核的稳定版本作为**可选的上板验证分支**来单独推进，例如做 FPGA bring-up、LED/UART/timer 冒烟和板级存储接口验证。但这条线不属于本仓库的主线目标，主线仍然按 `0834`、`0835`、`0836` 的顺序继续推进 data-side 可变延迟访问、验证收口和 AXI-Lite 过渡。

最小展示程序：UART 打印、timer 中断、GPIO/LED 翻转，能讲清楚“CPU 在板上跑起来了”。

#### 阶段 4：data-side 可变延迟 memory/MMIO、简化内部总线与流水线 backpressure

目标是先摆脱 LSU 数据侧“所有 memory/MMIO 都固定响应”的假设，为后续 AXI-Lite、慢外设和 accelerator 控制面打基础。

建议内容：

- data side/LSU 加入单 outstanding 的 request/response 或 ready/valid。
- 第一版 IMEM 仍可保持固定响应，避免同时改前端取指和 LSU。
- MEM 在 load/store/MMIO 未完成时 stall，并向 EX/ID/IF 施加 backpressure。
- response 中携带 `rdata/error`，把阶段2的 access fault 语义自然迁移到 response error。
- stall、flush、trap redirect、load-use、未完成 memory request 之间重新定义优先级。
- 慢响应期间不能重复发起同一笔 request，不能重复写 MMIO，不能重复触发 UART/GPIO/TIMER 副作用。
- 增加 wait state 仿真模型，测试 DMEM/MMIO 慢响应时流水线不丢指令。

这一阶段会明显改变 hazard/control 网络，所以应晚于 CSR/trap 和固定响应 MMIO 的第一版。

这一阶段可以理解为“先做项目内的简化 data bus”，不直接把 CPU 核内部接口改成 AXI-Lite：

```text
request:
  valid, ready, write, addr, wdata, be

response:
  valid, rdata, error
```

简化总线稳定后，再包装 AXI-Lite 会更自然。若直接把 AXI-Lite 五通道接进 CPU，容易同时引入地址译码、外设寄存器、握手状态机、流水线 stall 和错误响应，调试维度过多。

阶段 4 做完后当然可以，也应该继续用当前这种软件自检型 directed test 做基本功能验证。这类验证仍然属于阶段 4 的完成条件：

- 0 wait-state 下现有测试全部不退化。
- 插入固定 wait-state 后，load/store/MMIO 还能正常完成。
- 简单 timer/GPIO/UART 中断还能跑通。
- timeout/status RAM 机制还能工作。

这里的 wait-state 建模不要求 GPIO/UART/TIMER32 每个外设模块内部都改成多拍状态机。更推荐第一版在 `data_subsystem` 或 simple-bus responder/wrapper 层插入延迟：

```text
core LSU
  <-> simple data bus
  <-> wait-state responder / data_subsystem
  <-> fixed-response RAM/GPIO/UART/TIMER32 register block
```

这样外设寄存器 ABI 保持不变，外设模块仍可以是简单寄存器块；CPU 侧看到的是“data access 不一定同拍完成”。这更接近后续 AXI-Lite/APB bridge、仲裁、跨时钟外设或 accelerator 控制窗口带来的系统延迟。

但阶段 4 里新增的是 backpressure + request/response + delayed error + MMIO side effect once，它比之前 CSR/MMIO/interrupt 的“功能寄存器语义”更偏微架构控制。只靠软件程序很难充分证明这些边界，例如：

- `valid && !ready` 期间 payload 是否保持稳定。
- store/MMIO write 是否因为 request 保持而重复生效。
- W1C、读副作用、UART TX pulse 是否只在成功 transaction 上发生一次。
- memory wait 期间 younger branch redirect 是否真的不能越过 older memory。
- interrupt pending 是否只在 memory completion 边界被接受。
- delayed access fault 和 interrupt/redirect 的优先级是否精确。
- 随机 wait-state 下是否还满足这些性质。

这些更适合 TB monitor、断言 SVA、随机 wait-state、scoreboard，甚至后续 UVM agent 去覆盖。所以阶段 5 的价值不是“阶段 4 完全不能验证”，也不是“继续大改 CPU 功能”，而是：阶段 4 先把 RTL 和最小 smoke 跑通，阶段 5 再把这个新时序边界验证得可展示、可解释、可维护。

#### 阶段 5：wait-state 验证收口、SVA 与 UVM 入门 demo

目标是在进入 AXI-Lite 之前，把阶段 4 新增的可变延迟访问和 backpressure 语义验证住，同时用较小范围开始补 UVM 方法学。本阶段原则上不再大改 CPU 核功能；除非阶段 4 遗留 bug，主要工作应落在 testbench、monitor、assertion、scoreboard 和 UVM demo。

阶段 5 的随机/组合 wait-state 可以来自两类位置：

- SoC directed regression：继续使用当前 C/asm self-check 程序，TB 或 `data_subsystem` 通过参数、plusarg 或测试配置给 DMEM/MMIO responder 插入不同延迟。
- simple-bus/UVM demo：不必跑完整 SoC，可以单独实例化 simple-bus slave/peripheral wrapper，由 driver/sequence 随机控制 `ready` 和 `resp_valid` 延迟。

因此，阶段 5 不要求把 `simple_ram`、GPIO、UART、TIMER32 都改成真实慢设备。真正需要验证的是 simple data bus 边界：CPU 面对可变响应延迟时，流水线、异常、中断和 MMIO 副作用是否仍然满足阶段4 定义的协议。

建议内容：

- 扩展 wait-state directed test：慢 load 后续 RAW、慢 store 不重复写、慢 MMIO read/write、delayed access fault、memory stall 期间 interrupt pending。
- 增加关键 SVA：单 outstanding、request accepted 后等待 response 前不再发第二笔、response 必须对应 outstanding request、stall 时 pipeline register 保持、wrong-path 不产生 bus request、每条 store/MMIO side effect 最多发生一次。
- 保留 C/asm directed self-check 作为 CPU 功能回归主线。
- 建立第一版 UVM demo，不直接验证整颗 CPU，而是验证 simple-bus/peripheral：
  - driver 发简化 bus transaction。
  - monitor 观察 request/response。
  - scoreboard 检查 GPIO/UART/TIMER32 寄存器语义。
  - sequence 随机读写、随机 wait-state、W1C/读副作用组合。

UVM 不建议抢在阶段 2 之前大规模做，也不建议零基础时直接验证整颗 CPU。阶段 5 的定位是“用小而真实的 simple-bus/peripheral 环境学会 agent、sequence、driver、monitor、scoreboard 和 coverage 的基本组织”，同时保留现有软件自检测试作为端到端功能回归。

#### 阶段 6：AXI-Lite adapter/interconnect 与 accelerator 控制窗口定义

目标是在不污染 CPU 内部接口的前提下，把阶段4的简化 data bus 包装到 AXI-Lite，并定义后续 accelerator 通过总线被 CPU 调用的控制面协议。

推荐结构：

```text
core LSU
  -> internal simple data bus
  -> simple_bus_to_axi_lite
  -> axi_lite_interconnect/decoder
  -> DMEM / GPIO / UART / TIMER32 / reserved accelerator slot
```

建议内容：

- 支持 AXI4-Lite single-beat read/write。
- 不支持 burst、不支持 ID、不支持多 outstanding。
- `WSTRB` 对应原 byte enable。
- `BRESP/RRESP` 的 `SLVERR/DECERR` 映射回 simple bus `error`，再进入 CPU load/store access fault。
- 外设寄存器语义保持不变，软件 ABI 不因 AXI-Lite 改变。
- 地址图预留 `ACCEL0_BASE` 一类 accelerator control window，但不在本项目内实现 accelerator 本体。

后续 accelerator 控制窗口建议先按 AXI-Lite slave 的寄存器控制面定义：

```text
ACCEL_CTRL        bit0 start, bit1 irq_en
ACCEL_STATUS      bit0 busy, bit1 done, bit2 error
ACCEL_SRC_ADDR    输入 buffer 起始地址
ACCEL_DST_ADDR    输出 buffer 起始地址
ACCEL_LEN         数据长度或 tile 数量
ACCEL_CONFIG      算子/模式/参数
ACCEL_IRQ_PENDING W1C 完成/错误中断 pending
```

CPU 侧软件流程：

```text
写 SRC_ADDR/DST_ADDR/LEN/CONFIG
写 CTRL.start
轮询 STATUS.done，或等待 accelerator interrupt
读 STATUS/IRQ_PENDING，必要时 W1C 清 pending
```

这一步完成后，就可以提醒并单独展开 accelerator/NPU 的选择与设计。accelerator 属于独立专题，不纳入当前 RV32I core/SoC 主线；当前主线只保证 CPU 能通过 AXI-Lite 控制寄存器调用这类外设。

`0836` 完成后应提醒：此时已经具备 CPU 通过 AXI-Lite 控制外设/accelerator 的接口基础，可以另起 accelerator/NPU 专题选择算子、数据流和寄存器 ABI。但 accelerator 本体不属于当前 RV32I 教学核主线，避免把 CPU、总线、验证和计算加速器同时混在一个阶段里推进。

#### 阶段 7：AXI-Lite UVM 协议与外设寄存器验证

目标是把 UVM 从阶段5的 demo 推进到真正有展示价值的协议验证。

建议内容：

- 建立 AXI-Lite master/slave agent，第一版至少覆盖 CPU 访问外设方向。
- 随机 `AWREADY/WREADY/BVALID/ARREADY/RVALID` 延迟。
- 检查 AXI-Lite 握手稳定性：`VALID` 拉高后 payload 在 handshake 前保持，response 不丢、不重、不乱。
- 检查 write address/write data/write response 配对，read response 与地址匹配。
- scoreboard 对比 AXI-Lite transaction 和后端寄存器模型。
- coverage 覆盖 read/write、OKAY/SLVERR/DECERR、不同 wait-state、byte strobe 和不同 slave window。

这一阶段的目标不是替代 CPU directed test，而是把标准总线和外设寄存器语义验证扎实。面试中可以清楚说明为什么 AXI-Lite 适合当前单发射、单 outstanding、无 cache 的教学 SoC。

#### 阶段 8：SoC/CPU 级 UVM 集成

目标是在已有 C/asm directed self-check 的基础上，用 UVM 把 SoC/CPU 级观察点系统化，而不是从零开始写完整 ISS lockstep。

建议内容：

- UVM testbench 负责 reset/clock/config、memory image load、仿真结束管理。
- monitor 观察 commit、trap、AXI-Lite transaction、interrupt、GPIO/UART/TIMER32 事件。
- scoreboard 检查 PASS/FAIL、trap cause、MMIO 副作用、interrupt entry/return 边界。
- 保留现有 directed tests 作为主功能回归，用 UVM 做更强的观察、协议检查和覆盖统计。
- ISS lockstep 可作为更后续增强项，不阻塞本阶段目标。

这一阶段适合放在 AXI-Lite 和 AXI-Lite UVM 稳定之后。这样 UVM 环境面对的是相对稳定的 SoC 边界，避免零基础时一边改 CPU 总线接口、一边重构验证平台。

## 第 3 章 当前距离目标还差多少

从工作量上看，当前已经完成了 CPU 主干里最难讲清楚的一部分：五级流水线数据通路、data hazard 和 control hazard。下一阶段不是重写 core，而是在现有主干上补控制状态和系统接口。

大致差距可以这样估计：

| 类别 | 差距 | 影响范围 |
|---|---|---|
| ISA 补全 | 小到中 | `core_pkg.sv`、`decoder.sv`、`id_stage.sv` |
| CSR 文件 | 中 | 新增 `csr_file.sv`，影响 CSR 指令写回和 trap entry |
| precise trap | 中到大 | pipeline register、hazard/flush 优先级、MEM/WB side effect gating |
| interrupt | 中 | CSR、顶层接口、trap controller、testbench |
| MMIO | 中 | memory map、dmem wrapper、外设模块、软件测试 |
| wait state/简化总线 | 中到大 | MEM stall/backpressure、request/response、hazard 优先级 |
| AXI-Lite 接入 | 中 | simple bus adapter、AXI-Lite interconnect、response/error 映射、外设地址译码 |
| 验证收口 | 中到大 | directed test、SVA、scoreboard、UVM demo、AXI-Lite UVM、SoC/CPU 级 UVM |
| accelerator 控制接口预留 | 小到中 | AXI-Lite 控制窗口、寄存器 ABI、中断/状态语义；accelerator 本体另起专题 |

因此下一阶段最合理的判断是：

```text
数据通路主干已经完成；
要成为更完整的裸机 core，还需要补 4～5 个系统性能力包：
CSR/trap、MMIO、interrupt、可变延迟访问/简化总线、AXI-Lite 接入、验证收口。
AXI-Lite 控制面稳定后，就可以单独展开 accelerator/NPU 的选择与设计；当前 RV32I 项目只负责提供 CPU 调用 accelerator 的总线与寄存器接口边界。
```

## 第 4 章 为什么第一步选最小 M-mode CSR/trap

第一步选 CSR/trap，而不是先选 MMIO、interrupt 或 M 扩展，原因如下：

| 原因 | 说明 |
|---|---|
| 当前 RTL 已经有基础信号 | `illegal_instr_o`、`mem_misaligned_o`、valid bit、flush/kill 都能复用 |
| trap 是 interrupt 的前置条件 | interrupt entry/return 本质上复用 trap entry/`MRET` |
| trap 是错误处理的前置条件 | MMIO access fault、bus error、misaligned 都需要统一入口 |
| trap 能验证 precise kill 能力 | 这比普通 branch flush 更接近真实 CPU 提交语义 |
| CSR 是软件控制面的起点 | 没有 `mtvec/mepc/mcause`，软件无法可靠知道发生了什么 |

第一步完成后，当前核会从“只适合跑保证合法且对齐的测试程序”，升级为“可以跑带 trap handler 的 M-mode 裸机程序”。这会明显提升项目完整度。
