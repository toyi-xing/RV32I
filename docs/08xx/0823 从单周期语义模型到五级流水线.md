# 0823 从单周期语义模型到五级流水线

> 文档编号：0823  
> 所属系列：082x RISC-V 最小教学核项目实践  
> 文档定位：说明为什么建议先写一个单周期/顺序语义模型，再演进到五级流水线教学核  
> 前置文档：`0820 RISC-V最小教学核设计流程与方案.md`、`0821 RV32I最小教学核指令集、编码与译码参考.md`、`0822 最小教学核工程目录、顶层接口与命名约定.md`

本文解决的是实现路径问题：不要一上来就把 IF/ID/EX/MEM/WB、forwarding、load-use、branch flush 全部写完。更稳的做法是先写一个“每拍执行一条指令”的语义模型，把指令行为跑通；再把同一套 decoder、imm_gen、ALU、regfile 和 memory 访问逐步切成五级流水线。

本文默认项目假设：

| 项目 | 假设 |
|---|---|
| ISA | RV32I 主线可执行子集 |
| memory | 分离 imem/dmem，第一版固定响应 |
| reset PC | `32'h0000_0000` |
| 单周期模型 | 用于确认指令语义，不一定作为最终交付微架构 |
| 五级流水线 | IF、ID、EX、MEM、WB，单发射顺序提交 |
| hazard | 本篇只说明何时引入；具体规则见 `0825` |

## 第1章 为什么先做单周期语义模型

### 1.1 它不是为了性能

单周期语义模型的目标不是做一个高性能 CPU，而是先回答最基本的问题：每条指令的最终效果是不是对的。

例如：

| 指令 | 语义模型要证明什么 |
|---|---|
| `ADDI` | I-imm 符号扩展正确，`rs1 + imm` 写回正确 |
| `LUI` | U-imm 拼接正确，没有误读 `rs1` |
| `LW` | 地址计算、读 memory、写回路径正确 |
| `LB/LBU` | byte 选择和符号/零扩展正确 |
| `SW` | store 地址、写数据、byte enable 正确 |
| `BEQ` | 比较条件和 branch target 正确 |
| `JALR` | `rs1 + imm` 后 bit0 清零，`PC+4` 写回正确 |

这些问题和流水线无关。如果它们本身错了，直接写五级流水线会让 debug 变得很难，因为你不知道错误来自“指令语义”还是“流水线控制”。

### 1.2 单周期模型能当作对照

推荐把单周期模型当成一个简化 reference：

| 用法 | 价值 |
|---|---|
| 跑短 directed test | 快速确认每条指令语义 |
| 生成期望 commit trace | 和流水线版本对比 |
| 验证 decoder/imm_gen/ALU | 这些模块后续可以复用 |
| 缩小 bug 范围 | 单周期过、流水线不过，多半是 hazard/valid/flush 问题 |

它不一定要支持所有后续功能，也不一定要写得和最终 core 一样模块化。但 decoder、imm_gen、ALU、load 扩展这几块最好复用或保持同样语义。

## 第2章 单周期语义模型应该长什么样

### 2.1 一拍执行一条指令

单周期模型可以理解为：

```text
当前 PC
  ↓
imem 取 instruction
  ↓
decode + imm_gen
  ↓
读 GPR
  ↓
ALU/branch/load/store/writeback 组合计算
  ↓
时钟沿更新 PC、GPR、dmem
```

一个 cycle 内完成一条指令的所有组合计算，在时钟沿更新架构状态。这里的“单周期”是教学模型，不代表真实硬件时序一定能跑很高频率。

### 2.2 架构状态

第一版单周期模型只需要维护这些架构状态：

| 状态 | 说明 |
|---|---|
| `pc_q` | 当前取指地址 |
| GPR | 32 个通用寄存器，x0 恒为 0 |
| dmem | data memory，load/store 可见 |

没有 CSR、异常状态、特权级状态、cache 状态，所以状态空间很小。

### 2.3 单周期核心骨架

下面是结构骨架，不是完整 RTL：

```systemverilog
always_comb begin
    pc_d        = pc_q + 32'd4;
    reg_we      = 1'b0;
    rd_wdata    = 32'b0;
    dmem_we     = 1'b0;
    dmem_be     = 4'b0000;
    illegal     = 1'b0;

    instr       = imem_rdata_i;
    dec         = decode(instr);
    imm         = gen_imm(instr, dec.imm_sel);

    rs1_data    = gpr_rdata1;
    rs2_data    = gpr_rdata2;

    alu_result  = alu_calc(dec.alu_op, alu_a, alu_b);

    unique case (dec.kind)
        KIND_ALU: begin
            reg_we   = dec.reg_we;
            rd_wdata = alu_result;
        end

        KIND_LOAD: begin
            reg_we   = 1'b1;
            rd_wdata = load_extend(dmem_rdata_i, addr[1:0], dec.mem_size, dec.mem_unsigned);
        end

        KIND_STORE: begin
            dmem_we  = 1'b1;
            dmem_be  = store_be(addr[1:0], dec.mem_size);
        end

        KIND_BRANCH: begin
            if (branch_taken) begin
                pc_d = pc_q + imm;
            end
        end

        KIND_JUMP: begin
            reg_we   = dec.reg_we;
            rd_wdata = pc_q + 32'd4;
            pc_d     = jump_target;
        end

        default: begin
            illegal = 1'b1;
        end
    endcase
end

always_ff @(posedge clk_i or negedge rst_n_i) begin
    if (!rst_n_i) begin
        pc_q <= RESET_PC;
    end else begin
        pc_q <= pc_d;
        if (reg_we && (rd_addr != 5'd0)) begin
            gpr[rd_addr] <= rd_wdata;
        end
    end
end
```

真实代码不建议把 function、GPR、memory 全塞进一个文件。这里是为了说明“一拍内组合算出下一状态，时钟沿更新状态”的语义。

### 2.4 单周期模型也要保留控制信号

即使是单周期模型，也建议使用和流水线版本接近的 decoder 输出：

| 控制信号 | 单周期用途 | 流水线用途 |
|---|---|---|
| `uses_rs1/uses_rs2` | 可暂时不用 | hazard 检测需要 |
| `reg_we` | 控制本拍写 GPR | 随指令流到 WB |
| `imm_sel` | 选择立即数 | ID 生成立即数 |
| `alu_op` | ALU 计算 | EX 计算 |
| `branch_op` | branch 判断 | EX 判断 |
| `mem_we/mem_re` | dmem 访问 | MEM 阶段访问 |
| `mem_size/mem_unsigned` | store byte enable 和 load 扩展 | MEM 阶段使用 |
| `wb_sel` | 选择写回数据 | WB 阶段使用 |

这样后续切流水线时，decoder 不需要大改，只是控制信号从“本拍使用”变成“随指令流动到后级使用”。

## 第3章 单周期阶段的测试顺序

### 3.1 先测不依赖 hazard 的指令

单周期模型没有多条指令重叠执行，所以没有 forwarding/stall/flush 问题。测试重点应放在指令语义。

推荐顺序：

| 顺序 | 测试 | 目标 |
|---|---|---|
| 1 | `ADDI`、`ADD`、`SUB` | 跑通 GPR、ALU、WB |
| 2 | `LUI`、`AUIPC` | 验证 U-imm 和 PC 参与计算 |
| 3 | `AND/OR/XOR`、`SLL/SRL/SRA` | 验证 ALU 子功能 |
| 4 | `SLT/SLTU/SLTI/SLTIU` | 验证 signed/unsigned 比较 |
| 5 | `SW/LW` | 跑通 dmem 基础读写 |
| 6 | `SB/SH/LB/LH/LBU/LHU` | 验证 byte lane 和扩展 |
| 7 | branch | 验证 taken/not-taken 和 B-imm |
| 8 | `JAL/JALR` | 验证 `PC+4` 写回和 target |

### 3.2 directed test 要短

每个测试尽量只覆盖一个点。例如测试 `LB` 符号扩展时，不要同时测试复杂 branch 和 forwarding。这样失败时更容易定位。

| 不推荐 | 推荐 |
|---|---|
| 一个 200 行汇编测所有功能 | 每类指令一个短 test |
| 最后只检查一个总结果 | 每个关键阶段写入不同结果地址 |
| 出错后只看最后 memory | 保留 commit trace 或至少打印 PC/instr/rd/wdata |

## 第4章 从单周期切到五级流水线

### 4.1 五级流水线的切分

同一条指令在单周期里一拍完成；切成五级后，工作被分散到不同 stage：

| 单周期中的工作 | 五级流水位置 |
|---|---|
| 根据 PC 取 instruction | IF |
| decode、读 GPR、生成 immediate | ID |
| ALU、branch compare、target 计算 | EX |
| load/store 访问 memory、load 扩展 | MEM |
| 选择写回数据、写 GPR | WB |

切流水线不是改变 ISA 语义，而是改变这些工作发生的时间。

### 4.2 第一步：先加 pipeline register

先不要急着处理所有 hazard。第一步是让指令信息能流过五级：

```text
pc_reg -> IF -> IF/ID -> ID -> ID/EX -> EX -> EX/MEM -> MEM -> MEM/WB -> WB
```

每个 pipeline register 至少要有：

| 字段 | 目的 |
|---|---|
| `valid` | 表示这个槽里是否有真实指令 |
| `pc` | debug、branch target、commit trace |
| `instr` | decode/debug/trace |
| 必要数据 | 例如 rs1/rs2 值、imm、ALU result、load data |
| 必要控制 | 例如 reg_we、mem_we、wb_sel |

字段细节见 `0824`。

### 4.2.1 stage 模块和 pipeline register 的边界

`if_stage/id_stage/ex_stage/mem_stage/wb_stage` 这些 stage 模块不等于 stage 间寄存器。第一版建议把它们理解成“本阶段的组合数据通路”，而把 `IF/ID`、`ID/EX`、`EX/MEM`、`MEM/WB` 理解成单独的流水线槽位。PC 是取指前端的架构/微架构状态，建议单独放在 `pc_reg`，不要塞进纯组合的 `if_stage`。

典型连接关系是：

```text
pc_reg -> IF stage -> IF/ID reg -> ID stage -> ID/EX reg -> EX stage -> EX/MEM reg -> MEM stage -> MEM/WB reg -> WB stage
```

因此：

| 模块 | 是否自己保存 stage 间状态 | 说明 |
|---|---|---|
| `pc_reg` | 是 | 保存当前取指 PC；响应 redirect/stall/reset，输出给 IF stage |
| `if_stage` | 否 | 输入来自 `pc_reg` 和 imem，组合输出取指地址、PC、PC+4、instruction 和 valid |
| `wb_stage` | 否 | 只根据 MEM/WB 寄存器已经保存的 `wb_sel/alu_result/load_data/pc_plus4/imm` 组合选出写回数据 |
| `mem_stage` | 否 | 输入来自 EX/MEM 寄存器，输出 `dmem_*` 和 `load_data`，再由 MEM/WB 寄存器保存给 WB |
| `ex_stage` | 否 | 输入来自 ID/EX 寄存器，输出 ALU 结果、store data、branch/jump redirect，再由 EX/MEM 或前端控制使用 |
| pipeline register | 是 | 保存 `valid`、数据字段和控制字段；flush/kill/stall/bubble 主要作用在这些槽位上 |

`flush` 和 `kill` 本质上不是某个普通组合 stage 的功能，而是对流水线槽位的控制：把错误路径或 bubble 对应的 `valid` 清 0，必要时同时清零关键控制信号。只要所有架构副作用都用 `valid` 门控，invalid 槽位就不会写 GPR、写 dmem 或产生 commit。

在单周期阶段可以先不实例化这些 pipeline register，因为一条指令没有跨拍流动；等切到五级流水线时，必须补齐这些槽位。

### 4.3 第二步：先跑无相关程序

在 forwarding/stall 没写好前，可以用插入 NOP 的测试程序让流水线先跑通：

```asm
addi x1, x0, 3
addi x0, x0, 0
addi x0, x0, 0
addi x0, x0, 0
add  x2, x1, x1
```

这样 `add` 使用 `x1` 时，`addi` 已经写回，暂时不需要 forwarding。这个阶段的目标是确认：

- IF 能持续取指。
- 指令能按 stage 前进。
- WB 能正确写 GPR。
- valid 为 0 的 bubble 不产生副作用。
- commit trace 顺序正确。

### 4.4 第三步：逐步去掉 NOP

等无相关程序通过后，再去掉 NOP：

```asm
addi x1, x0, 3
add  x2, x1, x1
```

这时如果没有 forwarding，`add` 会读到旧 `x1`。这正是引入 forwarding 的时机。不要在流水线刚搭好时就用这种程序做第一条测试，否则你会同时面对太多问题。

### 4.5 第四步：引入 load-use stall

ALU forwarding 不能解决 load-use：

```asm
lw   x3, 0(x4)
add  x5, x3, x6
```

因为 `lw` 的数据要到 MEM 阶段才从 memory 返回，紧跟的 `add` 在下一拍进入 EX 时还拿不到数据。典型处理：

```text
冻结 PC
冻结 IF/ID
向 ID/EX 插入 bubble
让 lw 从 EX 继续进入 MEM
下一拍 add 再进入 EX，从 MEM/WB 前递 load data
```

具体条件和优先级见 `0825`。

### 4.6 第五步：引入 branch/JAL/JALR flush

无控制流程序跑通后，再加入 branch/jump。

如果 branch 在 EX 阶段决策，那么 branch 后面已经进入 IF/ID，甚至 ID/EX 的年轻指令可能来自旧路径。taken 时必须 flush/kill：

```text
branch 在 EX 判定 taken
  ↓
PC 改成 target
  ↓
旧路径上的年轻指令 valid 清 0
  ↓
这些指令不能写 GPR，也不能 store
```

这一步是从“流水线能跑顺序程序”到“流水线能跑真实控制流”的关键。

## 第5章 单周期到流水线的信号迁移

### 5.1 数据信号迁移

| 单周期信号 | 流水线中变成 |
|---|---|
| `pc_q` | `pc_reg` 保存的当前取指 PC；经 IF 后同时保存到 IF/ID、ID/EX、EX/MEM、MEM/WB |
| `instr` | IF 取到后保存到 IF/ID，后续随指令流动 |
| `rs1_rdata/rs2_rdata` | ID 读出后保存到 ID/EX |
| `imm` | ID 生成后保存到 ID/EX |
| `alu_result` | EX 计算后保存到 EX/MEM |
| `dmem_rdata` | MEM 读出并扩展后保存到 MEM/WB |
| `rd_wdata` | WB 根据 `wb_sel` 选择 |

不要在 WB 阶段重新 decode 原始 instruction 再猜写回来源。控制信号应该从 ID 生成后随指令流动。

### 5.2 控制信号迁移

| 控制信号 | 生成位置 | 使用位置 | 是否需要随流水保存 |
|---|---|---|---|
| `imm_sel` | ID | ID | 通常不用传到 EX，除非 EX 再生成 imm |
| `alu_op` | ID | EX | 需要保存到 ID/EX |
| `op_a_sel/op_b_sel` | ID | EX | 需要保存到 ID/EX |
| `branch_op` | ID | EX | 需要保存到 ID/EX |
| `jump/jalr` | ID | EX | 需要保存到 ID/EX |
| `mem_re/mem_we` | ID | MEM | 需要经 ID/EX、EX/MEM 传递 |
| `mem_size/mem_unsigned` | ID | MEM | 需要经 ID/EX、EX/MEM 传递 |
| `reg_we` | ID | WB | 需要一路传到 MEM/WB |
| `wb_sel` | ID | WB | 需要一路传到 MEM/WB |
| `uses_rs1/uses_rs2` | ID | hazard/forwarding | 通常保存到 ID/EX，ID 阶段也要用 |

如果某个控制信号在后级才使用，就必须跟着该指令保存。否则后级会拿到错误指令的控制意图。

## 第6章 五级流水线最小验收路线

建议按下面验收：

| 里程碑 | 测试特点 | 通过标准 |
|---|---|---|
| P0 | reset 后无有效指令 | pipeline valid 清零，PC 为 reset PC |
| P1 | NOP 或死循环 | PC 正常递增/跳转，无异常写回 |
| P2 | 带 NOP 间隔的 ALU 程序 | WB 写回正确，commit trace 顺序正确 |
| P3 | 带 NOP 间隔的 load/store | dmem 地址、byte enable、load 扩展正确 |
| P4 | 连续 ALU 相关 | forwarding 正确，不需要 stall |
| P5 | load-use | 正确停一拍并插入 bubble |
| P6 | branch taken | 错路径不写 GPR，不写 dmem |
| P7 | JAL/JALR | `PC+4` 写回和 redirect 正确 |
| P8 | 混合短程序 | trace 和预期一致 |

不要跳过 P2/P3 直接跑混合程序。混合程序失败时，debug 成本会明显上升。

## 第7章 常见迁移错误

| 错误 | 现象 | 原因 |
|---|---|---|
| 控制信号没有随指令流动 | 某条指令用到后一条指令的控制 | decoder 输出直接接到后级 |
| pipeline register 不保存 `pc` | commit trace PC 错，branch target 难 debug | 以为只有 IF 需要 PC |
| valid 没有 gating 写回 | bubble 写 GPR 或 store | `reg_we/mem_we` 没有和 valid 相与 |
| forwarding 从 x0 触发 | 读 x0 得到非 0 | 没排除 `rd=0` |
| flush 只改 PC 不 kill 指令 | taken 后错路径 store 真的写了 memory | 没清 younger 指令 valid |
| load-use 只停 IF 不插 bubble | load 和使用者时序错乱 | 控制动作不完整 |
| dmem 地址映射混乱 | C/汇编写的地址和 dmem 内部索引不一致 | 没定义 DMEM_BASE wrapper |
