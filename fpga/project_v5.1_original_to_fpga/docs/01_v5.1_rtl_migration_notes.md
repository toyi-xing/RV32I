# v5.1 RTL 迁移说明

本文档记录 `project_v5.1_original` 到 `project_v5.1_fpga` 的 RTL/工程侧迁移。重点是说明哪些设计边界保持不变、哪些文件为 FPGA 上板做了适配，以及为什么 DMEM 时序修复会影响中断程序。

## 迁移目标

v5.1 FPGA 版的目标不是重新设计 CPU，而是在尽量保持原 SoC 行为的前提下，把仿真友好的存储器和事件接口替换为 FPGA 可综合、可下载、可观察的实现。

主要约束如下：

- 目标板卡为 E10，FPGA 器件为 `EP4CE10F17C8`。
- 板载时钟按 50 MHz 使用，不引入 PLL。
- CPU core 目录保持不改，避免把上板适配问题混入指令流水线实现。
- SoC MMIO 基地址保持不改，软件侧仍按原有 GPIO、UART、TIMER 地址访问。
- IMEM/DMEM 容量从仿真工程的 256 KiB 缩小到各 16 KiB，以适配 EP4CE10 M9K 资源。
- FPGA 版通过镜像文件初始化片上 RAM，Quartus 编译时把当前 C 程序打包进 `.sof`。

## 不变边界

| 边界 | 结论 | 说明 |
| --- | --- | --- |
| CPU core RTL | 保持不变 | `rtl/core/*.sv` 两个版本文件哈希一致，流水线、CSR、trap、hazard/forwarding 均未因上板改动。 |
| RV32I 指令行为 | 保持不变 | 上板迁移没有新增或删除指令，也没有改变异常/中断提交语义。 |
| SoC MMIO 地址图 | 保持不变 | GPIO、UART、TIMER 仍挂在原 MMIO 区间，软件访问模型不需要重写。 |
| 仿真工程 | 保留镜像 | `project_v5.1_original/` 作为上板前基线，仍可用于 Verilator/仿真器回归，并于 FPGA 进行对比。 |
| FPGA 工程 | 独立 | `project_v5.1_fpga/` 独立保存 Quartus、板级 top、memory wrapper 和上板程序。 |

## RTL 文件变更总表

| 文件/目录 | 变更类型 | FPGA 版处理 | 原因 |
| --- | --- | --- | --- |
| `rtl/core/*.sv` | 保持 | 原样复制 | 保持 CPU core 行为稳定，把迁移风险限制在 SoC 包装和 FPGA 适配层。 |
| `rtl/common/core_pkg.sv` | 修改 | `IMEM_ADDR_WIDTH`/`DMEM_ADDR_WIDTH` 从 16 改为 12，`IMEM_SIZE_BYTES`/`DMEM_SIZE_BYTES` 从 256 KiB 改为 16 KiB | EP4CE10 M9K 资源有限，原 256 KiB + 256 KiB 无法经济地放入目标器件。 |
| `rtl/common/pipeline_pkg.sv` | 保持 | 原样复制 | 流水线控制数据结构未变化。 |
| `rtl/common/soc_pkg.sv` | 保持 | 原样复制 | MMIO 寄存器偏移、外设地址和 SoC 级常量保持不变。 |
| `rtl/soc/rv32i_soc.sv` | 小幅修改 | 增加 `uart0_tx_ready_i`，并继续向 `data_subsystem` 透传 | FPGA 上真实 UART TX 需要 busy/ready 反馈，避免软件写 TXDATA 时无条件发出一拍事件。 |
| `rtl/soc/data_subsystem.sv` | 小幅修改 | 增加 `uart0_tx_ready_i`，连接到 `mmio_uart.tx_ready_i` | 让 UART MMIO 状态寄存器和写 TXDATA 行为反映真实串口发送器状态。 |
| `rtl/periph/mmio_uart.sv` | 修改 | 新增 `tx_ready_i`；`STATUS.TX_READY` 由输入驱动；TX event 仅在 ready 时产生 | 把原仿真中的“永远 ready”改成可接真实 UART PHY 的握手模型。 |
| `rtl/periph/mmio_gpio.sv` | 保持 | 原样复制 | GPIO MMIO 行为不需要因上板改变。 |
| `rtl/periph/mmio_timer32.sv` | 保持 | 原样复制 | TIMER0 计数和中断模型不需要因上板改变。 |
| `rtl/mem/simple_rom.sv` | 移除/替代 | FPGA 版不再使用，改用 `rtl/fpga/fpga_imem.sv` | 原 ROM 面向仿真 `$readmemh` 和通用模型，FPGA 版需要明确约束到 M9K/Quartus 初始化文件。 |
| `rtl/mem/simple_ram.sv` | 移除/替代 | FPGA 版不再使用，改用 `rtl/fpga/fpga_dmem.sv` | 原 RAM 读时序与 FPGA block RAM 不完全一致，需要专门 wrapper 匹配 CPU 访存假设。 |
| `rtl/fpga/e10_rv32i_top.sv` | 新增 | 板级顶层，连接 E10 时钟/复位、KEY、LED、UART、SoC、IMEM、DMEM | 把 SoC 的抽象接口落到真实 FPGA 引脚和板载外设。 |
| `rtl/fpga/fpga_imem.sv` | 新增 | 用 M9K 风格数组实现指令存储器，从 `../mem/current_imem.mem` 初始化 | 生成 `.sof` 时把当前程序指令镜像固化进 FPGA 配置。 |
| `rtl/fpga/fpga_dmem.sv` | 新增 | 用 `altsyncram` 实现数据存储器，从 `../mem/current_dmem.mif` 初始化，支持 byte enable | 匹配 Cyclone IV E M9K、字节写使能和 DMEM 初值需求。 |
| `rtl/fpga/uart_tx_phy.sv` | 新增 | 50 MHz / 115200 baud UART 发送物理层，输出 `ready_o` | 把 UART MMIO 写事件转换为真实串口 TX 波形。 |
| `rtl/fpga/uart_rx_phy.sv` | 新增 | 50 MHz / 115200 baud UART 接收物理层，输出 `valid_o/data_o` | 把真实串口 RX 波形转换为 SoC UART RX event。 |
| `fpga/quartus/*` | 新增 | Quartus `qpf/qsf/sdc` 和工程说明 | 定义器件、顶层、引脚、时钟约束和编译入口。 |
| `fpga/mem/*` | 新增 | `default_*` 与 `current_*` 镜像文件 | 给 Quartus memory initialization 和快速切换测试程序使用。 |

## 关键适配说明

### FPGA top

`e10_rv32i_top.sv` 是 FPGA 版唯一板级顶层，职责是把原 SoC 的抽象接口接到真实板卡资源：

- `sys_clk` 直接作为 SoC、IMEM、DMEM、UART PHY 的工作时钟。
- `sys_rst_n` 经过两级同步后作为内部复位释放信号。
- `key[1:0]` 映射到 `gpio0_in[1:0]`。
- `gpio0_out[1:0] & gpio0_oe[1:0]` 驱动 `led[1:0]`。
- UART MMIO 的 TX/RX event 接到 `uart_tx_phy` 和 `uart_rx_phy`。
- 调试/仿真观测端口在板级顶层中悬空，不占用 FPGA 引脚。

### 存储容量缩小

original 版本中 `IMEM_ADDR_WIDTH = 16`、`DMEM_ADDR_WIDTH = 16`，按 32-bit word 寻址，对应 IMEM/DMEM 各 256 KiB。FPGA 版改为 `ADDR_WIDTH = 12`，对应各 4096 个 32-bit word，即各 16 KiB。

| 区域 | original | FPGA | 基地址是否变化 |
| --- | ---: | ---: | --- |
| IMEM | 256 KiB | 16 KiB | 否，仍为 `0x0000_0000` |
| DMEM | 256 KiB | 16 KiB | 否，仍为 `0x0004_0000` |
| MMIO | 64 KiB | 64 KiB | 否，仍为 `0x0008_0000` |

这意味着软件链接脚本和镜像生成流程必须保证 `.text`、`.data`、`.bss`、栈等内容不超过新的 16 KiB 边界。

当前 `16 KiB IMEM + 16 KiB DMEM` 是阶段性低风险配置，而不是资源利用最优配置。它的好处是 IMEM/DMEM 对称，`ADDR_WIDTH = 12` 后 wrapper、脚本、MIF depth 和链接脚本都容易同步；同时 C 裸机环境有比较宽松的数据区和栈空间，适合先完成 GPIO、TIMER、UART 和中断上板闭环。

从后续扩展看，IMEM 通常比 DMEM 更容易先变紧张。C demo、UART 打印和更多测试逻辑主要增加 `.text`，而当前裸机程序的数据需求通常较小。DMEM 主要承载保留区、`.rodata`、`.data`、`.bss` 和 stack。当前链接脚本预留 `0x2000` 字节栈空间，因此 16 KiB DMEM 可以避免 bring-up 阶段过早处理栈和数据边界问题。

若继续使用 `ADDR_WIDTH` 表达容量，最自然的容量点是 2 的幂：

| 容量 | 32-bit words | `ADDR_WIDTH` | 说明 |
| ---: | ---: | ---: | --- |
| 4 KiB | 1024 | 10 | 可用于很小的数据区，但必须下调 stack，并严查 `.data/.bss`。 |
| 8 KiB | 2048 | 11 | 比较适合作为后续精简 DMEM 的第一步。 |
| 16 KiB | 4096 | 12 | 当前配置，C 栈和数据区余量较宽松。 |
| 32 KiB | 8192 | 13 | 适合作为后续扩大 IMEM 的低改动方案。 |

因此后续更实用的调整路线可以是 `32 KiB IMEM + 8 KiB DMEM`：仍保持 2 的幂深度，只需要同步 RTL 参数、链接脚本、`platform.h`、镜像容量检查和 MIF depth。若希望做 `24 KiB IMEM + 8 KiB DMEM` 这类更精细的分配，则不能只依赖 `1 << ADDR_WIDTH`，需要改成显式 `DEPTH_WORDS` 参数，例如 `24 KiB / 4 = 6144 words`，并同步处理地址范围检查和镜像生成深度。

### UART ready 握手

仿真工程里的 UART TX 可以被理解为“写 TXDATA 立即产生一拍事件”，因此状态寄存器里的 TX ready 固定为 1。FPGA 上真实串口发送一个字节需要多个 bit time，在发送期间不能无条件接收新的 TX event。

FPGA 版因此增加了这条链路：

```text
uart_tx_phy.ready_o
  -> e10_rv32i_top.uart0_tx_ready
  -> rv32i_soc.uart0_tx_ready_i
  -> data_subsystem.uart0_tx_ready_i
  -> mmio_uart.tx_ready_i
```

`mmio_uart` 的变化有两点：

- `STATUS.TX_READY` 反映 `tx_ready_i`。
- 只有 `tx_ready_i == 1` 时，软件写 TXDATA 才产生 `tx_valid_o`。

当前上板 demo 偏向低速打印/状态提示，这个简单 ready 模型已经足够。后续若要提高串口吞吐，可在 MMIO UART 内部加入 TX FIFO。

## FPGA memory wrapper

### IMEM

`fpga_imem.sv` 使用 M9K 风格数组，并通过：

```systemverilog
$readmemh("../mem/current_imem.mem", mem);
```

加载当前程序指令镜像。读口在时钟负边沿更新 `rdata_o`，用于贴合现有 CPU 取指路径对指令数据可用时刻的假设。

### DMEM

`fpga_dmem.sv` 使用 Quartus `altsyncram`，配置为：

| 项目 | 配置 |
| --- | --- |
| 模式 | `SINGLE_PORT` |
| 数据宽度 | 32 bit |
| 地址宽度 | `DMEM_ADDR_WIDTH = 12` |
| 深度 | 4096 words |
| byte enable | 4 lanes，8 bit/byte |
| RAM 类型 | `M9K` |
| 初始化文件 | `../mem/current_dmem.mif` |
| 读写同址策略 | `NEW_DATA_NO_NBE_READ` |

DMEM wrapper 使用 `~clk_i` 驱动 `altsyncram.clock0`。这个处理的核心目的，是让 M9K 同步读数据在 CPU 需要的阶段前稳定下来，避免原本仿真 RAM 与真实 block RAM 的读时序差异破坏 load/stack 恢复路径。

## DMEM 同步读时序问题

这次迁移中最值得记录的问题是：简单 GPIO 或 TIMER polling 程序可以通过，但 TIMER interrupt 程序曾经出现中断返回后寄存器/栈恢复异常。

原因链路如下：

| 现象 | 根因 |
| --- | --- |
| polling 类程序基本正常 | 访存模式简单，对栈保存/恢复依赖弱，不容易暴露时序错位。 |
| interrupt 程序异常 | trap entry 和 C handler 会频繁 `sw/lw` 保存、恢复寄存器，尤其依赖 DMEM load 返回数据的准确周期。 |
| 仿真版本未暴露 | original RAM 更接近组合读或理想读模型，和 FPGA M9K 的同步读行为不同。 |
| FPGA 版修复方向 | 调整 DMEM wrapper 的时钟/读出时序，使 CPU 在既有固定访存假设下读到正确数据。 |

这个问题说明：从仿真 RAM 切到 FPGA block RAM 时，不能只检查地址宽度和初始化文件，还必须检查 CPU pipeline 对 load 数据返回周期的具体假设。特别是中断、函数调用、栈访问，比简单 MMIO 轮询更适合作为存储时序验证用例。

## 上板验证程序

本次上板后做了一下程序验证：

| 顺序 | 测试 | 目的 |
| ---: | --- | --- |
| 1 | `0500_fpga_led_spark.c` | 确认复位、IMEM、GPIO output 和基础执行流。 |
| 2 | `0502_fpga_led_key.c` | 确认 GPIO input/output 映射。 |
| 3 | `0503_fpga_timer_blink.c` | 确认 TIMER0 polling 和 MMIO 读路径。 |
| 4 | `0504_fpga_timer_irq.c` | 确认 trap、CSR、中断、DMEM 栈保存/恢复。 |
| 5 | `0505_fpga_key_led_uart.c` | 确认 GPIO、KEY、TIMER、UART 的组合使用。 |

其中 `0504` 应被视为 FPGA memory timing 的关键测试。通过本次最简中断进出测试确定 FPGA block RAM 的同步读行为导致中断退出时恢复栈出错，而当前 CPU 核不支持可变延迟访存，因此需要将 `fpga_dmem.sv` 的时钟信号 `.clock0 (~clk_i)` 。
